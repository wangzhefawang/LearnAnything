## 定位

软件需求分析与系统设计是「软件系统开发与维护」子域的起点，它先回答"系统应该解决什么问题"，再回答"系统准备怎样解决这个问题"喵。

后端服务、数据库设计、接口开发、测试和运维都以这里形成的需求与设计为依据，因此起点出现的偏差往往会被后续环节不断放大喵。

## 类比

可以把开发软件类比为建造一座公路养护站喵。

业主最初可能只说："我们需要一座能支持日常养护工作的站点"喵。

这句话表达了愿望，却不能直接交给施工队，因为大家对"支持养护工作"的理解可能完全不同喵。

有人以为只需要办公室，有人认为必须有机械车库，还有人觉得夜间值守、应急物资储存和视频监控才是重点喵。

需求分析就像建筑师与业主、养护人员和安全负责人反复沟通，弄清楚站点由谁使用、在什么情况下使用、必须完成哪些工作，以及需要满足哪些安全和容量要求喵。

经过分析后，模糊诉求会变成可检查的要求，例如"车库至少容纳八辆养护车辆""值班人员应能在一分钟内调取应急预案""停电后关键监控设备继续运行两小时"喵。

这些要求不只描述"有什么房间"，还描述容量、速度、安全性和可靠性喵。

系统设计则像根据这些要求绘制总平面图、建筑图、电气图和管线图喵。

设计人员会决定办公室、仓库和车库如何分区，人员和车辆怎样通行，供电和监控系统如何连接，以及每个房间内部如何布置喵。

在这个类比中，需求分析关注"要建成什么样才算满足使用者"，系统设计关注"用怎样的结构和部件把它建出来"喵。

如果没有需求分析，团队可能把一座不适用的建筑建得非常精致喵。

如果只有需求而没有系统设计，施工人员即使知道目标，也可能各自采用不同做法，最终得到无法顺利协作的局部成果喵。

## 核心机制

### 为什么需要需求分析

软件项目最危险的问题通常不是程序写不出来，而是程序写出来以后才发现解决错了问题喵。

以公路巡检为例，业务人员说"希望快速上报路面病害"，开发人员可能理解为上传一张照片，而养护人员实际需要的可能是照片、位置、病害类型、严重程度和交通影响的完整记录喵。

如果不澄清这些差异，开发团队可能按时交付一个能上传照片的系统，却无法支持后续派单和处置喵。

需求分析的作用，就是在编写大量代码之前，以较低成本发现目标不清、范围冲突、规则遗漏和术语歧义喵。

它还帮助产品、开发、测试、运维和业务人员建立共同语言，避免同一个词在不同角色那里代表不同含义喵。

例如，"事件关闭"可能被开发人员理解为数据库状态变为关闭，却可能被业务人员理解为养护完成、复核通过并保留现场证据喵。

### 功能性需求与非功能性需求

功能性需求描述系统需要提供什么能力，也就是系统"要做什么"喵。

例如，"巡检员可以创建路面病害事件""养护负责人可以把事件指派给处置班组""复核人员可以驳回不合格的处置结果"都属于功能性需求喵。

非功能性需求描述系统需要达到怎样的质量或约束，也就是系统"要做到什么程度"喵。

常见非功能性需求包括性能、可靠性、安全性、可用性、兼容性、可维护性和可观测性喵。

例如，"正常网络条件下，事件列表的百分之九十五请求应在两秒内返回""事件照片必须保存至少五年""只有所属辖区的负责人才能派单"都属于非功能性需求喵。

初学者容易把非功能性需求当成锦上添花，但在交通业务中，它们常常决定系统能否真正投入使用喵。

一个功能齐全却经常丢失事件、无法追踪责任或在高峰期严重卡顿的系统，仍然是不合格的系统喵。

### 把模糊诉求转化为可验证需求

需求分析可以从五个问题开始：谁使用、在什么条件下使用、执行什么操作、产生什么结果、受到什么约束喵。

原始诉求是："我们要一个系统管理巡检发现的路面问题"喵。

首先识别角色，可以得到巡检员、养护负责人、处置班组、复核人员和系统管理员喵。

然后识别主流程，可以得到发现问题、创建事件、审核派单、现场处置、提交结果、复核关闭等步骤喵。

接着识别异常流程，例如定位失败、照片上传失败、重复上报、事件超时、处置不合格和错误派单喵。

最后把讨论结果写成可验证的需求条目，例如："巡检员提交事件时，系统必须记录道路编号、桩号或经纬度、病害类型、严重程度、现场照片和上报时间"喵。

还可以补充验收条件："当病害类型、位置或现场照片缺失时，系统拒绝提交并明确提示缺失字段"喵。

这与测试用例设计直接相关，因为一条好需求应该让测试人员知道应准备什么输入、执行什么操作并检查什么结果喵。

如果一条需求无法设计出明确的测试方法，它通常仍然包含"快速""友好""适当"或"尽量"等未量化表述喵。

例如，"系统应快速展示事件"难以验证，而"在十万条事件数据下，常用筛选查询的百分之九十五响应时间不超过两秒"就可以被测试喵。

### 用例与用户故事

用例适合描述角色与系统之间的完整交互过程喵。

一个用例通常包括参与者、前置条件、触发事件、主成功流程、异常流程和完成后的状态喵。

例如，"提交巡检事件"用例的前置条件是巡检员已经登录，主流程包括填写位置、选择病害类型、上传照片并提交，完成后系统生成事件编号并将状态设为"待审核"喵。

异常流程可以说明照片上传失败时不创建正式事件，而是保留草稿并提示巡检员重试喵。

用户故事则用更简洁的方式表达用户价值，常见格式是"作为某类用户，我希望完成某件事，以便获得某种价值"喵。

例如："作为巡检员，我希望通过手机快速上报带定位和照片的路面病害，以便养护中心及时安排处置"喵。

用户故事还需要验收条件，否则它仍然只是一个方向性描述喵。

对应的验收条件可以是："提交成功后生成唯一事件编号""缺少位置或照片时不可提交""断网时允许保存本地草稿"喵。

用例更擅长展开复杂流程，用户故事更适合组织迭代任务，两者可以配合使用，不必二选一喵。

### 从需求推导系统设计

需求确定以后，系统设计要把业务能力转换为软件模块、接口和数据流喵。

模块划分可以先按相对稳定的业务职责进行，例如用户与权限模块、事件管理模块、派单模块、处置模块、复核模块和通知模块喵。

接口定义需要说明模块或系统之间如何交换信息，包括操作名称、输入字段、输出结构、错误情况和权限要求喵。

数据流需要说明数据从哪里产生、经过哪些处理、保存在哪里，以及最终被谁使用喵。

例如，巡检员提交事件后，数据先进入后端 API，后端验证字段和权限，再把事件信息写入数据库，把照片写入对象存储，并向消息队列发布"事件已创建"消息喵。

通知服务消费该消息后，可以向对应辖区的养护负责人发送提醒喵。

这种推导不是为了堆砌技术，而是为了让每个设计决定都能追溯到某条需求喵。

如果业务要求断网时也能记录事件，那么设计中就要考虑移动端草稿和恢复上传机制喵。

如果业务要求事件全过程可追责，那么设计中就要增加不可随意覆盖的状态变更记录或审计日志喵。

如果业务要求高峰期大量照片同时上传，那么设计中就要避免把大文件直接塞进普通关系数据库，并考虑对象存储、限流和异步处理喵。

### 概要设计与详细设计

概要设计描述系统的整体结构，重点回答系统由哪些部分组成、各部分承担什么职责以及它们如何连接喵。

它可能包含系统上下文图、模块图、部署结构、关键数据流和主要技术选型喵。

详细设计继续深入每个模块，说明数据表字段、接口参数、状态转换、类与函数职责、异常处理和关键算法喵。

可以把概要设计理解为城市交通规划图，把详细设计理解为某个路口的信号配时、车道线和设备接线图喵。

概要设计过于模糊会导致团队无法协调，详细设计过早则可能在需求尚未稳定时浪费大量时间喵。

### 贯穿示例的需求分析结果

对于"公路巡检事件上报与处置系统"，可以先确定核心目标：让路面问题从发现到处置关闭形成可追踪的闭环喵。

第一条功能需求可以写为："巡检员能够创建事件，并填写道路编号、位置、病害类型、严重程度、描述和至少一张现场照片"喵。

第二条功能需求可以写为："系统根据事件位置确定所属辖区，并允许该辖区的养护负责人将事件指派给处置班组"喵。

第三条功能需求可以写为："处置人员能够提交处置说明、完成时间和处置后照片，并申请复核"喵。

第四条功能需求可以写为："复核人员能够通过或驳回处置结果，驳回时必须填写原因"喵。

为了避免状态混乱，可以定义事件依次经历"待审核、待处置、处置中、待复核、已关闭"几个主要状态喵。

还需要定义允许的状态转换，例如只有"待复核"事件才能被关闭，已关闭事件不能直接重新编辑原始上报信息喵。

对应的非功能性需求可以包括权限隔离、操作留痕、照片持久保存、接口响应时间和服务可用性喵。

由这些需求可以推导出事件管理、工作流、附件、权限、通知和审计等模块喵。

核心数据至少包括事件、用户、班组、处置记录、附件和状态变更记录喵。

核心接口可以包括创建事件、查询事件、指派事件、开始处置、提交处置结果、复核事件和查询操作历史喵。

### 常见架构分层

前端负责采集用户输入和展示处理进度，可以包括巡检员使用的移动端和管理人员使用的网页端喵。

后端 API 负责权限检查、业务规则、状态转换和数据协调，是系统业务逻辑的主要承载位置喵。

关系数据库适合保存事件、用户、班组和状态记录等结构化数据喵。

对象存储适合保存现场照片和处置后照片，数据库只记录文件标识、地址和元数据喵。

消息队列可以用于发送通知、触发图片分析或生成统计数据，让这些耗时工作不阻塞核心上报流程喵。

对于规模较小的首个版本，可以先采用模块化单体应用，而不是立即拆分许多微服务喵。

架构复杂度应由实际需求推动，因为更多组件也意味着更多部署、监控和故障处理成本喵。

### 面试场景提示

面试官可能会问："拿到一个模糊需求时，你会怎样开始分析"喵。

回答时可以按"明确业务目标、识别角色、梳理主流程与异常流程、区分功能和非功能需求、补充验收条件、确认优先级"的顺序展开，并用公路巡检示例说明喵。

面试官也可能会问："如何保证系统设计没有偏离需求"喵。

回答思路是建立需求到设计的追踪关系，让每个核心模块、接口和数据字段都能对应业务需求，同时通过需求评审、设计评审、原型验证和测试用例共同检查喵。

## 代码示例

下面的 Python 示例把核心需求落实为数据模型、状态规则和服务接口骨架喵。

```python
from dataclasses import dataclass, field
from datetime import datetime, timezone
from enum import Enum
from typing import Protocol
from uuid import uuid4


class EventStatus(str, Enum):
    PENDING_REVIEW = "待审核"
    PENDING_HANDLING = "待处置"
    HANDLING = "处置中"
    PENDING_VERIFICATION = "待复核"
    CLOSED = "已关闭"


@dataclass(frozen=True)
class CreateEventCommand:
    road_code: str
    location: str
    damage_type: str
    severity: str
    description: str
    photo_ids: tuple[str, ...]
    reporter_id: str


@dataclass
class InspectionEvent:
    event_id: str
    road_code: str
    location: str
    damage_type: str
    severity: str
    description: str
    photo_ids: tuple[str, ...]
    reporter_id: str
    status: EventStatus
    assigned_team_id: str | None
    created_at: datetime
    version: int = 1


class EventRepository(Protocol):
    def save(self, event: InspectionEvent) -> None:
        ...

    def get(self, event_id: str) -> InspectionEvent:
        ...


@dataclass
class InMemoryEventRepository:
    events: dict[str, InspectionEvent] = field(default_factory=dict)

    def save(self, event: InspectionEvent) -> None:
        self.events[event.event_id] = event

    def get(self, event_id: str) -> InspectionEvent:
        if event_id not in self.events:
            raise LookupError("事件不存在")
        return self.events[event_id]


class EventService:
    def __init__(self, repository: EventRepository) -> None:
        self.repository = repository

    def create_event(
        self,
        command: CreateEventCommand,
    ) -> InspectionEvent:
        if not command.road_code.strip():
            raise ValueError("道路编号不能为空")
        if not command.location.strip():
            raise ValueError("事件位置不能为空")
        if not command.photo_ids:
            raise ValueError("至少需要一张现场照片")

        event = InspectionEvent(
            event_id=str(uuid4()),
            road_code=command.road_code,
            location=command.location,
            damage_type=command.damage_type,
            severity=command.severity,
            description=command.description,
            photo_ids=command.photo_ids,
            reporter_id=command.reporter_id,
            status=EventStatus.PENDING_REVIEW,
            assigned_team_id=None,
            created_at=datetime.now(timezone.utc),
        )
        self.repository.save(event)
        return event

    def assign_event(
        self,
        event_id: str,
        team_id: str,
    ) -> InspectionEvent:
        event = self.repository.get(event_id)

        if event.status is not EventStatus.PENDING_REVIEW:
            raise ValueError("只有待审核事件可以派单")
        if not team_id.strip():
            raise ValueError("处置班组不能为空")

        event.assigned_team_id = team_id
        event.status = EventStatus.PENDING_HANDLING
        event.version += 1
        self.repository.save(event)
        return event


repository = InMemoryEventRepository()
service = EventService(repository)

event = service.create_event(
    CreateEventCommand(
        road_code="G15",
        location="K120+500",
        damage_type="路面坑槽",
        severity="严重",
        description="坑槽位于行车道，影响车辆通行",
        photo_ids=("photo-before-001",),
        reporter_id="inspector-007",
    )
)

assigned_event = service.assign_event(
    event_id=event.event_id,
    team_id="maintenance-team-03",
)

print(assigned_event.event_id, assigned_event.status.value)
```

`CreateEventCommand` 表示"创建事件"接口需要接收的数据，它直接对应道路编号、位置、病害类型、严重程度、描述、照片和上报人等需求字段喵。

命令对象与持久化模型分开，可以避免调用者直接控制事件编号、初始状态和创建时间等应由系统决定的字段喵。

`InspectionEvent` 是核心领域数据模型，其中 `status` 支持处置流程，`assigned_team_id` 支持派单，`photo_ids` 则对应现场证据要求喵。

照片字段保存的是文件标识而不是二进制内容，这对应"照片放入对象存储、业务数据库保存引用"的设计喵。

`EventStatus` 把允许使用的状态集中定义出来，比在代码中到处填写"处理中"之类的自由文本更不容易出错喵。

`create_event` 中的校验对应需求的验收条件，例如没有位置或现场照片时必须拒绝提交喵。

创建成功后，系统自动生成唯一事件编号，并把状态设置为"待审核"，对应上报流程的完成结果喵。

`assign_event` 不只是修改一个字段，还会检查当前状态，体现了"只有待审核事件可以派单"的业务规则喵。

`EventRepository` 定义了业务层需要的数据访问能力，却没有绑定具体数据库，因此后续可以把内存实现替换为 PostgreSQL 等持久化实现喵。

`version` 字段为并发更新控制预留了位置，可以避免两名负责人同时操作时后提交的数据悄悄覆盖先提交的数据喵。

真实项目还需要补充身份认证、辖区权限、审计日志、事务、附件上传、异常响应和自动化测试，但这些都可以在当前骨架上按需求逐步增加喵。

## 常见误区

第一个误区是把客户说出的第一句话直接当作完整需求喵。

业务人员通常更擅长描述痛点和目标，分析人员需要继续追问角色、场景、规则、异常和验收标准喵。

第二个误区是只记录功能，不记录性能、安全、可靠性和可追踪性等非功能性要求喵。

这些质量要求如果到上线前才考虑，往往会迫使团队大幅修改架构喵。

第三个误区是先选技术，再寻找使用技术的理由喵。

系统是否需要微服务、消息队列或复杂缓存，应由规模、性能、团队协作和可靠性需求决定喵。

第四个误区是把系统设计理解为只画一张架构图喵。

架构图还需要配合模块职责、接口契约、数据模型、状态规则、异常处理和关键决策说明，才能真正指导开发喵。

第五个误区是认为需求一旦确认就永远不能变化喵。

需求可以变化，但变化应经过影响分析和版本管理，团队需要知道它会影响哪些设计、代码、测试和交付计划喵。

## 苏格拉底式检查

假设业务人员提出"严重病害必须立即通知养护负责人"，你认为其中哪些词还需要进一步澄清，才能把它写成可测试的需求喵。

如果不确定，可以分别追问"什么情况算严重""通知给哪个负责人""通过什么渠道通知""多长时间内送达""通知失败怎么办"喵。

再想一想，如果系统规定已关闭事件不能修改，那么巡检员发现位置填写错误时，系统应该直接允许修改、重新打开原事件，还是创建一条更正记录喵。

如果不确定，可以从数据真实性、责任追踪和业务便利性三个角度比较，并思考哪种方案能够保留完整的操作历史喵。

## Quick Summary

- 需求分析回答"系统要解决什么问题以及怎样才算解决成功"，系统设计回答"用什么结构把需求实现出来"喵。
- 功能性需求描述系统要做什么，非功能性需求描述系统要做到什么程度喵。
- 好需求应明确角色、条件、操作、结果和约束，并能够转化为验收条件与测试用例喵。
- 系统设计要从需求推导模块、接口、数据模型和数据流，并区分概要设计与详细设计喵。
- 技术和架构应服务于业务需求，所有关键设计决定都应能够追溯到需求依据喵。
