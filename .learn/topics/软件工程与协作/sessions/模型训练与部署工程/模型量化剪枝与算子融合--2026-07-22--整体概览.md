## Positioning

模型量化、剪枝与算子融合位于模型训练完成到高效部署之间的优化层：它们分别减少数值表示成本、删除冗余计算和降低 GPU 数据搬运开销，并最终通过 TensorRT、`torch.compile` 或自定义 CUDA 算子服务于 GPU、边缘设备和本地大模型部署喵。

## Analogy

可以把模型部署想象成让一家机器人仓库更快地处理货物：权重和激活是货物，GPU 显存是仓库空间，CUDA Core 或 Tensor Core 是工人，而 kernel 是一次被派发的工作任务喵。

量化像是把原本精细而占空间的大包装换成更紧凑的小包装，使同一辆车能运输更多货物喵。剪枝像是检查订单后移除不产生价值的货物，但只有把整箱或固定位置的货物移除，运输系统才容易真正少跑几趟喵。

算子融合则像把称重、贴标签和装箱三个工位合并为一条流水线，货物不必反复进出仓库喵。三种手段共同追求的不是让数学模型“看起来更小”，而是让真实硬件以更少的存储、搬运和计算完成同一项任务喵。

## Core Mechanism

### 1．量化：用更少的比特表示数值

神经网络通常用浮点数保存权重和激活，所谓 FP32、FP16 和 BF16，分别表示使用 32 位、16 位和 16 位的浮点格式喵。FP32 具有 8 位指数和 23 位尾数，精度较高但占用空间较大；FP16 具有 5 位指数和 10 位尾数，精度较细但动态范围较小；BF16 具有 8 位指数和 7 位尾数，因此保留了接近 FP32 的动态范围，却牺牲了部分有效数字精度喵。

INT8 使用 8 位整数，有符号取值范围为 $[-128,127]$，INT4 的有符号取值范围通常为 $[-8,7]$ 喵。一个 FP32 参数需要 4 字节，而 INT8 和 INT4 分别只需要约 1 字节和半字节，因此同样数量的参数可以减少到约四分之一或八分之一的存储体积喵。

低精度能够降低模型权重、激活和 KV cache 的显存占用，也能减少数据从显存搬到计算单元时消耗的带宽喵。如果硬件提供 INT8 或 INT4 Tensor Core，并且推理框架拥有匹配的高效 kernel，那么一次指令还能处理更多数值，从而进一步提高吞吐量喵。

把实数映射成整数时，常用仿射量化公式为：

$$
q=\operatorname{clip}\left(\operatorname{round}\left(\frac{x}{s}\right)+z,\ q_{\min},q_{\max}\right)
$$

对应的反量化公式为：

$$
\hat{x}=s(q-z)
$$

其中 $x$ 是原始实数，$q$ 是量化后的整数，$s$ 是正数 scale，$z$ 是整数 zero-point，$\hat{x}$ 是恢复出的近似实数喵。scale 决定整数相邻刻度在实数空间中的距离，zero-point 则让整数编码能够准确表示实数零点喵。

例如，对于范围为 $[x_{\min},x_{\max}]$ 的非对称 INT8 量化，可以近似取：

$$
s=\frac{x_{\max}-x_{\min}}{q_{\max}-q_{\min}},
\qquad
z=\operatorname{round}\left(q_{\min}-\frac{x_{\min}}{s}\right)
$$

对称量化通常令 $z=0$，并用绝对值最大的数确定 scale，例如 $s=\max(|x|)/127$ 喵。它的计算和实现较简单，但若数据分布明显偏向正数或负数，就可能浪费一部分整数表示范围喵。

量化误差主要来自舍入和截断，因此 scale 的选择非常重要喵。工程中还要决定按整个张量使用一个 scale，还是为不同通道或权重分组分别计算 scale；后者通常更准确，但会增加量化元数据和 kernel 实现复杂度喵。

PTQ 即训练后量化，它在模型训练完成后使用少量校准数据估计数值范围，再直接量化权重和激活喵。PTQ 成本低、流程快，但低比特量化或异常值较多时可能出现明显精度损失喵。

QAT 即量化感知训练，它在训练或微调过程中插入“伪量化”操作，让前向传播模拟舍入和截断误差，同时仍用浮点数完成可微分的参数更新喵。模型因此能够主动适应量化噪声，通常比 PTQ 更准确，但需要额外训练数据、算力和训练稳定性调试喵。

在 LLM 中，权重经常占据主要存储空间，因此常见方案是 weight-only 量化：权重保存为 INT8 或 INT4，而激活仍使用 FP16 或 BF16 喵。执行矩阵乘法时，高效 kernel 会在片上反量化权重并立即参与计算，从而避免把完整浮点权重写回显存喵。

GPTQ 是一种训练后权重量化方法，它按块处理权重，并利用近似二阶信息补偿量化误差喵。AWQ 则利用激活统计识别对输出特别重要的权重通道，通过缩放保护这些显著权重，因此二者都不是简单地对所有权重统一四舍五入喵。

### 2．剪枝：移除模型中的冗余连接或结构

剪枝利用了神经网络通常存在参数冗余这一事实，将贡献较小的权重、通道、神经元、注意力头或网络层删除喵。最常见的基础方法是幅值剪枝，即认为绝对值较小的权重对输出影响相对较弱，并优先把它们设为零喵。

非结构化剪枝可以在权重矩阵中的任意位置产生零值，因而能够获得很高的参数稀疏度，同时较灵活地保留重要连接喵。但是，普通 GPU 的稠密矩阵乘法 kernel 仍会读取这些零值并执行乘加运算，若改用通用稀疏格式，又会引入索引存储、不规则访存和线程负载不均衡等成本喵。

因此，一个模型即使有 80% 的权重等于零，也不代表其端到端延迟一定减少 80% 喵。只有推理框架、稀疏格式和硬件能够共同跳过这些零值时，非结构化稀疏才可能转化为真实加速喵。

结构化剪枝会移除完整通道、卷积核、注意力头、MLP 中间维度甚至网络层喵。剪枝后得到的张量维度直接变小，可以继续使用成熟的稠密矩阵乘法 kernel，因此更容易获得真实加速，但结构约束更强，也更可能损伤模型能力喵。

2:4 半结构化稀疏位于两者之间，它要求每组连续的 4 个权重中恰好有 2 个为零喵。部分 NVIDIA GPU 的稀疏 Tensor Core 能识别这种固定模式，在满足数据类型、维度、布局和 kernel 支持等条件时跳过零值计算，因此比任意稀疏更容易硬件加速喵。

典型剪枝流程是先训练一个稠密模型，再根据幅值、梯度或敏感度评估参数重要性，然后逐步提高剪枝比例，最后对剩余参数进行微调喵。逐步剪枝通常比一次删除大量参数更稳定，因为模型在每个阶段都有机会重新分配表示能力喵。

实际项目不能只报告“参数中有多少零”，还应同时检查导出后的模型尺寸、硬件是否使用稀疏 kernel、端到端延迟、吞吐量、显存峰值和任务精度喵。对于机器人模型，还应关注单帧最坏延迟和功耗，因为平均吞吐量并不能完整描述控制闭环的实时性喵。

### 3．算子融合：减少 kernel 启动和显存往返

在 GPU 上执行一个 PyTorch 算子，通常意味着 CPU 向 GPU 提交一次 kernel launch，GPU 再调度大量线程完成该算子喵。单次启动开销可能不大，但模型含有大量短小算子时，启动、同步和调度成本会不断累积喵。

更关键的问题通常是显存读写喵。若先执行加法，再执行激活函数，朴素实现可能先从显存读取输入、写回加法结果，然后重新读取中间结果、执行激活并再次写回显存喵。

当一个算子的运行速度主要受算术计算能力限制时，它被称为 compute-bound，例如规模足够大的矩阵乘法往往更接近这一类型喵。当速度主要受显存带宽、缓存命中率或数据搬运限制时，它被称为 memory-bound，许多逐元素操作、归一化和小规模算子属于这一类型喵。

算子融合把多个相邻操作编译进同一个 kernel，使中间结果尽量保留在线程寄存器或共享内存中，而不是反复写入全局显存喵。这样既减少 kernel launch 次数，也减少中间张量的读写和分配，因此对 memory-bound 链路尤其有效喵。

在推理模式下，BatchNorm 使用固定的运行均值 $\mu$ 和方差 $\sigma^2$，因此可以折叠进前面的卷积喵。设卷积输出为 $y=Wx+b$，BatchNorm 参数为缩放系数 $\gamma$、偏移 $\beta$ 和稳定常数 $\epsilon$，则：

$$
W'=\frac{\gamma}{\sqrt{\sigma^2+\epsilon}}W
$$

$$
b'=\beta+\frac{\gamma}{\sqrt{\sigma^2+\epsilon}}(b-\mu)
$$

使用 $W'$ 和 $b'$ 的新卷积可以直接产生原来 Conv 加 BatchNorm 的结果，仅存在浮点舍入级别的误差喵。ReLU 不是线性变换，不能像 BatchNorm 一样吸收到卷积权重中，但执行引擎仍可把 Conv、偏置处理和 ReLU 调度为一个融合 kernel，而不物化中间结果喵。

FlashAttention 也体现了融合与减少数据搬运的思想喵。它通过分块计算，把 $QK^\mathsf{T}$、在线 softmax 和与 $V$ 的乘法组织在片上存储中，避免在显存中完整物化巨大的注意力分数矩阵，因此在结果等价的前提下显著降低高带宽显存读写喵。

TensorRT 会在构建推理引擎时进行层融合、常量折叠、精度选择和 kernel 搜索喵。`torch.compile` 也会捕获 PyTorch 计算图，并尝试生成包含多个操作的融合代码，但动态形状、图断裂、不支持的操作和数据依赖控制流都可能限制自动融合效果喵。

一个典型部署流水线是先训练基线模型并保存精度与延迟指标，再实施结构化或 2:4 剪枝并微调恢复精度，然后冻结模型、折叠 BatchNorm 并导出静态计算图喵。接着使用代表性数据进行 PTQ 校准，或在更高精度要求下执行 QAT，再由 TensorRT、`torch.compile` 或自定义 CUDA kernel 完成算子融合和硬件映射喵。

最后必须在目标 GPU 或边缘设备上测量任务精度、模型大小、峰值显存、预热后延迟、吞吐量和功耗喵。三种优化会相互影响，例如剪枝改变权重分布，量化改变可融合的数据类型，而融合 kernel 又可能只支持特定稀疏模式，因此部署优化通常是迭代过程喵。

## Code Example

下面的示例仅依赖 PyTorch，并分别演示对称 INT8 量化与反量化、Linear 权重幅值剪枝，以及推理模式下 Conv 与 BatchNorm 的手动折叠喵。

```python
import torch
import torch.nn as nn

torch.manual_seed(42)

# 1. 对称 INT8 量化与反量化
x = torch.tensor([-3.20, -1.10, -0.08, 0.00, 0.75, 2.60, 4.00])

# 对称量化通常使用 [-127, 127]，让正负范围完全对称并令 zero_point=0
qmin, qmax = -127, 127
zero_point = 0
scale = x.abs().max() / qmax

q = torch.round(x / scale).clamp(qmin, qmax).to(torch.int8)
x_dequant = scale * (q.to(torch.float32) - zero_point)

print("=== 对称 INT8 量化 ===")
print("原始值:", x)
print("量化值:", q)
print("反量化:", x_dequant)
print("最大绝对误差:", (x - x_dequant).abs().max().item())
print("均方误差:", torch.mean((x - x_dequant) ** 2).item())


# 2. 对 Linear 权重做幅值剪枝
linear = nn.Linear(8, 4, bias=False)
weight = linear.weight.detach().clone()

prune_ratio = 0.40
flat_weight = weight.flatten()
num_to_prune = int(flat_weight.numel() * prune_ratio)

# 找出绝对值最小的 num_to_prune 个权重
pruned_indices = torch.argsort(flat_weight.abs())[:num_to_prune]
flat_mask = torch.ones_like(flat_weight)
flat_mask[pruned_indices] = 0

mask = flat_mask.view_as(weight)
pruned_weight = weight * mask
sparsity = (pruned_weight == 0).float().mean().item()

print("\n=== Linear 幅值剪枝 ===")
print("权重总数:", weight.numel())
print("置零数量:", int((pruned_weight == 0).sum().item()))
print(f"稀疏度: {sparsity:.2%}")


# 3. 手动折叠 Conv2d 与 BatchNorm2d
conv = nn.Conv2d(
    in_channels=3,
    out_channels=4,
    kernel_size=3,
    padding=1,
    bias=False,
)
bn = nn.BatchNorm2d(4)

# 设置非平凡的推理期 BatchNorm 参数，便于验证折叠公式
with torch.no_grad():
    bn.weight.copy_(torch.tensor([0.8, 1.1, 0.9, 1.2]))
    bn.bias.copy_(torch.tensor([0.1, -0.2, 0.3, 0.0]))
    bn.running_mean.copy_(torch.tensor([0.5, -0.4, 0.2, 0.1]))
    bn.running_var.copy_(torch.tensor([1.2, 0.7, 1.5, 0.9]))

conv.eval()
bn.eval()

gamma = bn.weight
beta = bn.bias
mean = bn.running_mean
var = bn.running_var
eps = bn.eps

# alpha 对应 gamma / sqrt(var + eps)
alpha = gamma / torch.sqrt(var + eps)

# 原卷积没有 bias，因此把 b 视为全零
original_bias = torch.zeros_like(mean)
fused_weight = conv.weight * alpha.reshape(-1, 1, 1, 1)
fused_bias = beta + (original_bias - mean) * alpha

fused_conv = nn.Conv2d(
    in_channels=3,
    out_channels=4,
    kernel_size=3,
    padding=1,
    bias=True,
)

with torch.no_grad():
    fused_conv.weight.copy_(fused_weight)
    fused_conv.bias.copy_(fused_bias)

fused_conv.eval()

input_tensor = torch.randn(2, 3, 8, 8)

with torch.no_grad():
    original_output = bn(conv(input_tensor))
    fused_output = fused_conv(input_tensor)

fusion_error = (original_output - fused_output).abs().max().item()

print("\n=== Conv + BatchNorm 折叠 ===")
print("原始输出形状:", tuple(original_output.shape))
print("融合输出形状:", tuple(fused_output.shape))
print("最大绝对误差:", fusion_error)
```

第一段代码令 zero-point 为 0，并用输入绝对值最大值除以 127 得到 scale 喵。INT8 类型本身的范围是 $[-128,127]$，但对称量化常主动使用 $[-127,127]$，以便正负两侧具有相同的最大幅值喵。

量化值只是整数编码，必须与 scale 和 zero-point 一起解释才具有原来的数值意义喵。反量化结果不会与原值完全一致，因为多个连续实数会映射到同一个离散整数刻度喵。

第二段代码将 Linear 权重按绝对值排序，并把最小的 40% 置零喵。这里得到的是非结构化稀疏权重，它展示了剪枝机制，却不会自动让普通 `nn.Linear` 跳过零值计算喵。

第三段代码使用 BatchNorm 的运行均值和运行方差计算融合后的卷积参数，并比较融合前后的输出喵。该折叠只适用于 BatchNorm 的推理行为，因为训练模式下 BatchNorm 会根据当前批次重新计算均值和方差，不能提前固化进卷积参数喵。

## Common Misconceptions

- **误区一：量化一定会让模型精度大幅下降喵。** 合理的校准、逐通道量化、混合精度和 QAT 可以把误差控制得很小，而模型不同层对低精度的敏感度也并不相同喵。

- **误区二：权重剪成零以后，模型一定会按稀疏比例加速喵。** 普通稠密 kernel 仍可能读取零值并执行乘法，只有结构化裁剪或受硬件支持的稀疏模式才容易转化为真实延迟收益喵。

- **误区三：算子融合是用近似公式替换原始计算喵。** 大多数融合保持相同的数学语义，只改变操作的调度、数据驻留位置和中间结果存储方式，差异通常只来自浮点运算次序与舍入误差喵。

- **误区四：INT8 量化就是把浮点数直接四舍五入成整数喵。** 实际量化还要选择 scale、zero-point、裁剪范围、通道粒度和校准方法，否则较大的数会溢出，较小的数也可能全部挤在少数几个整数刻度上喵。

## Socratic Check

**问题一：一个 Linear 层有 90% 的权重被置零，但仍通过普通 FP16 稠密矩阵乘法执行，它为什么可能几乎没有加速喵？**

参考答案：因为普通稠密 kernel 不理解这些零值的特殊含义，仍会读取完整矩阵并执行对应的乘加操作喵。若想获得真实加速，需要把稀疏性转换为更小的结构化矩阵，或使用与稀疏格式及目标硬件匹配的 kernel 喵。

**问题二：连续的 Add、ReLU 和 Clamp 都不包含大量乘法，为什么融合后仍可能明显变快喵？**

参考答案：这些逐元素操作往往是 memory-bound，时间主要消耗在读取输入、写出中间张量和多次启动 kernel 上，而不是算术计算本身喵。融合后，中间值可以留在寄存器中连续处理，只需较少的全局显存往返和 kernel launch 喵。

## Quick Summary

- 量化用更少的比特保存权重和激活，主要收益来自降低显存容量与带宽压力，并可能借助低精度硬件提高计算吞吐量喵。
- 剪枝删除冗余参数，但参数为零不等于硬件会跳过计算，结构化和受支持的半结构化稀疏更容易带来真实加速喵。
- 算子融合通过减少 kernel launch 和中间张量的显存读写加速计算，尤其适合逐元素操作等 memory-bound 链路喵。
- 三种技术需要与目标硬件、编译器和推理框架共同设计，并以设备上的精度、延迟、显存和功耗实测结果为准喵。

### Q1（2026-07-22）大包装为何更精细与量化是否等于保留小数位

**问：** 为什么大包装会更加精细？而且量化好像是什么精确到小数点后几位吧

**答：**

先说类比里的"大包装更精细"喵。这里的"大"不是指包装本身做工精细，而是指"包装上能写下的信息更多"喵：一个数用越多比特来存，能区分的不同数值就越多，相邻两个可表示数值之间的间隔就越小喵。

换一个更准确的类比：FP32 像一个刻度很密的大量杯，占地方，但能精确读到毫升；FP16 像一个只有粗刻度的小量杯，省地方，但两条刻度线之间的液面高度都只能读成同一个数喵。具体到浮点格式，"刻度密度"由尾数位数决定：FP32 有 23 位尾数，大约对应 7 位有效十进制数字；FP16 只有 10 位尾数，大约只剩 3~4 位有效数字喵。

所以"占空间大"和"精细"是同一枚硬币的两面：想要更密的刻度，就必须花更多比特去记录，这正是量化想省掉的成本喵。

再说"量化是不是精确到小数点后几位"喵。你的直觉方向是对的：日常说的"保留两位小数"确实就是一种量化——把连续值舍入到间距 0.01 的格子上喵。但模型量化和它有三个关键区别喵。

第一，格子间距不是按十进制位数定的，而是由 scale 决定，并且 scale 会根据数据的实际范围自适应喵。比如讲解代码里 scale = 4.0 / 127 ≈ 0.0315，格子间距是这个"不整"的数，而不是 0.01 或 0.001 喵。这样做的好处是：不管这层权重的数值范围是 [-4, 4] 还是 [-0.02, 0.02]，256 个格子总能刚好铺满整个范围，不浪费喵。

第二，也是最重要的：目的不同喵。把 3.14159 四舍五入成 3.14 之后，它仍然以 FP32 存储，还是占 4 字节，显存和带宽一点都没省喵。模型量化的关键动作是"换存储格式"——把数值编码成 INT8/INT4 整数，每个数只占 1 字节或半字节，硬件搬运和计算的数据量才真正下降喵。舍入只是这个映射过程的副作用（也是误差来源），而不是目的喵。

第三，格子总数是有限的，而且有裁剪范围喵。INT8 只有 256 个刻度，落在校准范围之外的值会被直接截断到边界喵；而"保留两位小数"对任意大的数都适用，没有格子数量的上限喵。

顺带一提，"精确到小数点后几位"其实更接近"定点数"的思路（格子间距固定），而浮点数本身像科学计数法，它的格子间距是随数值大小变化的：数越大，相邻可表示数之间的间隔也越大喵。整数量化则是在一个选定范围内铺上均匀的格子，这也是为什么 scale 的选取如此重要喵。
