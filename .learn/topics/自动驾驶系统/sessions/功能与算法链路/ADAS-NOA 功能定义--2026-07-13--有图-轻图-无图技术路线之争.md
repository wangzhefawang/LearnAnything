## Positioning

前篇第 12 节已经把有图、轻图、无图的权衡压缩成一句话，本次把它展开成一条工程主线：地图到底是系统真值、场景先验，还是在线感知之后的校验参考喵。

## Analogy

可以把三条路线想成去陌生城市旅行喵。

有图路线像拿着一本逐米标注的深度攻略，连哪个路口有几条队、哪个红绿灯管哪条道都提前写好喵。

轻图路线像带着普通地图 App 和几条朋友提醒，知道大方向和关键路口，但真正走到现场还要看路牌、车流和临时封路喵。

无图路线像只给你目的地和大致路线，主要靠你现场观察路牌、车道线、红绿灯和周围人的走法来实时理解道路喵。

这三个旅行者不只是装备不同，更是把风险放在了不同地方：攻略怕过期，普通地图怕不够细，现场观察怕反应慢或看漏喵。

## Core Mechanism

**1. 三条路线到底指什么喵。**
有图通常指强依赖 HD map，也就是高精地图喵。
HD map 不是导航地图的放大版，而是包含厘米级或车道级几何、车道拓扑、停止线、斑马线、灯组绑定、限速、坡度、曲率、匝道连接关系等结构化信息的自动驾驶先验喵。
如果系统知道「当前位置是第 2 条直行车道，前方 120 米停止线之后是五岔路口，本车道受右上角这组圆灯控制」，规划和规则理解会容易很多喵。
轻图通常指 SD map 或导航级地图加少量车道级先验，再由在线感知补全细节喵。
SD map 能告诉系统道路级路径、转向意图、路口大致形状和导航约束，但不会稳定提供完整的厘米级车道真值喵。
轻图系统会把地图当作路线提示和粗粒度先验，再用车端感知确认当前到底有几条车道、哪条可行、哪里有施工或临停喵。
无图通常指不依赖离线 HD map 做车道级决策，而是靠纯在线感知实时建图喵。
这里的在线建图包括 BEV 空间表达、在线车道拓扑推理、路沿和停止线检测、交通灯与车道关系推理、occupancy 占用空间估计等能力喵。
但无图不等于完全不用任何地图喵。
量产 NOA 仍然通常会用导航地图做目的地路径、道路级引导和剩余路线管理，只是不把离线 HD map 当成车道级真值喵。

**2. HD map 路线的收益与代价喵。**
HD map 的收益首先是超视距先验喵。
摄像头和雷达只能看见传感器视野内的东西，而地图可以提前告诉系统前方有急弯、匝道分叉、车道消失、坡顶、隧道或复杂路口喵。
第二类收益是灯与车道绑定，因为城市路口里识别出一个红灯并不够，系统还要知道这个红灯管不管本车道喵。
第三类收益是复杂路口拓扑，五岔路口、环岛、辅路汇入、潮汐车道和连续匝道都需要结构化骨架喵。
第四类收益是定位匹配底座，车辆可以把感知到的车道线、路牌、杆件、路沿与 HD map 匹配，从而获得更稳定的车道级定位喵。
但 HD map 的第一类代价是鲜度喵。
如果道路施工、临时改道、锥桶占道或车道重画已经发生，而地图还停留在旧版本，错误先验可能比没有先验更危险喵。
第二类代价是覆盖，因为城市扩张、新区道路、临时管制和小路结构变化很快，采集、制图、质检、发布的周期很难追上全国所有道路喵。
第三类代价是成本，因为 HD map 需要采集车队、传感器设备、标注制图产线、质量审核、版本发布和城市级运营喵。
第四类代价是中国特有的合规约束，包括甲级测绘资质、导航电子地图制作资质、地图审核、审图号、敏感区域处理、坐标偏转加密和更新审批喵。
这些合规要求不是纯技术问题，却会直接影响城市 NOA 的开城速度、数据闭环和地图更新频率喵。
所以国内城市 NOA 从强有图走向轻图或无图，不只是算法自信增强，也有鲜度、覆盖、成本和合规共同推动喵。

**3. 无图路线把难度转移到了哪里喵。**
无图路线并没有让问题消失，而是把难度从离线地图生产转移到了车端在线理解喵。
第一块难度是在线车道拓扑重建，系统要先检测车道线、路沿、停止线、箭头和可行驶区域，再推理这些元素如何组成车道级拓扑喵。
在路口无线区域、车道线磨损、施工改道、雨夜反光或多条旧线重叠时，系统还要脑补虚拟车道和可通行走廊喵。
第二块难度是时序记忆与遮挡补全，因为某一帧看不见的停止线、被大车挡住的车道边界、刚刚被前车遮住的路口箭头，都需要跨帧记忆喵。
第三块难度是灯与车道绑定要现场推理，系统要结合灯组位置、车道箭头、停止线、前车行为、路口几何和导航意图来判断喵。
第四块难度是验证负担变大，因为无图路线更像每个路口都是新路口，系统要证明自己面对未知拓扑仍然稳健喵。
BEV 加 Transformer 在线建图的直觉是：环视相机先提取多视角图像特征，再投到俯视空间，形成车辆周围的一张鸟瞰语义画布喵。
Transformer 负责在这张画布上建立长距离关系，例如把断裂车道线连起来、把停止线和灯组关联起来、把跨帧出现的同一条边界合并起来喵。
occupancy 则补上「哪里被占用、哪里可能可通行、哪里是不确定空间」这类更直接服务规划的表达喵。

**4. 轻图作为工程折中喵。**
轻图路线承认两个事实：完全依赖 HD map 很难规模化，完全抛开先验又会让在线系统背太多负担喵。
因此轻图通常用 SD map 提供路径级引导，例如「前方路口右转」「两公里后进入匝道」「当前道路大致向东北延伸」喵。
车道级细节则交给在线感知，例如当前到底有几条车道、右转车道是否被公交车挡住、地面箭头是否允许直行和右转喵。
轻图路线的关键挑战是 SD 与 HD 粒度不匹配喵。
导航说「前方右转」只是道路级意图，但在线感知可能只看到直行车道，或者右转车道被围挡临时封闭喵。
此时系统不能把导航意图硬压成右转轨迹，而要让地图先验降级为提示，不再把它当真值喵。
一个工程化说法是：地图给候选，感知给证据，规划做风险约束下的仲裁喵。
如果感知证据不足以支持右转，系统可以减速观察、继续直行重规划、提示用户接管，或者抑制危险变道喵。

**5. 行业演进案例喵。**
特斯拉长期坚持以视觉和在线神经网络能力为核心，不以城市级 HD map 作为主要依赖喵。
这条路线的代表意义不是「所有人都该只用摄像头」，而是说明无强高精图路线可以把规模化压力从地图生产转向车端模型和数据闭环喵。
小鹏早期城市 XNGP 更强调高精地图和按城市开放能力，后来逐步走向弱化地图依赖、扩大无图化可用范围的方向喵。
这个演进很适合作为面试表达案例，因为它体现了国内量产 NOA 从「先开重点城市」走向「全国更多道路泛化」的产品压力喵。
华为、理想等中国厂商也在城市 NCA、城市 NOA 的表达中逐渐强调不依赖高精地图或轻图泛化喵。
不要在面试里硬背某家厂商的具体版本号、城市数或发布日期，因为这些数字变化快，也容易说错喵。
更稳妥的总结是：共同趋势是地图从真值降级为先验，在线感知升级为主感知喵。

**6. 置信度仲裁与记忆机制喵。**
地图先验与在线感知冲突时，真正的问题不是简单问谁更高级，而是问当前谁更可信、错了会造成什么风险喵。
地图侧需要鲜度置信度，例如这个路段是否最近更新、是否有众源异常回传、是否处于施工高发区域、是否存在历史冲突记录喵。
感知侧需要观测置信度，例如车道线是否清晰、遮挡是否严重、跨帧是否一致、多个传感器是否支持同一结论喵。
如果地图和感知一致，系统可以正常执行车道选择、变道或路口通行喵。
如果地图说有三条车道，在线感知稳定看到两条车道，并且 occupancy 显示第三条被围挡占用，系统应提高感知权重，降低地图权重喵。
如果地图和感知冲突且双方都弱，系统应保守化，抑制高风险动作，例如取消主动变道、减速观察或退回基础车道内控制喵。
这与前篇 ODD 和降级策略是连着的：冲突不是单点判断，而是会影响功能是否仍在可用边界内喵。
众源更新是解决鲜度的一条工程路径喵。
用户车队在日常行驶中发现车道数变化、限速变化、道路封闭或新建路口，可以把匿名化、合规处理后的片段回传，用于更新轻量先验或触发地图质检喵。
记忆行车或通勤模式也可以理解为个人或车队尺度的轻量先验喵。
系统记住用户常走路线上的车道选择、易错路口、匝道入口和临停高发点，下次行驶时作为提示，但仍需要现场感知确认喵。
这类记忆不能变成新的绝对真值，否则只是把旧 HD map 的鲜度问题缩小到用户路线里喵。

**7. 面试视角总结喵。**
有图、轻图、无图不是简单的技术淘汰赛喵。
它们本质上是在重新分配三件事的成本和风险：先验从哪里来、鲜度如何保障、在线能力能承担多少喵。
HD map 把复杂拓扑提前结构化，降低车端理解压力，但付出覆盖、鲜度、成本和合规代价喵。
无图把规模化潜力做大，但要求在线感知、时序建图、拓扑推理、灯车道绑定和验证体系都更强喵。
轻图位于中间，把地图当路线级先验，把车道级真值交给在线系统确认喵。
面试时可以用一条主线收束：地图角色从真值到先验，再到校验参考，在线感知从辅助校验升级为主感知喵。
如果再补一句工程判断，可以说冲突场景下系统不该争论地图和感知谁永远正确，而该用置信度、风险等级和降级策略决定当前动作喵。

## Code Example

```python
from dataclasses import dataclass


@dataclass
class LaneEvidence:
    lane_count: int
    confidence: float


def clamp(value: float, low: float = 0.0, high: float = 1.0) -> float:
    return max(low, min(high, value))


def arbitrate_lane_topology(
    map_evidence: LaneEvidence,
    perception_evidence: LaneEvidence,
) -> dict[str, object]:
    # 地图鲜度置信度越低，地图越应该从真值退回先验喵。
    map_score = clamp(map_evidence.confidence)

    # 感知置信度代表现场证据是否稳定，而不是模型是否永远正确喵。
    perception_score = clamp(perception_evidence.confidence)

    if map_evidence.lane_count == perception_evidence.lane_count:
        return {
            "trusted_lane_count": perception_evidence.lane_count,
            "source": "map_and_perception_agree",
            "strategy": "normal_noa",
            "map_weight": round(map_score, 2),
        }

    if perception_score >= 0.75 and perception_score - map_score >= 0.15:
        return {
            "trusted_lane_count": perception_evidence.lane_count,
            "source": "perception_overrides_stale_map",
            "strategy": "trust_perception_reduce_map_weight",
            "map_weight": round(map_score * 0.4, 2),
        }

    if map_score >= 0.80 and map_score - perception_score >= 0.25:
        return {
            "trusted_lane_count": map_evidence.lane_count,
            "source": "map_prior_with_weak_perception",
            "strategy": "keep_lane_only_no_aggressive_lane_change",
            "map_weight": round(map_score, 2),
        }

    return {
        "trusted_lane_count": min(map_evidence.lane_count, perception_evidence.lane_count),
        "source": "conflict_uncertain",
        "strategy": "conservative_suppress_lane_change",
        "map_weight": round(map_score * 0.5, 2),
    }


cases = [
    (
        "一致场景",
        LaneEvidence(lane_count=3, confidence=0.90),
        LaneEvidence(lane_count=3, confidence=0.86),
    ),
    (
        "施工改道，感知更强",
        LaneEvidence(lane_count=3, confidence=0.45),
        LaneEvidence(lane_count=2, confidence=0.88),
    ),
    (
        "雨夜遮挡，双方都弱",
        LaneEvidence(lane_count=3, confidence=0.55),
        LaneEvidence(lane_count=2, confidence=0.50),
    ),
]


for name, map_data, perception_data in cases:
    decision = arbitrate_lane_topology(map_data, perception_data)
    print(f"{name}: {decision}")
```

这个例子只模拟一个很小的仲裁问题：地图说有几条车道，在线感知说有几条车道，系统该如何采信喵。

`LaneEvidence` 里有两个字段，`lane_count` 表示车道数判断，`confidence` 表示证据可信度喵。

对地图而言，置信度更像鲜度和定位匹配质量；对感知而言，置信度更像现场观测质量和跨帧稳定性喵。

当地图和感知一致时，系统进入 `normal_noa`，因为先验和现场证据互相支持喵。

当二者冲突且感知明显更强时，函数返回 `trust_perception_reduce_map_weight`，意思是信现场，同时降低旧地图权重喵。

当地图很强但感知较弱时，系统也不应激进执行导航动作，而是保守地保持车道内控制，避免在看不清时按地图强行变道喵。

当双方都弱且冲突时，函数返回 `conservative_suppress_lane_change`，这体现了地图是先验而不是真值，感知也不是无条件真值喵。

这段代码和前两篇的降级思想衔接：冲突不可怕，真正危险的是冲突后仍然高置信地执行高风险动作喵。

## Common Misconceptions

1. **误以为无图等于不用任何地图喵。** 无图通常是不依赖离线 HD map 做车道级真值，导航地图仍然常用于路径级引导喵。

2. **误以为有图路线已经死亡喵。** HD map 在复杂拓扑、灯车道绑定和定位匹配上仍有价值，只是它从绝对真值逐渐退到强先验喵。

3. **误以为无图只是为了省钱喵。** 成本是原因之一，但更核心的是鲜度、覆盖、合规和全国泛化压力喵。

4. **误以为轻图就是低配方案喵。** 轻图的难点在于先验与在线感知如何融合，做得好反而更贴近量产规模化需求喵。

5. **误以为地图和感知冲突时必须固定相信一方喵。** 工程系统更常用置信度、风险等级和动作类型来仲裁，高风险动作尤其要保守喵。

## Socratic Check

1. 如果地图显示前方有三条车道，但在线感知连续 3 秒只看到两条车道，并且右侧有锥桶占用，系统应该直接按地图变道吗喵。

   提示：不应该直接按地图变道，地图此时应降级为先验，系统应提高感知权重，减速观察或抑制高风险变道喵。

2. 为什么「无图」系统的验证压力可能比「有图」系统更大喵。

   提示：因为它不能只验证已开城地图覆盖内的已知拓扑，而要证明在线感知和拓扑推理面对未知路口、遮挡、施工和长尾组合时仍能保守可靠喵。

## Quick Summary

- 有图依赖 HD map 提供车道级拓扑、灯车道绑定、坡度曲率和定位匹配底座，但受鲜度、覆盖、成本和合规约束喵。
- 轻图用 SD map 做路径级引导，把车道级细节交给在线感知确认，本质是把地图从真值降级为提示喵。
- 无图不是不用地图，而是不依赖离线 HD map 做车道级真值，难度转移到 BEV 在线建图、拓扑推理、时序记忆和 occupancy 表达喵。
- 地图与感知冲突时不该固定相信一方，而要根据鲜度置信度、感知置信度和动作风险做仲裁喵。
- 面试表达可用「地图角色从真值 → 先验 → 校验，在线感知从辅助 → 主感知」作为主线喵。
