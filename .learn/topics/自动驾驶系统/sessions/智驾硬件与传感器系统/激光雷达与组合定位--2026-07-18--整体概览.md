## Positioning

激光雷达负责测量车辆周围的三维几何结构，组合定位负责回答车辆此刻在世界中的位置与姿态，两者共同构成自动驾驶感知和规划所依赖的时空基准喵。

## Analogy

激光雷达像站在黑暗房间里快速抛出成千上万个弹力球，再根据每个球返回的时间和方向还原墙壁、家具与行人的轮廓喵。

一次完整扫描并不是瞬间拍下的照片，更像一边走动一边用画笔描摹房间，因此车辆自身的运动会把原本笔直的物体画弯，需要根据每一笔落下时的姿态进行校正喵。

组合定位则像一支由三类向导组成的队伍：GNSS 偶尔看路牌确认绝对位置，IMU 时刻感受转弯和加减速，轮速计不断数车轮转了多少圈喵。

路牌看不见时，队伍仍能靠步数和转向继续前进，但误差会逐渐积累；重新看到可信路牌后，队伍便能修正此前的偏差喵。

## Core Mechanism

### 1. 激光雷达如何测出距离喵

飞行时间法（Time of Flight，ToF）发射短激光脉冲并测量往返时间，距离近似等于光速乘以往返时间的一半喵。

雷达同时记录激光发射方向和回波强度，于是每次有效回波都能转换成一个带有三维坐标、强度和时间戳的点喵。

调频连续波（Frequency-Modulated Continuous Wave，FMCW）激光雷达持续发射频率随时间变化的光，并比较发射光与回波的频率差喵。

这个频率差可以用于估计距离，结合多普勒频移还可能直接得到目标径向速度，但其光学相干检测、芯片制造和信号处理通常更复杂喵。

物体并不保证返回一个干净回波，距离、入射角、表面反射率、雨滴以及接收器阈值都会影响一个点能否被记录喵。

因此，点云不是物体表面的均匀复制品，而是由扫描角度、光学能力、材料特性和环境共同筛选出的采样结果喵。

### 2. 机械式、半固态与固态扫描喵

机械式激光雷达通过旋转整个测距组件或主要光学部件覆盖大范围视场，容易获得规律的环形扫描和较宽的水平视野喵。

旋转结构也会带来体积、振动、磨损、密封和造型集成方面的挑战，因此更常见于研发平台或对全向视野要求较强的系统喵。

半固态激光雷达通常保留部分微型运动结构，例如转镜、棱镜或 MEMS 微振镜，并让激光器和接收器主体保持固定喵。

它能在尺寸、成本和可靠性之间取得折中，但扫描轨迹可能不均匀，算法必须理解其实际扫描图案，而不能假设点总是规则排列喵。

固态方案希望进一步减少宏观运动部件，例如采用光学相控阵或闪光式成像，但视场、功率密度、散热、探测距离和量产一致性仍需共同权衡喵。

所以「固态」并不自动等于距离更远、点更多或性能更好，真正需要比较的是目标场景下的有效探测能力和系统成本喵。

### 3. 点云为什么会发生运动畸变喵

一帧点云通常由一段时间内依次发射的激光束组成，而不是在同一个时刻同时得到的所有点喵。

假设车辆在 100 ms 的扫描周期内持续转弯，帧首和帧尾的点实际上是在不同车辆姿态下测得的，把它们直接当作同一时刻的数据就会让墙面扭曲、车辆拉长喵。

运动畸变补偿也称去畸变（Deskew），其基本做法是利用 IMU、轮速或定位系统估计扫描期间的连续运动轨迹喵。

系统依据每个点的时间戳，把它从采样时刻的传感器姿态变换到统一参考时刻，例如整帧开始、结束或中间时刻喵。

这个过程像把一卷边走边拍的视频逐帧对齐后再拼成全景图，轨迹越准确、时间戳越可靠，拼接后的几何结构就越稳定喵。

如果时间偏差或外参误差存在，即使补偿算法本身完全正确，也可能把直杆补成弯杆，因此去畸变同时依赖定位、同步与标定喵。

### 4. 点云特性与安装位置的工程取舍喵

较高的车顶位置通常能减少车身遮挡并扩大远处可见区域，但会增加整车高度、风阻、造型难度和暴露风险喵。

较低的保险杠或翼子板位置更容易观察路沿与近处障碍物，也更容易被泥水、积雪、飞石和前车尾气污染喵。

前向安装可以把有限的光学资源集中到高速行驶方向，但车辆侧后方可能形成盲区；多雷达环绕能改善覆盖，却会增加成本、带宽、标定和维护压力喵。

工程上不能只问「能看多远」，还要检查不同距离和高度上的视场、车身自遮挡，以及相邻传感器是否具有足够重叠区域喵。

重叠区域既能减少单个传感器失效带来的风险，也可用于传感器间标定和一致性检查，但重复覆盖过多又会浪费点数与算力喵。

脏污检测、加热除雾、清洗装置、排水结构和故障诊断同样属于安装设计，因为一个理论视场很好的雷达，被泥膜遮挡后并不会继续提供可靠点云喵。

### 5. GNSS、RTK、IMU 与轮速分别提供什么喵

GNSS 通过卫星信号估计全球坐标下的绝对位置，优点是误差通常不会像纯积分那样随行驶时间无限增长喵。

建筑反射造成的多路径效应、卫星遮挡、电离层误差和接收机噪声会让 GNSS 发生跳变或失锁，因此它更像间歇出现的绝对锚点，而不是连续无误的真值喵。

实时动态差分（Real-Time Kinematic，RTK）利用基准站或网络提供的改正数据，并分析载波相位来消除部分共同误差喵。

当载波模糊度成功固定且观测条件良好时，RTK 可以达到厘米级精度，但隧道、城市峡谷、通信中断和卫星几何恶化仍会使解退化甚至消失喵。

IMU 测量三轴角速度和比力，更新频率高，能快速捕捉颠簸、转弯和加减速喵。

然而零偏和噪声会被积分为速度、位置与姿态误差，尤其微小的姿态误差会把重力的一部分误认为车辆加速度，从而迅速放大位置漂移喵。

轮速计根据车轮转速和轮胎半径估计纵向速度，在平整路面和正常附着条件下能有效约束 IMU 的速度漂移喵。

轮胎磨损、胎压变化、急刹打滑、转弯时内外轮速差以及车辆侧滑都会破坏「轮子转动等于车辆前进」这一假设喵。

### 6. 为什么需要滤波器融合喵

融合的目标不是简单平均多个数值，而是根据各传感器的误差模型、当前可用性和估计不确定度，判断此刻应该更相信谁喵。

卡尔曼滤波器可以类比为一名边开车边维护账本的领航员，账本不仅记录「我在哪里」，还记录「我对这个答案有多不确定」喵。

在预测阶段，系统利用 IMU 和轮速把上一时刻的状态推进到当前时刻，同时让不确定度随着积分和模型误差增长喵。

在校正阶段，GNSS 或其他绝对观测到来后，系统计算预测与观测之间的残差，再依据双方的不确定度决定修正幅度喵。

如果 GNSS 精度良好而预测已经很不确定，校正幅度会较大；如果 GNSS 正受到多路径干扰而预测仍连续稳定，滤波器就应谨慎使用该观测喵。

扩展卡尔曼滤波器（Extended Kalman Filter，EKF）会在当前估计附近对车辆运动和传感器观测的非线性关系做局部线性化喵。

误差状态卡尔曼滤波器（Error-State Kalman Filter，ESKF）则维护一条名义运动轨迹，并让滤波器估计这条轨迹的小误差，再把误差注入名义状态喵。

ESKF 像先让一名跑者按当前方向持续前进，再让教练只记录「向左偏了多少、速度快了多少」，这对旋转、IMU 偏置和高频积分尤其自然喵。

工程系统还会检查残差是否异常，避免把突然跳到道路另一侧的 GNSS 当成可靠锚点强行写入状态喵。

### 7. GNSS 失效时会怎样退化喵

车辆进入隧道后，滤波器可以继续使用 IMU 和轮速进行航位推算，也就是根据上一位置和之后的相对运动推测当前位置喵。

此时位置不一定立刻明显错误，但由于系统失去了绝对锚点，协方差会逐步增大，意味着系统越来越不确定喵。

直线匀速行驶时，轮速能够较好地约束前进距离；连续弯道、坡道、低附着路面或长时间停车再起步会让偏航角、轮胎尺度和 IMU 零偏误差更快显现喵。

这种退化本质上没有一个固定的「安全坚持时间」，因为误差增长速度取决于 IMU 等级、轮胎状态、车辆运动和标定质量喵。

GNSS 恢复后，滤波器会用新的绝对观测逐步拉回位置，但工程上还要防止瞬间跳变影响规划控制，并确认恢复信号不是隧道出口处的多路径异常值喵。

激光雷达地图匹配或视觉定位可以在 GNSS 缺失时提供额外的绝对或半绝对约束，让系统不必只靠航位推算坚持喵。

### 8. 时间同步为什么是融合前提喵

融合算法默认不同传感器描述的是同一个物理时刻，但各设备具有不同采样频率、内部时钟、传输延迟和处理延迟喵。

车辆以 20 m/s 行驶时，10 ms 的时间偏差就对应约 20 cm 的纵向位移，转弯时还会同时产生方向误差喵。

硬件触发可以用同一电信号启动多个设备采样，从源头约束采样时刻；它像摄影棚里让多台相机听同一个快门指令喵。

GNSS 的每秒脉冲（Pulse Per Second，PPS）可以提供精确的秒边界，再配合时间报文把设备本地时钟对齐到统一时间基准喵。

时间戳对齐还要明确时间戳表示的是「开始曝光」「激光发射」「数据接收」还是「算法输出」，否则相同数值未必代表相同物理事件喵。

软件插值和缓存可以把异步数据对齐到目标时刻，但软件无法完全补救未知且不断变化的硬件延迟喵。

### 9. 坐标系与外参为什么同样重要喵

传感器坐标系描述点相对于激光雷达或 IMU 的位置，车身坐标系描述它相对于车辆参考点的位置，世界坐标系则用于表达车辆在地图或地球上的位置喵。

坐标变换可以类比为把分别以「我的左手边」「车头方向」和「地图正北」描述的路线，统一翻译成同一种说法喵。

外参标定确定传感器坐标系与车身坐标系之间的旋转和平移，它描述的是传感器安装姿态，而不是某一帧临时测出的车辆状态喵。

哪怕雷达只倾斜了一个很小角度，投射到远处后也会形成明显位置误差；IMU 若与车身轴不对齐，还会把前向加速度错误分配到侧向或垂向喵。

GNSS 天线与 IMU 之间的杆臂距离也必须建模，因为车辆转弯时，车顶天线和车辆参考点的瞬时速度并不完全相同喵。

因此，统一时间回答「什么时候测的」，统一坐标系回答「相对于谁测的」，两者共同构成多传感器融合的数据契约喵。

## Code Example

下面的示例使用一维位置和速度状态，先用 IMU 加速度推进状态，再用轮速约束速度，并在 GNSS 可用时校正绝对位置喵。

代码只依赖 Python 标准库，可以直接运行，并故意为 IMU 和轮速加入系统误差，以观察 GNSS 中断时的位置漂移喵。

```python
import math
import random

random.seed(7)

dt = 0.1
steps = 350

# 状态 x = [位置, 速度]，P 是 2×2 协方差矩阵喵
x_pos, x_vel = 0.0, 0.0
P00, P01, P10, P11 = 4.0, 0.0, 0.0, 1.0

true_pos, true_vel = 0.0, 0.0
imu_bias = 0.025
records = []


def predict_with_imu(pos, vel, p00, p01, p10, p11, accel):
    """用 IMU 加速度推进状态和不确定度喵"""
    pos = pos + vel * dt + 0.5 * accel * dt * dt
    vel = vel + accel * dt

    # 运动和传感器误差会让预测不确定度持续增长喵
    q_pos, q_vel = 0.005, 0.01

    # A = [[1, dt], [0, 1]] 对协方差的传播喵
    n00 = p00 + dt * (p01 + p10) + dt * dt * p11 + q_pos
    n01 = p01 + dt * p11
    n10 = p10 + dt * p11
    n11 = p11 + q_vel

    return pos, vel, n00, n01, n10, n11


def update_velocity(pos, vel, p00, p01, p10, p11, measured_vel):
    """把轮速当作速度观测，抑制纯 IMU 积分漂移喵"""
    measurement_variance = 0.15 ** 2
    innovation = measured_vel - vel
    innovation_variance = p11 + measurement_variance

    k_pos = p01 / innovation_variance
    k_vel = p11 / innovation_variance

    pos += k_pos * innovation
    vel += k_vel * innovation

    old00, old01, old10, old11 = p00, p01, p10, p11
    p00 = old00 - k_pos * old10
    p01 = old01 - k_pos * old11
    p10 = old10 - k_vel * old10
    p11 = old11 - k_vel * old11

    return pos, vel, p00, p01, p10, p11


def update_position(pos, vel, p00, p01, p10, p11, measured_pos):
    """用 GNSS 绝对位置把长期漂移拉回喵"""
    measurement_variance = 0.8 ** 2
    innovation = measured_pos - pos
    innovation_variance = p00 + measurement_variance

    k_pos = p00 / innovation_variance
    k_vel = p10 / innovation_variance

    pos += k_pos * innovation
    vel += k_vel * innovation

    old00, old01, old10, old11 = p00, p01, p10, p11
    p00 = old00 - k_pos * old00
    p01 = old01 - k_pos * old01
    p10 = old10 - k_vel * old00
    p11 = old11 - k_vel * old01

    return pos, vel, p00, p01, p10, p11


for i in range(steps):
    t = i * dt

    # 构造一段先加速、后匀速、再减速的真实运动喵
    if t < 5.0:
        true_accel = 0.4
    elif t < 25.0:
        true_accel = 0.0
    elif t < 30.0:
        true_accel = -0.3
    else:
        true_accel = 0.0

    true_pos += true_vel * dt + 0.5 * true_accel * dt * dt
    true_vel += true_accel * dt

    # IMU 有零偏和随机噪声喵
    imu_accel = (
        true_accel
        + imu_bias
        + random.gauss(0.0, 0.04)
    )

    # 轮速存在 2% 比例误差和随机噪声喵
    wheel_vel = (
        true_vel * 1.02
        + random.gauss(0.0, 0.15)
    )

    # 高频预测阶段：IMU 推进，轮速约束速度喵
    x_pos, x_vel, P00, P01, P10, P11 = predict_with_imu(
        x_pos,
        x_vel,
        P00,
        P01,
        P10,
        P11,
        imu_accel,
    )

    x_pos, x_vel, P00, P01, P10, P11 = update_velocity(
        x_pos,
        x_vel,
        P00,
        P01,
        P10,
        P11,
        wheel_vel,
    )

    # GNSS 为 1 Hz，并在 12～24 秒完全中断喵
    gnss_used = (
        i % 10 == 0
        and not (12.0 <= t < 24.0)
    )

    if gnss_used:
        gnss_pos = (
            true_pos
            + random.gauss(0.0, 0.8)
        )

        x_pos, x_vel, P00, P01, P10, P11 = update_position(
            x_pos,
            x_vel,
            P00,
            P01,
            P10,
            P11,
            gnss_pos,
        )

    records.append(
        (
            t,
            true_pos,
            x_pos,
            x_pos - true_pos,
            gnss_used,
        )
    )


print(" time | true_pos | estimate | error | GNSS")

for index in (0, 110, 180, 239, 240, 250, 349):
    t, truth, estimate, error, used = records[index]

    print(
        f"{t:5.1f} | "
        f"{truth:8.2f} | "
        f"{estimate:8.2f} | "
        f"{error:+6.2f} | "
        f"{used}"
    )


outage_errors = [
    error
    for t, _, _, error, _ in records
    if 12.0 <= t < 24.0
]

outage_rmse = math.sqrt(
    sum(error * error for error in outage_errors)
    / len(outage_errors)
)

print(
    f"GNSS 中断期间位置 RMSE: "
    f"{outage_rmse:.2f} m"
)
```

`predict_with_imu` 对应滤波器的预测步骤，它根据上一时刻的位置、速度和 IMU 加速度推算下一状态，并同步增大协方差喵。

协方差可以理解为估计结果周围的「不确定范围」，过程噪声越大或预测时间越长，这个范围通常就越大喵。

`update_velocity` 把轮速作为高频速度观测，使用轮速与预测速度之间的残差修正状态喵。

轮速虽然能抑制 IMU 的快速积分漂移，但代码故意加入了 2% 的比例误差，所以它无法提供长期准确的绝对位置喵。

`update_position` 是 GNSS 校正步骤，卡尔曼增益根据当前预测协方差和 GNSS 测量方差分配修正权重喵。

GNSS 中断期间，这一步被跳过，系统只能进行航位推算，因此位置误差会从中断初期的较小值逐渐增长喵。

固定随机种子下，示例在 23.9 秒附近会积累约 0.7 m 的位置误差，GNSS 于 24 秒恢复后会先把误差缩小到约 0.4 m，之后继续收敛喵。

恢复后的估计没有被强制瞬移到 GNSS 测量值，因为滤波器同时考虑 GNSS 噪声和已有预测，这正是预测—校正循环的核心直觉喵。

## Common Misconceptions

- **误区 1：点数越多，激光雷达就一定越好喵。**
  **纠正：** 有效性能还取决于点的空间分布、回波质量、扫描延迟、目标反射特性和算法能否利用这些点，重复堆积在近处的点未必能改善远处探测喵。
- **误区 2：RTK 达到厘米级后，车辆就不再需要 IMU 和轮速喵。**
  **纠正：** RTK 更新率、连续性和异常恢复能力有限，IMU 与轮速仍负责填补两次绝对定位之间的运动，并在遮挡期间维持短时连续性喵。
- **误区 3：滤波融合就是把多个传感器结果取平均值喵。**
  **纠正：** 滤波器需要传播车辆状态和不确定度，并依据误差模型、观测残差与当前环境动态调整权重，异常观测还可能被拒绝喵。
- **误区 4：只要各传感器都经过标定，数据就能直接融合喵。**
  **纠正：** 空间外参正确但时间错位时，系统仍会把不同时刻的数据强行叠加，因此坐标统一和时间同步缺一不可喵。
- **误区 5：GNSS 中断后，组合定位能无限期保持原有精度喵。**
  **纠正：** 航位推算只能延缓误差增长，不能消除缺少绝对约束带来的长期漂移，系统必须跟踪不确定度并据此调整功能边界喵。

## Socratic Check

1. 如果一辆车正在转弯，而激光雷达一帧扫描耗时 100 ms，把所有点直接放进同一个车身坐标系会发生什么喵？
   **提示：** 想一想帧首和帧尾对应的车辆朝向是否相同，以及去畸变需要哪两类信息喵。
2. 车辆进入长隧道后，GNSS 消失但 IMU 和轮速仍然正常，为什么滤波器的位置输出可以连续，却不能继续宣称原来的定位精度喵？
   **提示：** 区分「仍能给出一个数值」和「知道这个数值有多可靠」，答案藏在协方差随预测增长的过程中喵。

## Quick Summary

- 激光雷达通过 ToF 或 FMCW 等机制测距，并把距离、方向、强度和时间戳组织成点云喵。
- 一帧点云跨越一段采样时间，车辆运动会造成几何畸变，需要依靠逐点时间戳和连续姿态轨迹去畸变喵。
- 安装位置需要同时权衡视场、盲区、污染、车身遮挡、冗余覆盖、造型和维护条件喵。
- GNSS 或 RTK 提供绝对锚点，IMU 提供高频运动变化，轮速约束纵向速度，滤波器通过预测—校正循环融合三者喵。
- 时间同步解决「什么时候测的」，坐标变换和外参标定解决「相对于谁测的」，二者是可靠融合的共同前提喵。
