## 定位

这是「LLM 工作原理与 Token」的第三次深入：从单个 Token 如何被生成，进一步走向多个 Token 如何被选择、组织和压缩，聚焦上下文窗口工程（context window engineering）。

## 类比

可以把上下文窗口想象成一张面积固定的会议桌。

系统提示词是会议规则，对话历史是此前的会议纪要，外部资料是临时搬来的参考文件，用户问题是当前议题，模型即将生成的回答则需要一块空白记录区。

桌面放满以后，新文件不会凭空获得空间。你必须移走旧材料、压缩纪要、只保留相关页面，或者缩短准备写下的回答。如果不做管理，即使所有材料都很重要，它们也可能互相遮挡，使模型更难找到真正关键的信息。

上下文窗口工程的目标不是「尽可能把桌子堆满」，而是让当前任务最需要的信息，在正确的位置，以合适的密度出现在桌面上。

## 核心机制

### 1. 上下文窗口到底限制了什么

上下文窗口（context window）限制的是模型在一次推理过程中能够处理的 Token 总量。通常需要满足：

```text
系统提示词
+ 对话历史
+ 外部资料
+ 用户问题
+ 工具调用及返回结果
+ 本次生成的输出
≤ 上下文窗口上限
```

也可以写成预算公式：

```text
T_system
+ T_history
+ T_rag
+ T_user
+ T_tools
+ T_output_reserved
+ T_margin
≤ T_context
```

这里最容易忽略的是：输入和输出通常共享同一总预算。

假设模型的上下文窗口上限为 128K Token。如果请求已经占用 120K Token，即使模型理论上允许生成 16K Token，也不代表还能完整生成 16K；在共享窗口约束下，实际只剩约 8K，而且工程上还应留出安全余量。

有些 API 会同时设置「上下文窗口上限」和「单次最大输出上限」。此时必须同时满足两项约束：输出既不能超过单次输出上限，也不能挤破输入与输出共享的总窗口。

#### 窗口为什么不能无限扩大

扩大窗口不只是修改一个配置数字，它会同时影响以下资源：

- **KV cache 显存：** 模型需要在每一层保存已有 Token 的 Key 和 Value。对于固定模型和精度，KV cache 占用通常随序列长度近似线性增长。窗口越长，每个并发请求占用的显存越多，服务器可承载的并发数越低。

- **预填充计算：** 处理输入上下文的阶段称为预填充（prefill）。在标准稠密自注意力中，Token 之间需要建立大量两两关系，其计算量会随序列长度快速增长，经典复杂度可近似理解为 O(n²)。FlashAttention 等优化能降低中间内存和常数开销，但不会自动让超长上下文变成零成本。

- **逐 Token 解码：** 生成新 Token 时，模型会查询此前的 KV cache。上下文越长，每一步需要关注的历史位置越多，因此单步解码通常也会变慢。

- **API 成本：** 输入 Token、输出 Token 和缓存命中的 Token 往往采用不同计价方式，但长上下文通常仍意味着更高费用。塞入大量无关资料，即使没有超出窗口，也可能是在为噪声付费。

- **注意力质量：** 模型能够接收某个长度，不代表它能在所有位置上同样可靠地理解和调用信息。长上下文还会增加信息冲突、干扰和检索失败的概率。

- **训练与位置泛化：** 模型需要在训练中学会处理不同位置。如果实际长度远超训练时常见的长度，位置表示和注意力模式未必能稳定外推。

因此，上下文窗口是计算能力、显存容量、服务并发、成本、延迟和模型质量之间的综合权衡。

### 2. 有限窗口下如何分配空间

预算分配没有适用于所有任务的固定比例。更实用的方法是先按优先级划分，再根据任务动态调整。

通常可以采用以下顺序：

1. **先预留输出空间。** 在拼接输入前确定回答最多需要多少 Token，避免输入把输出空间全部吃掉。

2. **保留不可缺少的控制信息。** 系统提示词、安全规则、输出格式和当前用户问题通常具有较高优先级。

3. **按相关性选择外部资料。** 检索增强生成（Retrieval-Augmented Generation, RAG）不应把搜索到的所有内容都塞进窗口，而应选择与问题直接相关、彼此不过度重复的片段。

4. **为对话历史设置上限。** 最近几轮通常比很早的闲聊更重要，但长期目标、用户约束和已经确认的决策不能仅按时间粗暴删除。

5. **保留安全余量。** Token 估算可能与模型的真实 Tokenizer 不完全一致，工具返回结果也可能临时增长，因此不宜把预算使用率推到 100%。

以 128K 窗口为例，一次资料型问答可以先设置如下上限：

| 区域 | 示例上限 | 主要内容 |
| --- | ---: | --- |
| 系统提示词 | 8K | 角色、约束、工具说明、输出格式 |
| 对话历史 | 24K | 最近对话、长期目标、已确认决策 |
| RAG 资料 | 56K | 与问题最相关的原文片段 |
| 当前用户问题 | 4K | 问题、附件说明和局部约束 |
| 预留输出 | 20K | 分析、回答、代码或结构化结果 |
| 安全余量 | 16K | Token 误差、工具结果和突发增长 |

这些数字是容量上限，不是必须填满的配额。如果只需要 6K 的资料，就不应为了「充分利用窗口」而补入 50K 的低相关内容。

一个成熟的预算管理器还会根据任务动态调节：

```text
简单问答
→ 缩小历史与资料预算，降低延迟

文档综合分析
→ 扩大资料预算，保留足够输出空间

连续协作任务
→ 扩大历史、状态摘要和决策记录预算

代码生成或长篇报告
→ 优先增加输出预留，减少重复资料
```

### 3. 长文本与长对话的工程策略

#### 截断

截断（truncation）是在内容超出预算时直接删除一部分。

- **头部截断（head truncation）：** 删除开头，保留结尾。适合最近内容最重要的日志流、即时对话和事件序列，但可能丢失最初的任务目标、定义和约束。

- **尾部截断（tail truncation）：** 删除结尾，保留开头。适合开头包含标题、规则或定义的文档，但可能丢失最新状态、结论和用户真正的问题。

截断速度快、成本低，但它不了解语义边界。删除半个函数、半条规则或否定句中的关键部分，都可能改变剩余文本的含义。

更稳妥的做法是按消息、段落、章节或代码符号边界截断，而不是按字符数机械剪切。

#### 滑动窗口

滑动窗口（sliding window）只保留最近一段连续内容。当新内容进入时，最旧内容从窗口另一端移出：

```text
时刻 1：[A B C D]
时刻 2：  [B C D E]
时刻 3：    [C D E F]
```

它适合实时日志、语音转写、连续事件监测和局部依赖较强的文本。其优势是内存使用稳定、实现简单；代价是跨越窗口边界的长期依赖容易丢失。

例如，用户在对话开头规定「所有金额使用人民币」，几十轮后这条信息可能滑出窗口。系统若没有把它提取到长期状态中，模型便可能违反约束。

#### 对话历史裁剪

对话历史裁剪（conversation history pruning）以「消息」为单位选择保留内容，而不是简单保留最后若干字符。

常见优先级是：

```text
系统规则
> 当前问题
> 用户长期偏好与明确约束
> 已确认的决策和工具结果
> 最近几轮相关对话
> 早期探索、寒暄和重复内容
```

裁剪可以同时考虑时间、相关性和消息类型。例如，旧的闲聊可以删除，但早期确定的数据库 schema 不能因为时间较早就被丢弃。

它的主要风险是把一句后来仍被依赖的信息误判为不重要。因此，裁剪通常需要与结构化状态或摘要配合使用。

#### 摘要压缩

摘要压缩（summarization compaction）把较早的详细内容改写成更短的语义摘要：

```text
原始历史：18,000 Token
        ↓ 提取目标、约束、决策、未解决问题
压缩摘要：1,500 Token
        +
最近对话：6,000 Token
```

与直接截断相比，摘要能保留跨轮次的重要语义，特别适合长期 Agent 任务、项目协作和多轮学习。

但摘要不是无损压缩。每次压缩都可能：

- 丢失限定条件、数字、引用来源或少数例外。
- 把尚未确定的讨论写成已经确认的结论。
- 在多轮「摘要的摘要」中逐渐产生语义漂移。
- 消耗额外的模型调用、时间和费用。

因此，重要事实最好同时保存为结构化状态，例如：

```text
目标：完成支付模块重构
已确认：数据库使用 PostgreSQL 16
硬约束：不得修改公开 API
待决定：重试次数与退避策略
证据来源：设计文档第 4.2 节
```

结构化状态适合保存精确事实，摘要适合保留对话脉络，两者可以互补。

#### 分块处理

分块处理（chunking）把长文档拆成多个较小单元，分别处理，再汇总结果。

```text
长文档
  ↓ 按章节或语义边界切分
块 A  块 B  块 C  块 D
  ↓    ↓    ↓    ↓
局部提取或局部摘要
  └────┬────┬────┘
       ↓
全局合并与校验
```

它适合长报告、代码仓库、会议记录和多文档分析。相比一次塞入全文，分块可以控制每次调用的窗口占用，还能并行处理。

主要代价是全局关系可能被切断。例如，一项定义出现在块 A，例外条款出现在块 D，只分析单块可能得到错误结论。

降低风险的方法包括：

- 按语义边界切分，而不是固定字符数硬切。
- 在相邻块之间保留少量重叠。
- 为每个块附加文档标题、章节路径等元数据。
- 最后进行跨块综合，并回查原文证据。
- 对代码优先按类、函数和模块边界切分。

### 4. 位置效应与「中间遗失」

长上下文并不意味着模型能够均匀利用每一个位置。

「中间遗失」（lost in the middle）描述的是一种常见现象：当上下文很长时，模型往往更容易利用开头和结尾附近的信息，而对埋在中间的关键事实调用得不够稳定。

可以把它粗略画成：

```text
信息被有效利用的概率

较高  ████▆▅▄▃▂▂▃▄▅▆████  较高
      开头        中间        结尾
```

这不是每个模型、每个任务都严格遵循的固定曲线，但它提醒我们：最大窗口长度只是容量指标，不是信息利用率保证。

工程上的常见做法包括：

- 把任务目标、角色和硬约束放在稳定的开头。
- 把当前问题、期望输出和最关键证据放在靠近结尾的位置。
- 对长资料先做检索，让最相关片段靠近当前问题。
- 使用标题、编号和明确引用，减少模型寻找信息的难度。
- 删除重复或低相关内容，提升关键内容在注意力中的相对信号。
- 在末尾简短重申关键约束，但避免制造与开头不一致的第二套规则。

「放在开头或结尾」只是经验策略。更重要的是让信息简洁、无冲突、与当前任务直接相关。

### 5. Prompt caching 与上下文组织

提示词缓存（prompt caching），也称前缀缓存（prefix caching），允许服务端复用相同提示词前缀已经完成的部分计算。

假设连续请求具有如下结构：

```text
请求 1：
[系统规则][工具定义][长参考手册][用户问题 A]

请求 2：
[系统规则][工具定义][长参考手册][用户问题 B]

请求 3：
[系统规则][工具定义][长参考手册][用户问题 C]
```

前三个请求的开头完全相同，服务端便可能复用这个稳定前缀，减少重复的预填充计算。根据平台的计费规则，缓存命中的输入 Token 也可能更便宜。

因此，上下文通常适合按以下顺序组织：

```text
稳定、可复用的内容
    系统规则
    工具定义
    长期不变的参考资料
    固定输出规范

半稳定内容
    会话摘要
    项目状态

高度可变的内容
    最新工具结果
    当前用户问题
    本轮临时要求
```

如果把时间戳、随机 ID 或当前问题放在最前面，即使后面有一大段完全相同的资料，前缀也会很早发生变化，从而降低缓存命中范围。

这与已经学过的 KV cache 有联系，但两者不应混为一谈：

- **单次生成中的 KV cache：** 保存当前请求已经处理过的 Token 的 Key 和 Value，使模型生成下一个 Token 时不必重新计算全部历史。

- **跨请求的 prompt caching：** 尝试复用多个请求之间相同前缀的处理结果。它是否存在、保存多久、如何计费和如何判定命中，取决于服务商实现。

可以把前者理解为「写当前回答时，不重复阅读前几页」，后者理解为「下一场会议复用了上一场已经摆好的相同资料」。

Prompt caching 主要优化成本和延迟，不保证回答更一致。即使前缀完全相同，采样随机性、后续问题、模型版本和服务端执行环境仍可能使输出不同。

### 6. 扩展上下文的技术方向

模型研究中会通过位置编码外推（positional encoding extrapolation）、RoPE scaling 等方法，让模型处理比原始训练长度更长的序列。

这些技术可以扩大可接受的长度，但不能消除 KV cache、计算成本、位置效应和信息噪声。因此，「模型支持更长窗口」与「应用不再需要上下文管理」是两回事。

## 代码示例

下面的 Python 程序不调用任何模型 API。它模拟一个简单的上下文预算管理器，演示以下行为：

- 先预留输出预算。
- 限制 RAG 资料的最大占用。
- 把剩余空间分配给对话历史。
- 优先保留最近的完整消息。
- 用摘要占位符代表被压缩的早期历史。
- 验证最终上下文没有超过总预算。

```python
import re
from dataclasses import dataclass
from typing import List, Tuple


# 这里只实现“伪 Tokenizer”，用于演示预算算法。
# 它不能替代真实模型的 Tokenizer。
TOKEN_PATTERN = re.compile(
    r"[一-鿿]|[A-Za-z0-9_]+|[^\s]",
    re.UNICODE,
)


def count_tokens(text: str) -> int:
    """粗略估算 Token 数：中文单字、英文词和标点分别计数。"""
    return len(TOKEN_PATTERN.findall(text))


def truncate_to_budget(
    text: str,
    budget: int,
    keep_tail: bool = False,
) -> str:
    """把文本截到预算以内，支持保留开头或保留结尾。"""
    if budget <= 0:
        return ""

    if count_tokens(text) <= budget:
        return text

    # 二分搜索可保留的字符数，避免逐字符反复扫描。
    low, high = 0, len(text)

    while low < high:
        middle = (low + high + 1) // 2
        candidate = text[-middle:] if keep_tail else text[:middle]

        if count_tokens(candidate) <= budget:
            low = middle
        else:
            high = middle - 1

    kept = text[-low:] if keep_tail else text[:low]
    marker = "…"

    # 省略号本身也占预算，因此再做一次保护。
    while kept and count_tokens(marker + kept) > budget:
        kept = kept[1:] if keep_tail else kept[:-1]

    return marker + kept if keep_tail else kept + marker


@dataclass
class Message:
    role: str
    content: str

    def render(self) -> str:
        return f"[{self.role}] {self.content}"


@dataclass
class ContextResult:
    system: str
    rag: str
    history: List[str]
    user: str
    reserved_output: int
    used_input: int
    total_budget: int


class ContextBudgetManager:
    def __init__(
        self,
        max_context: int,
        reserved_output: int,
        rag_cap: int,
        summary_cap: int,
        safety_margin: int,
    ) -> None:
        self.max_context = max_context
        self.reserved_output = reserved_output
        self.rag_cap = rag_cap
        self.summary_cap = summary_cap
        self.safety_margin = safety_margin

    def _fit_rag(self, chunks: List[str]) -> str:
        """按相关性顺序装入 RAG 片段，超出上限时截断最后一块。"""
        selected: List[str] = []
        used = 0

        for index, chunk in enumerate(chunks, start=1):
            rendered = f"[资料 {index}] {chunk}"
            remaining = self.rag_cap - used

            if remaining <= 0:
                break

            if count_tokens(rendered) <= remaining:
                selected.append(rendered)
                used += count_tokens(rendered)
            else:
                selected.append(
                    truncate_to_budget(rendered, remaining)
                )
                break

        return "\n".join(selected)

    def _compact_history(
        self,
        history: List[Message],
        budget: int,
    ) -> List[str]:
        """保留最近消息，并为被删除的早期历史生成摘要占位符。"""
        if budget <= 0:
            return []

        rendered = [message.render() for message in history]
        total = sum(count_tokens(item) for item in rendered)

        if total <= budget:
            return rendered

        # 最多拿出一部分历史预算存放摘要。
        summary_budget = min(self.summary_cap, budget // 3)
        recent_budget = budget - summary_budget

        kept_reversed: List[str] = []
        used = 0

        # 从最新消息向前选择，优先保留完整的最近消息。
        for item in reversed(rendered):
            item_cost = count_tokens(item)

            if used + item_cost <= recent_budget:
                kept_reversed.append(item)
                used += item_cost
            else:
                break

        kept = list(reversed(kept_reversed))
        dropped_count = len(rendered) - len(kept)

        if dropped_count > 0 and summary_budget > 0:
            summary = (
                f"[历史摘要] 较早的 {dropped_count} 条消息已被压缩。"
                "真实系统应在这里保存目标、硬约束、已确认决策、"
                "关键事实和未解决问题，而不是只记录消息数量。"
            )
            summary = truncate_to_budget(summary, summary_budget)
            kept.insert(0, summary)

        return kept

    def build(
        self,
        system_prompt: str,
        rag_chunks: List[str],
        history: List[Message],
        user_question: str,
    ) -> ContextResult:
        rag = self._fit_rag(rag_chunks)

        fixed_cost = (
            count_tokens(system_prompt)
            + count_tokens(rag)
            + count_tokens(user_question)
            + self.reserved_output
            + self.safety_margin
        )

        history_budget = self.max_context - fixed_cost

        if history_budget < 0:
            raise ValueError(
                "系统提示词、RAG、用户问题、输出预留和安全余量"
                "已经超过上下文上限，应先缩减固定内容。"
            )

        compacted_history = self._compact_history(
            history,
            history_budget,
        )

        used_input = (
            count_tokens(system_prompt)
            + count_tokens(rag)
            + sum(count_tokens(item) for item in compacted_history)
            + count_tokens(user_question)
        )

        total_reserved = (
            used_input
            + self.reserved_output
            + self.safety_margin
        )

        assert total_reserved <= self.max_context

        return ContextResult(
            system=system_prompt,
            rag=rag,
            history=compacted_history,
            user=user_question,
            reserved_output=self.reserved_output,
            used_input=used_input,
            total_budget=self.max_context,
        )


def main() -> None:
    manager = ContextBudgetManager(
        max_context=180,
        reserved_output=40,
        rag_cap=45,
        summary_cap=18,
        safety_margin=10,
    )

    system_prompt = (
        "你是技术导师。回答必须基于资料，区分事实与推测，"
        "并使用简体中文。"
    )

    # 假设这些片段已经按相关性从高到低排列。
    rag_chunks = [
        "上下文窗口由输入 Token 和生成 Token 共同占用。",
        "KV cache 的容量通常随序列长度近似线性增长。",
        "长上下文可能出现 lost in the middle 位置效应。",
    ]

    history = [
        Message("user", "我们正在设计一个长对话学习助手。"),
        Message("assistant", "已确认目标是保存长期学习进度。"),
        Message("user", "所有示例都必须能够离线运行。"),
        Message("assistant", "我会避免调用外部模型 API。"),
        Message("user", "还要区分摘要、结构化状态和原始记录。"),
        Message("assistant", "结构化状态保存精确事实，摘要保存语义脉络。"),
        Message("user", "最近几轮应该优先保留，但不能丢掉硬约束。"),
        Message("assistant", "可以为硬约束设置高于时间顺序的优先级。"),
    ]

    result = manager.build(

        user_question=user_question,
    )

    print("=== 最终上下文 ===")
    print("\n[系统提示词]")
    print(result.system)

    print("\n[RAG 资料]")
    print(result.rag)

    print("\n[对话历史]")
    for item in result.history:
        print(item)

    print("\n[当前问题]")
    print(result.user)

    print("\n=== 预算统计 ===")
    print(f"输入已用：{result.used_input}")
    print(f"输出预留：{result.reserved_output}")
    print(f"窗口总量：{result.total_budget}")

```

这段代码中的 `count_tokens` 只是教学用估算器。真实应用必须调用目标模型对应的 Tokenizer，因为不同模型可能把同一段文字拆成不同数量的 Token。

`_fit_rag` 假设检索结果已经按相关性排序。它依次装入片段，并限制 RAG 内容最多占用 `rag_cap`。这说明检索系统不仅要「找到资料」，还要决定哪些资料值得占用窗口。

`build` 先计算系统提示词、RAG、用户问题、输出预留和安全余量的固定成本，再把剩余空间交给历史消息。这样可以避免对话历史无限增长，挤掉本次回答所需的空间。

`_compact_history` 从最新消息开始倒序选择，因此优先保留最近的完整消息。较早消息被删除后，程序插入一条摘要占位符。真实系统中的摘要器应提取目标、约束、决策和未解决问题，不能只写「前面讨论过一些内容」。

这个示例仍然是简化模型。生产系统通常还需要消息重要度、语义相关性、工具结果过期策略、来源引用、结构化长期状态，以及真实 Tokenizer 的精确计数。

## 常见误区

### 误区 1：窗口越大，就越应该把所有资料塞进去

窗口容量不是推荐使用量。低相关、重复或相互矛盾的资料会增加成本和延迟，还可能稀释关键证据。更好的目标是「足够且相关」，而不是「尽量装满」。

### 误区 2：模型对窗口内所有位置一视同仁

Token 虽然都在合法窗口内，但它们被有效调用的概率并不完全相同。长上下文中的中间信息可能更容易被忽略，因此还需要检索、排序、压缩和位置设计。

### 误区 3：Prompt caching 能让回答更一致

Prompt caching 复用的是相同前缀的部分计算，主要价值是减少延迟和费用。它不固定采样结果，也不能消除模型更新、并发执行和后续输入变化带来的差异。

### 误区 4：截断只是把文字剪短，无关紧要

截断可能删除任务目标、否定词、单位、例外条款或代码依赖。即使只少了几个 Token，也可能改变整段内容的语义。截断策略必须考虑内容结构和信息优先级。

## 苏格拉底式检查

### 问题 1

一个模型支持 128K 上下文。当前请求的输入已经达到 124K Token，而你希望模型生成最多 12K Token。能否直接把最大输出设置为 12K？

**提示：** 输入与输出需要满足什么总量关系？是否还需要安全余量？

**答案：** 通常不能。输入与输出共享 128K 总预算，124K 输入只留下约 4K，更没有计算 Token 误差和工具结果增长所需的余量。应先缩减输入，或者降低输出上限，并保留适当安全空间。

### 问题 2

一个长期对话助手把系统规则、当前时间、用户问题、固定工具定义和产品手册依次拼接。为什么它的 prompt caching 效果可能很差？应该怎样调整？

**提示：** 前缀缓存需要多次请求的哪一部分保持一致？

**答案：** 当前时间和用户问题位于前面，会使请求在很早的位置就发生变化，后面的固定工具定义和产品手册难以形成可复用的连续前缀。可以把稳定的系统规则、工具定义和固定资料放在前面，把会话状态、当前时间和用户问题放在后面。

---

## 快速总结

- 上下文窗口限制一次推理中输入与输出 Token 的总量，生成空间必须提前预留。

- 长窗口会增加 KV cache 显存、预填充计算、逐 Token 解码时间、服务成本和并发压力。

- 预算分配应优先保证系统规则、当前问题和输出空间，再按相关性选择资料和历史。

- 截断、滑动窗口、历史裁剪、摘要压缩和分块处理各有适用场景，也都会带来不同形式的信息损失。

- 长上下文不等于均匀利用；重要信息通常应保持简洁，并放在开头或靠近当前问题的位置。

- Prompt caching 偏好「稳定前缀在前、可变内容在后」，它优化成本和延迟，但不保证输出一致。
