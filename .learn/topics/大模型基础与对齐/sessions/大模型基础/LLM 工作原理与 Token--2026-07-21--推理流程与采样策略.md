## 定位

「LLM 工作原理与 Token」位于大模型基础知识的入口位置；本次是第二遍加深，重点从“模型会文字接龙”深入到一次完整的推理流程，以及模型如何通过不同采样策略选择下一个 Token。

## 类比

继续把大语言模型想象成一位“文字接龙大师”。

当轮到它接下一个词时，它不会立刻说出答案，而是先给词典里的每个候选词打一个原始分数。例如，“苹果”得 4.2 分，“香蕉”得 3.7 分，“火车”得 0.8 分。这些未经归一化的原始分数就是 **logits**。

接下来，计分员使用 **softmax**，把这些高低不限、总和也没有意义的分数转换为总和等于 100% 的候选概率。于是，“苹果”可能占 58%，“香蕉”占 35%，“火车”占 7%。

最后，裁判按照指定的采样规则选词：可以永远选概率最高的词，也可以保留若干合理候选后抽签。选出的词会被接到原句末尾，接龙大师再以更新后的整句话为依据，开始下一轮打分、换算和选择。

因此，生成文本不是一次写完整段落，而是不断重复：

```text
候选打分 → 概率换算 → 按规则选一个 Token → 接回原文 → 下一轮
```

## 核心机制

### 快速复习

大语言模型（Large Language Model，LLM）可以看成一台非常复杂的“下一个 Token 概率预测机”。它每次根据当前上下文，预测接下来最可能出现哪些 Token，再从中选出一个。

**Token** 是模型处理文本时使用的基本单位。它可能是一个汉字、一个词的一部分、一个标点符号，甚至是一段常见字符，而不一定等于一个完整单词或单个字符。

**上下文窗口（context window）** 是模型一次能够处理的 Token 总量限制，通常需要同时容纳输入提示、对话历史和本轮生成的输出。生成得越长，留给输入内容的空间通常就越少。

上次把生成理解为“文字接龙”，这次继续拆开接龙大师每一轮内部究竟做了什么。

### 一次完整的生成推理流程

完整流程可以概括为：

```text
输入文本
   │
   ▼
Tokenizer 分词
   │
   ▼
Token IDs
   │
   ▼
Embedding 向量
   │
   ▼
多层 Transformer
   │
   ▼
词表中每个 Token 的 logits
   │
   ▼
温度调整、top-k、top-p 等处理
   │
   ▼
softmax 概率
   │
   ▼
采样出下一个 Token
   │
   ▼
拼回输入，继续下一轮
   │
   └──────────────► 直到生成 EOS 或达到其他停止条件
```

#### 第一步：文本进入 Tokenizer

模型不能直接处理人类看到的字符串，因此需要先经过 **Tokenizer（分词器）**。

假设输入是：

```text
今天天气很好
```

分词器可能把它拆成：

```text
["今天", "天气", "很", "好"]
```

这只是示意。真实拆分方式由具体模型的词表和分词算法决定，不同模型可能得到不同结果。

#### 第二步：Token 转换为 Token IDs

模型的词表会为每个 Token 分配一个整数编号，即 **Token ID**。例如：

```text
["今天", "天气", "很", "好"]
    ↓
[7312, 2456, 892, 1963]
```

Token ID 只是索引，不表示大小关系。“7312”并不比“892”更重要，也不意味着两者在语义上相近。

#### 第三步：Token IDs 转换为 Embedding

神经网络需要连续数值，因此每个 Token ID 会被查表转换成一个高维向量，这个向量称为 **Embedding（嵌入）**。

Embedding 可以理解为 Token 在模型内部的“数字化表示”。语义或使用方式相近的 Token，经过模型训练后往往会呈现某些相似的向量特征。

除了 Token 本身的信息，模型还需要知道它在序列中的位置。具体模型会通过位置编码或旋转位置编码等机制，把顺序信息融入计算。

#### 第四步：通过多层 Transformer

Embedding 会依次通过许多层 **Transformer**。

在这些层中，模型会综合考虑当前 Token、前文中的其他 Token，以及它们之间可能存在的关系。例如，模型可能判断某个代词指向谁、某个动词通常搭配什么宾语，或者当前内容属于代码、对话还是说明文。

经过一层又一层变换后，模型得到当前上下文的内部表示。对于生成任务，我们最关心最后一个位置的表示，因为它将用于预测下一个 Token。

#### 第五步：为词表中的每个 Token 输出 logits

假设模型的词表包含 100,000 个 Token，那么模型会输出一个长度为 100,000 的分数向量。每个位置对应词表中的一个候选 Token。

这些原始分数称为 **logits**。logit 可以是任意实数，不要求位于 0 到 1 之间，也不要求总和等于 1。

例如：

```text
候选 Token    logit
-------------------
"。"           3.9
"！"           2.8
"啊"           1.7
"冰箱"        -0.6
```

logit 越高，表示模型在当前上下文中越偏好该 Token，但它还不是概率。

#### 第六步：调整候选分布并转换为概率

通常会先根据 temperature、top-k 或 top-p 等参数调整候选 logits，然后使用 **softmax** 把保留下来的分数转换为概率。

softmax 的直觉是：

```text
先对每个 logit 做指数运算，再除以所有指数结果之和
```

可写成简单算式：

```text
某 Token 的概率 = exp(它的 logit) / 所有候选 exp(logit) 的总和
```

这样得到的每个概率都位于 0 到 1 之间，而且全部候选概率之和等于 1。

#### 第七步：采样出下一个 Token

系统按照指定策略从概率分布中选出一个 Token。它既可以直接选择概率最大的候选，也可以按照概率随机抽样。

例如，模型选择了“！”，那么这个 Token 会被接到已有序列末尾：

```text
今天天气很好
       ↓
今天天气很好！
```

#### 第八步：循环直到停止

新 Token 被加入上下文后，模型再次预测下一个 Token。这个循环会持续进行，直到出现以下情况之一：

- 模型生成 **EOS（End of Sequence，序列结束）Token**。
- 达到允许生成的最大 Token 数。
- 命中调用方设置的停止字符串或停止 Token。
- 用户或系统主动中断生成。

EOS 是词表中的特殊 Token。它表达的是“序列可以在这里结束”，通常不会作为普通文字显示给用户。

### 采样策略

#### Greedy：每次选择概率最高的 Token

**Greedy decoding（贪心解码）** 每一步都选择当前概率最高的 Token：

```text
选择结果 = argmax(logits)
```

例如：

```text
"苹果"  55%
"香蕉"  30%
"梨"    15%
```

Greedy 永远选择“苹果”。

它的优点是简单、稳定、速度快，适合分类式回答、格式严格的抽取任务，以及希望结果尽量可复现的场景。

它的缺点是只考虑当前一步最优，不保证整段文本在全局上最好。连续选择最常见的表达，也可能让文本重复、僵硬或缺少变化。

#### Temperature：调节概率分布的尖锐程度

**Temperature（温度）** 会在 softmax 之前调整 logits：

```text
调整后的 logit = 原始 logit / T
```

然后再执行 softmax：

```text
概率 = softmax(logits / T)
```

当 `T < 1` 时，logits 之间的差距被放大，概率分布变得更尖锐。高分候选更容易胜出，结果通常更保守、更稳定。

当 `T = 1` 时，不改变原始 logits 的相对尺度。

当 `T > 1` 时，logits 之间的差距被缩小，概率分布变得更平坦。低概率候选获得更多机会，结果通常更多样，但也更容易偏题或产生不合理内容。

例如，原始 logits 是：

```text
[4, 3, 1]
```

使用低温度 `T = 0.5` 后：

```text
[8, 6, 2]
```

第一名相对其他候选的优势变得更大。

使用高温度 `T = 2` 后：

```text
[2, 1.5, 0.5]
```

候选之间更接近，抽中第二名或第三名的机会随之增加。

严格来说，`temperature = 0` 无法直接代入除法公式。工程实现通常把它特殊处理为 greedy，即直接选择最大 logit，但不同框架的具体约定可能不同。

#### Top-k：只在固定数量的高分候选中选择

**Top-k sampling** 只保留 logit 最高的 `k` 个 Token，其余候选被排除，然后在保留集合中重新计算概率并采样。

假设排序后的概率是：

```text
A: 40%
B: 25%
C: 15%
D: 12%
E:  8%
```

当 `top-k = 3` 时，只保留 A、B、C。它们原来的概率之和是 80%，重新归一化后变为：

```text
A: 50.00%
B: 31.25%
C: 18.75%
```

Top-k 的问题是候选数量固定。当模型非常确定时，保留 50 个候选可能太宽松；当模型很不确定时，只保留 5 个候选又可能太严格。

#### Top-p：保留累计概率达到阈值的最小候选集合

**Top-p sampling** 又称 **nucleus sampling（核采样）**。它先按概率从高到低排序，再保留累计概率达到 `p` 所需的最小候选集合。

假设概率是：

```text
A: 45%
B: 30%
C: 15%
D:  7%
E:  3%
```

当 `top-p = 0.8` 时：

```text
A 的累计概率：45%
A+B 的累计概率：75%
A+B+C 的累计概率：90%
```

加入 C 后累计概率第一次达到 80%，因此保留 A、B、C，再对它们重新归一化并采样。

Top-p 的候选数量会随上下文变化。模型很确定时，可能只保留少数几个候选；模型不确定时，则会自动保留更多候选。这种自适应特性使它成为文本生成中的常用策略。

### 如何组合这些参数

一个常见的工程顺序是：

```text
logits
  → temperature 调整
  → top-k 截断
  → top-p 截断
  → softmax 与归一化
  → 随机采样
```

不同框架的细节可能略有差异，因此使用具体推理库时仍应查看其文档。

实际配置可以从以下思路出发：

- 对事实问答、信息抽取和格式严格的输出，可以使用 greedy 或较低的 temperature，减少随机性。
- 对一般对话和内容改写，可以尝试 `temperature = 0.7` 左右，并配合 `top-p = 0.9～0.95`。
- 对创意写作，可以适度提高 temperature，但仍用 top-p 或 top-k 排除极不合理的长尾候选。
- 不要一开始就同时把 temperature、top-k 和 top-p 都限制得很严，否则合理候选可能被过度压缩，输出容易单调。
- 参数没有对所有模型都通用的“最佳值”。模型、提示词、任务目标和推理框架都会影响结果，应通过固定测试集比较配置。

### KV cache：为什么不必每一步重算全部历史

模型逐 Token 生成时，前面的文本不会在每一步发生变化。如果每生成一个新 Token，都从头计算整个上下文，很多工作会被重复执行。

Transformer 的注意力机制会为每个历史 Token 计算并使用 **Key（K）** 和 **Value（V）**。**KV cache（键值缓存）** 会保存已经计算过的 K 和 V。

第一轮处理提示词时，模型仍需读取整个提示词并建立缓存。之后每生成一个新 Token，模型只需计算这个新 Token 对应的新信息，并让它读取缓存中的历史 K、V，而不必重新生成所有旧 Token 的 K、V。

可以把它理解为接龙大师旁边放了一本笔记：

```text
没有 KV cache：
每接一个词，都重新阅读并整理整篇文章。

使用 KV cache：
先整理一次旧文章，以后只记录新接上的词并查阅已有笔记。
```

KV cache 不会改变模型理论上要表达的概率规则，它主要是一种推理性能优化。不过，它会占用显存或内存，而且缓存通常会随上下文长度增加。

由于自回归模型本来就是逐 Token 生成的，每得到一个 Token，就可以立即解码并发送给用户，这形成了**流式输出（streaming output）**。KV cache 降低了后续 Token 的重复计算，使连续生成和流式展示拥有更可接受的延迟，但流式输出本身是一种结果传输方式，并不是 KV cache 创造出的生成机制。

## 代码示例

下面的代码不调用任何模型 API，而是假设模型已经给出了几个候选词的 logits。代码手写 temperature、top-k 和 top-p 处理，并观察不同参数如何改变选词结果。

```python
import numpy as np


def softmax(logits):
    """数值稳定的 softmax。"""
    shifted = logits - np.max(logits)
    exp_values = np.exp(shifted)
    return exp_values / exp_values.sum()


def sample_from_logits(
    logits,
    tokens,
    temperature=1.0,
    top_k=None,
    top_p=1.0,
    rng=None,
):
    """
    根据 logits 采样一个 Token。

    temperature = 0：按 greedy 方式选择最大 logit
    top_k = None：不启用 top-k
    top_p = 1.0：不启用 top-p
    """
    logits = np.asarray(logits, dtype=np.float64)

    if logits.ndim != 1 or len(logits) != len(tokens):
        raise ValueError("logits 必须是一维数组，并且与 tokens 等长")
    if temperature < 0:
        raise ValueError("temperature 不能小于 0")
    if not 0 < top_p <= 1:
        raise ValueError("top_p 必须位于 (0, 1]")

    # 工程中通常把 temperature=0 特殊处理为 greedy。
    if temperature == 0:
        index = int(np.argmax(logits))
        return tokens[index], softmax(logits)

    rng = np.random.default_rng() if rng is None else rng

    # 1. 温度调整：低温使分布更尖锐，高温使分布更平坦。
    filtered_logits = logits / temperature

    # 2. top-k：只保留分数最高的 k 个候选。
    if top_k is not None:
        if top_k < 1:
            raise ValueError("top_k 必须大于等于 1")

        k = min(top_k, len(filtered_logits))
        keep_indices = np.argpartition(filtered_logits, -k)[-k:]
        keep_mask = np.zeros(len(filtered_logits), dtype=bool)
        keep_mask[keep_indices] = True
        filtered_logits = np.where(keep_mask, filtered_logits, -np.inf)

    # 3. 先转为概率，为 top-p 的累计概率筛选做准备。
    probabilities = softmax(filtered_logits)

    # 4. top-p：保留累计概率达到 p 的最小候选集合。
    if top_p < 1.0:
        sorted_indices = np.argsort(probabilities)[::-1]
        sorted_probabilities = probabilities[sorted_indices]
        cumulative = np.cumsum(sorted_probabilities)

        # 第一个让累计概率超过 p 的候选仍然需要保留。
        remove_sorted = cumulative > top_p
        remove_sorted[1:] = remove_sorted[:-1]
        remove_sorted[0] = False

        remove_mask = np.zeros(len(probabilities), dtype=bool)
        remove_mask[sorted_indices] = remove_sorted
        probabilities[remove_mask] = 0.0
        probabilities /= probabilities.sum()

    # 5. 按最终概率抽样。
    chosen_index = int(rng.choice(len(tokens), p=probabilities))
    return tokens[chosen_index], probabilities


def show_distribution(name, logits, tokens, **sampling_options):
    """打印一次配置的最终概率，并重复采样以展示差异。"""
    rng = np.random.default_rng(7)

    _, probabilities = sample_from_logits(
        logits, tokens, rng=rng, **sampling_options
    )

    samples = [
        sample_from_logits(
            logits, tokens, rng=rng, **sampling_options
        )[0]
        for _ in range(12)
    ]

    readable = {
        token: f"{probability:.1%}"
        for token, probability in zip(tokens, probabilities)
        if probability > 0
    }

    print(f"\n{name}")
    print("最终概率：", readable)
    print("12 次选择：", samples)


tokens = ["苹果", "香蕉", "梨", "冰箱", "火星"]
logits = np.array([4.0, 3.4, 2.8, 1.2, 0.4])

show_distribution(
    "Greedy",
    logits,
    tokens,
    temperature=0,
)

show_distribution(
    "较高温度，不截断候选",
    logits,
    tokens,
    temperature=1.5,
)

show_distribution(
    "temperature=0.8 + top-k=3",
    logits,
    tokens,
    temperature=0.8,
    top_k=3,
)

show_distribution(
    "temperature=0.8 + top-p=0.8",
    logits,
    tokens,
    temperature=0.8,
    top_p=0.8,
)
```

`softmax` 先减去最大 logit，是为了避免指数运算产生过大的数值。所有 logits 同时减去同一个常数不会改变最终的概率比例。

`temperature=0` 被单独处理为 greedy。此时函数直接选择最大 logit 对应的词，而不执行除以零的运算。

top-k 使用掩码把候选范围之外的 logits 设置为负无穷。经过 softmax 后，这些候选的概率就会变为零。

top-p 先对概率降序排列，再计算累计概率。代码将超过阈值之前的候选保留，并且保留第一个使累计概率达到或超过阈值的候选，从而形成满足阈值要求的最小集合。

最后，`rng.choice` 根据处理后的概率抽样。示例使用固定随机种子，使演示结果较容易复现；相同概率分布在不同随机种子下仍可能得到不同选择。

运行示例时，greedy 会始终选择“苹果”。提高 temperature 后，“香蕉”“梨”乃至低分候选更可能出现；使用 top-k 或 top-p 后，不合理的低分候选会被直接排除。

## 常见误区

### 误区一：top-p 是选择“概率排名前 p% 的词”

不是。`top-p = 0.9` 并不表示保留排名前 90% 的 Token，也不表示保留单个概率大于 90% 的 Token。

它表示从概率最高的候选开始累加，保留累计概率达到 90% 所需的最小候选集合。这个集合可能只有两个 Token，也可能包含几百个 Token。

### 误区二：temperature 越高，模型掌握的知识就越多

temperature 不会为模型增加知识，也不会让 logits 本身变得更正确。它只改变候选概率分布的尖锐程度，从而影响选择的随机性。

高温度可能带来新颖表达，也可能提高跑题和错误的概率。低温度通常更稳定，但无法保证事实一定正确。

### 误区三：temperature=0 时，每次输出一定逐字符完全相同

`temperature=0` 通常意味着 greedy，因此在模型版本、完整输入、分词、停止条件和运行实现都固定时，结果通常更容易复现。

但模型是按 Token 而不是按字符生成的，而且系统提示词变化、模型升级、服务端处理差异、并列分数处理或底层计算的非确定性，都可能使结果不同。因此不能把 `temperature=0` 理解成跨环境的绝对字符级保证。

### 误区四：KV cache 会改变模型选择下一个 Token 的规则

KV cache 的主要作用是复用历史 Token 已经计算出的中间结果，减少重复计算。理想情况下，开启或关闭它应得到等价的 logits 和采样分布。

实际系统可能因数值精度、缓存量化或实现差异出现微小偏差，但 KV cache 的设计目的不是增强知识或主动改变输出，而是降低生成延迟。

## 苏格拉底式检查

**问题一：** 假设某一步只有三个候选，其概率分别为 60%、25% 和 15%。使用 `top-p = 0.7` 时，应保留哪些候选？

**提示与答案：** 先保留第一个候选，累计概率是 60%，还没有达到 70%；加入第二个候选后累计为 85%，第一次达到阈值。因此保留前两个候选，再把它们重新归一化。

**问题二：** 为什么流式输出中，后面的 Token 通常要等待前面的 Token 生成，而不能一次并行生成整段？

**提示与答案：** 因为模型采用自回归生成。第 `n+1` 个 Token 的概率依赖包括第 `n` 个 Token 在内的当前上下文，所以必须先确定前一个 Token，才能准确开始下一轮预测。KV cache 能减少每轮的重复计算，却不能消除这种先后依赖。

---

## 快速总结

- LLM 的生成过程是反复预测并选择下一个 Token，而不是一次性写完整段文本。
- 文本依次经过 Tokenizer、Token IDs、Embedding 和 Transformer，最终得到词表中每个 Token 的 logits。
- softmax 把 logits 转为概率；temperature 通过 `logits / T` 调节概率分布的尖锐程度。
- Greedy 永远选择当前最高分候选，top-k 保留固定数量的高分候选，top-p 保留累计概率达到阈值的最小候选集合。
- temperature、top-k 和 top-p 可以组合，但参数越严格并不一定越好，应根据任务和模型测试。
- KV cache 复用历史 Token 的注意力中间结果，使逐 Token 生成和流式输出拥有更低延迟。

## 追问与解答

### Q1（2026-07-22）为什么是 ID 被转换成高维向量，而不是 Token 本身

**问：** 为什么是id被转换成高维向量？而不是token本身呢

**答：**

好问题，这其实是个「表述顺序」带来的小误会：并不是 ID 有什么特殊魔力，而是**「Token 本身」是字符串，神经网络根本没法对字符串做数学运算**喵。所以必须先把它变成某种数字形式，才能进入网络喵。

具体来说是这样一条链喵：

1. **Token 是文字碎片**（比如 `"天气"`），它只是人类可读的符号，没有可以加减乘除的数值含义喵。
2. **ID 只是它在词表里的行号**（比如 `2456`），作用等同于图书馆的索书号——它自己不携带任何语义，`2456` 并不比 `892` 「大」或「更重要」喵。
3. **Embedding 表是一个大矩阵**：词表有 10 万个 Token，向量维度是 4096，那这张表就是 100000 行 × 4096 列喵。「ID 转换成向量」的真实操作，就是**拿 ID 当行号，去这张表里取出第 2456 行**——一次查表，仅此而已喵。

所以更准确的说法是：**被转换成向量的就是 Token 本身，ID 只是查表用的门牌号**喵。语义并不存在 ID 里，而是存在 Embedding 表那一行行「被训练出来的数字」里——训练过程中模型不断调整每一行的数值，才让「天气」和「气候」的向量变得相近喵。

用类比说：学号（ID）→ 储物柜（Embedding 表的某一行）→ 柜子里的东西（语义向量）喵。学号本身不说明这个学生是谁，它只负责让你找到正确的柜子；真正描述这个学生的，是柜子里存的内容喵。

补一个数学视角：查表操作在数学上完全等价于「把 Token 写成 one-hot 向量（10 万维、只有第 2456 位是 1），再乘以 Embedding 矩阵」喵。乘出来的结果恰好就是矩阵的第 2456 行，所以工程上直接用 ID 索引，省掉那次浪费的大矩阵乘法喵。

反过来想你就更清楚为什么绕不开这一步：如果不查表，你想让「Token 本身」直接进网络，那也得先把字符编码成数字（比如字节值），而那同样是一种「离散符号 → 数字编号」的映射——只是粒度更细而已，本质上仍然是「先变编号、再查表变向量」这条路喵。

### Q2（2026-07-22）为什么温度设为 0，每次回答还是不一样

**问：** 为什么有些模型温度设置为0，每次回答还是不一样？

**答：**

这是个非常好的追问，正好接上了会话里「误区三」的话题喵。temperature=0 只是把「采样」这一步变成了确定性的 greedy（永远选 logits 最大的那个），但它**管不到 logits 本身是怎么算出来的**——只要两次前向计算得到的 logits 有一丁点不同，argmax 就可能翻转，后面的文字就整段分叉了喵。

不确定性主要来自这几个来源喵：

1. **浮点数加法不满足结合律**：计算机里 `(a+b)+c` 和 `a+(b+c)` 的结果可能在小数点后十几位不一样；GPU 大规模并行计算时，成千上万个乘加的求和顺序每次都可能不同，所以 logits 会带有约 1e-6 量级的抖动喵。平时无所谓，但当第一名和第二名的 logit 非常接近时，这点抖动足以让 argmax 选出不同的词——而自回归生成会把差异滚雪球式放大：一个词变了，后面整段都跟着变喵。

2. **动态批处理（dynamic batching）**：线上服务会把你的请求和别人的请求拼成一个 batch 一起算；batch 的大小和形状不同，底层会选择不同的计算 kernel、采用不同的求和顺序，数值结果就不完全一致喵。你无法控制自己和谁「拼车」，所以同样的输入两次得到的 logits 就可能不同——这是商业 API 上最主要的来源喵。

3. **MoE（Mixture of Experts，专家混合）架构**：这类模型每个 Token 要被路由到某几个「专家」子网络，路由决策对数值抖动和 batch 组成更敏感，还存在专家容量限制——同 batch 里别人的 Token 可能把某个专家「挤满」，你的 Token 就被分给了别的专家喵。所以 MoE 类模型在 temperature=0 时的不确定性往往格外明显喵。

4. **服务端本身不承诺严格 greedy**：有些实现把 temperature=0 处理成「一个趋近于 0 的小温度」，有些在遇到并列分数时做随机 tie-breaking；再加上服务端模型静默升级、注入的系统提示词变化等，都会让两次结果不同喵。

反过来验证这个理解：如果你在**本地**跑开源模型，固定随机种子、batch 固定为 1、启用确定性算子，temperature=0 通常就能做到每次逐字一致喵。这说明「不一样」不是模型「故意随机」，而是**分布式浮点计算 + 共享服务架构**层面的非确定性喵。

工程启示：不要把 temperature=0 当成「可复现」的保证喵。需要严格一致的输出时，靠的是约束输出格式（JSON Schema、枚举选项）、结果校验和缓存命中，而不是指望模型逐字符复现喵。

### Q3（2026-07-22）读取了缓存，回答就会一样了吗

**问：** 那么如果读取了缓存，回答就会一样了对吧

**答：**

对了一半，关键要区分你说的是**哪一层缓存**喵——这两种「缓存」经常被混为一谈，结论正好相反喵。

**如果是「回答缓存」（response cache）：是的，完全一样喵。** 这种缓存存的是**最终生成好的文本**：系统发现这次请求和之前某次完全相同，就直接把存下来的答案原样返回，模型根本没有被调用喵。没有前向计算，自然没有任何不确定性——你读到的是「录像回放」，不是「重新表演」喵。我上一轮说的「靠缓存命中保证一致」指的就是这一种喵。

**如果是 KV cache 或 prompt caching（比如各大 API 的「提示词缓存」）：不是，回答仍然可能不同喵。** 这类缓存存的是**前缀部分的 K、V 中间计算结果**，作用只是让模型不用重算你输入里重复的那一段，省钱、省时间喵。但接下来的每一个新 Token 依然要**实打实地跑前向计算 + 采样**——上一轮说的浮点抖动、动态批处理、MoE 路由这些不确定性来源一个都没少，所以两次请求即使命中了同样的 prompt cache，生成出的文字仍可能分叉喵。

一句话区分：**缓存「答案」才能保证一样；缓存「计算中间结果」只能保证更快更便宜，不保证一样**喵。

顺带把这个理解闭环一下：这也解释了为什么 KV cache 被归类为「性能优化」而不是「行为改变」——它的设计目标就是在**不改变**概率分布的前提下省计算，而「不改变分布」不等于「消除随机性」喵。
