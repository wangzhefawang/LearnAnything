## Positioning

LoRA 属于训练与对齐子域，通常用于在预训练模型之上进行监督微调（SFT），也可用于 RLHF 等对齐流程中的策略模型适配；它不是全量微调之外的另一种训练目标，而是用少量可训练参数近似全量权重更新的一种高效实现方式。

## Analogy

可以把一个预训练大模型想象成一套昂贵、做工精良的成衣西装。它已经掌握通用语言、知识和推理能力，就像西装的主体版型已经非常成熟。现在，你希望它适合一个具体任务，例如理解机器人指令、根据视觉观察输出动作，或模仿专家轨迹。

全量微调相当于把整套西装拆开：领口、袖子、内衬、纽扣和每一片布料都允许重新裁剪。这样自由度最高，但成本也很大，需要保存和调整大量参数。如果每位客户都需要一套完整修改后的西装，存储和管理也会变得困难。

LoRA 更像是在几个关键位置增加可调节的“小裁片”：缩短一点袖口、调整一点腰线、改变领口的贴合方式。西装主体保持不动，只训练这些新增的小部件。由于很多任务并不需要重建模型的全部能力，而只是需要对原有行为进行有方向的调整，这些小部件往往已经足够。

训练完成后，这些修改可以单独保存。一个通用底座模型可以搭配“机器人抓取 LoRA”“视觉问答 LoRA”或“室内导航 LoRA”等不同适配器。需要部署时，还可以把裁片对应的修改合并回原权重，得到一套已经改好的完整西装。

这个类比也揭示了 LoRA 的边界：如果任务只是调整已有能力的使用方式，小范围修改往往很有效；如果新任务与原模型相差极远，或者需要重建大量底层表征，几块小裁片就未必足够。

## Core Mechanism

Transformer 内部包含大量权重矩阵，例如注意力模块中的 Q、K、V 投影，以及输出投影和前馈网络。设其中一个预训练权重矩阵为 `W ∈ R^{d×k}`。全量微调会直接训练 `W`，得到新的权重 `W' = W + ΔW`。

这种方式昂贵的不只是因为要保存模型参数。训练时还要保存参数梯度；使用 AdamW 等优化器时，通常还要为每个可训练参数维护一阶动量和二阶动量；混合精度训练有时还会保留额外的高精度参数副本。再加上反向传播所需的激活值，训练显存往往远大于模型文件本身。

LoRA 的关键观察是：微调虽然可能改变一个很大的权重矩阵，但有效更新 `ΔW` 往往集中在一个低维子空间中，也就是具有较低的本征秩。直观地说，模型已经在预训练阶段学会了大量通用模式，下游微调通常不需要独立地改变矩阵中的每一个元素，而更像是沿少数几个重要方向调整模型行为。

因此，LoRA 不直接训练完整的 `ΔW`，而是用两个较小矩阵的乘积近似它：

`W' = W + B·A`

其中，`B ∈ R^{d×r}`，`A ∈ R^{r×k}`，并且 `r ≪ min(d, k)`。这里的 `r` 称为秩或低秩维度。训练期间，原始权重 `W` 被冻结，只更新 `A` 和 `B`。

为了让训练刚开始时模型行为与底座模型一致，常见初始化方式是随机初始化 `A`，将 `B` 初始化为零。此时 `B·A = 0`，所以初始状态下仍然有 `W' = W`，训练随后逐渐学出有用的增量。

LoRA 实际计算时通常使用缩放后的形式：

`W' = W + (α/r)·B·A`

其中 `α` 对应常见配置项 `lora_alpha`，`α/r` 是缩放因子。它控制 LoRA 更新相对于原始权重的影响强度。使用缩放因子可以让不同 `r` 下的更新幅度更容易控制，但 `r` 和 `α` 仍然需要结合任务、模型与学习率调节。

另一个常用超参数是 `lora_dropout`。它只作用于 LoRA 分支，训练时随机丢弃部分输入，以降低小数据集上的过拟合风险。数据非常充足时可以设为零或较小值；数据量较少、样本重复度较高时，适当的 dropout 往往更稳健。

LoRA 最常放在注意力模块的 `q_proj` 和 `v_proj` 上。`q_proj` 影响模型如何提出查询，`v_proj` 影响被聚合的信息内容，因此只修改这两处通常能以很少的参数获得不错效果。更充分的配置还可能覆盖 `k_proj`、`o_proj`，甚至前馈网络中的 `gate_proj`、`up_proj` 和 `down_proj`。覆盖的模块越多，可训练容量越大，但显存、训练时间和过拟合风险也会增加。

参数节省可以通过一个具体例子理解。假设 `W` 的形状为 `4096 × 4096`，全量训练这个矩阵需要更新 `16,777,216` 个参数。如果使用 `r = 8` 的 LoRA，那么两个低秩矩阵一共只有：

`4096 × 8 + 8 × 4096 = 65,536`

也就是原矩阵参数量的约 `0.39%`。如果同时对同样大小的 `q_proj` 和 `v_proj` 使用 LoRA，可训练参数为 `131,072`，而全量训练这两个矩阵需要更新 `33,554,432` 个参数。

LoRA 不会消除全部训练成本。模型仍然要完成前向传播和反向传播，也仍需保存一部分激活值，但冻结底座后，不再需要为大多数参数保存梯度和优化器状态，因此显存需求通常会显著下降。

推理时有两种使用方式。第一种是保留底座模型和独立 LoRA 适配器，在运行时计算 LoRA 分支，这便于热切换多个适配器，但会引入少量额外计算。第二种是提前计算 `W ← W + (α/r)·B·A`，把增量合并到原权重中。合并后模型结构与普通模型相同，因此通常没有 LoRA 分支带来的额外推理延迟，但也失去了即时切换适配器的便利。

QLoRA 则是在 LoRA 基础上进一步节省显存：先把冻结的底座模型量化到 4 bit 等低精度格式，再以较高精度训练 LoRA 参数。可以把它理解为“量化底座 + LoRA”，而不是一种完全不同的微调目标。

在 VLM 或 VLA 中，模型通常不只有语言模型，还包含视觉编码器、视觉到语言的投影层，以及动作头等组件，因此“LoRA 加在哪里”会直接决定适配重点。

如果视觉塔已经能够稳定提取任务所需的视觉特征，而主要问题是理解机器人指令、融合视觉与语言上下文，或生成正确的动作 token，那么常见做法是冻结视觉塔，主要给 LLM 的注意力层添加 LoRA，同时训练视觉投影层或动作头。这种方案成本较低，也更容易保留底座模型的通用视觉能力。

如果机器人图像与视觉塔的预训练分布差异很大，例如使用特殊的腕部相机、深度图、热成像、工业场景或极端视角，那么只调整 LLM 可能不够。此时可以给视觉塔的注意力投影增加 LoRA，让视觉表征适应新的相机和环境。代价是训练显存更高，并且在数据不足时更容易破坏原有视觉泛化能力。

更强但也更昂贵的方案，是同时在视觉塔和 LLM 上使用 LoRA，并单独训练跨模态投影层与动作头。实践中可以先从“冻结视觉塔，仅适配 LLM 和投影层”开始；只有当错误分析显示模型确实没有识别出任务相关视觉线索时，再逐步解冻或为视觉塔添加 LoRA。

在具身智能论文常见的模仿学习式微调中，训练数据通常是专家演示轨迹，模型学习从视觉观察、语言指令和历史状态映射到动作或动作序列。LoRA 将更新限制在较小的参数子空间内，通常比全量微调更不容易覆盖底座模型原有的通用语言和视觉能力，因此有助于减轻灾难性遗忘。

不过，“减轻”并不等于“彻底避免”。如果演示数据过于单一、训练轮数过多或学习率过高，LoRA 同样可能让模型过度适应某些环境和动作模式。稳健的做法仍然包括保留通用数据、进行跨场景验证、使用早停，并分别评估任务成功率与底座能力保持程度。

## Code Example

下面的示例使用 Hugging Face Transformers 和 PEFT，对一个小型因果语言模型添加 LoRA，并用几条指令数据演示训练、保存适配器和合并权重的完整流程。

```python
import torch
from datasets import Dataset
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer,
    DataCollatorForLanguageModeling,
    Trainer,
    TrainingArguments,
)
from peft import LoraConfig, TaskType, get_peft_model

# 1. 加载底座模型和分词器
model_name = "Qwen/Qwen2.5-0.5B-Instruct"

tokenizer = AutoTokenizer.from_pretrained(model_name)
if tokenizer.pad_token is None:
    tokenizer.pad_token = tokenizer.eos_token

model = AutoModelForCausalLM.from_pretrained(
    model_name,
    torch_dtype="auto",
)

# 训练时关闭 KV Cache，避免与梯度训练冲突并节省部分显存
model.config.use_cache = False

# 2. 定义 LoRA 配置
lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=8,
    lora_alpha=16,
    lora_dropout=0.05,
    target_modules=["q_proj", "v_proj"],
    bias="none",
)

# 3. 将 LoRA 注入底座模型
model = get_peft_model(model, lora_config)
model.print_trainable_parameters()

# 4. 准备一个最小训练数据集
examples = {
    "text": [
        "用户：机器人看到桌上的红色方块，应如何抓取？\n助手：先定位红色方块，再移动机械臂并从上方闭合夹爪。",
        "用户：目标被遮挡时应立即抓取吗？\n助手：不应立即抓取，应先调整视角并重新确认目标位置。",
        "用户：机械臂接近易碎物体时应该怎样控制？\n助手：降低移动速度和夹爪力度，并持续检查接触状态。",
        "用户：收到停止指令后应该做什么？\n助手：立即停止当前动作，并将执行状态反馈给操作者。",
    ]
}
dataset = Dataset.from_dict(examples)

def tokenize_batch(batch):
    return tokenizer(
        batch["text"],
        truncation=True,
        max_length=256,
    )

tokenized_dataset = dataset.map(
    tokenize_batch,
    batched=True,
    remove_columns=["text"],
)

data_collator = DataCollatorForLanguageModeling(
    tokenizer=tokenizer,
    mlm=False,
)

# 5. 配置并启动演示训练
training_args = TrainingArguments(
    output_dir="./lora_demo_output",
    per_device_train_batch_size=1,
    gradient_accumulation_steps=4,
    num_train_epochs=1,
    learning_rate=2e-4,
    logging_steps=1,
    save_strategy="no",
    report_to="none",
    fp16=torch.cuda.is_available(),
)

trainer = Trainer(
    model=model,
    args=training_args,
    train_dataset=tokenized_dataset,
    data_collator=data_collator,
)

trainer.train()

# 6. 单独保存 LoRA 适配器
model.save_pretrained("./robot_lora_adapter")
tokenizer.save_pretrained("./robot_lora_adapter")

# 7. 将 LoRA 增量合并到底座权重
model.config.use_cache = True
merged_model = model.merge_and_unload()

merged_model.save_pretrained(
    "./robot_merged_model",
    safe_serialization=True,
)
tokenizer.save_pretrained("./robot_merged_model")
```

第一段加载普通的因果语言模型。这里没有量化底座，因此它是标准 LoRA 示例，而不是 QLoRA。`torch_dtype="auto"` 会尽量采用模型推荐的数据类型；在显存较小的设备上，可以进一步使用量化加载，但需要额外配置 BitsAndBytes。

第二段通过 `LoraConfig` 定义适配器。`task_type=TaskType.CAUSAL_LM` 告诉 PEFT 当前任务是因果语言建模；`r=8` 指定低秩维度；`lora_alpha=16` 使缩放因子为 `16/8 = 2`；`lora_dropout=0.05` 为 LoRA 分支加入少量随机丢弃。

`target_modules=["q_proj", "v_proj"]` 表示只在注意力层的查询投影和值投影中加入 LoRA。不同模型的模块名称可能不同，因此换用其他架构时，应先检查模型结构，不能假设所有模型都使用 `q_proj` 和 `v_proj` 这两个名称。

`get_peft_model` 会冻结原始模型参数，并在目标模块中注入 LoRA 参数。随后调用 `print_trainable_parameters()`，可以看到可训练参数数量及其占总参数的比例。这个比例通常远低于全量微调。

数据部分只是一个可以说明 API 用法的最小示例。真实的指令微调通常要明确区分用户输入和目标回答，并对非回答部分的 token 设置 `labels=-100`，避免模型把提示词也当作学习目标。VLA 的模仿学习数据还会包含图像、状态、动作 token 或连续动作标签，需要使用与具体模型匹配的数据处理器和损失函数。

`Trainer` 负责完成训练循环。LoRA 通常可以使用比全量微调稍高的学习率，例如 `1e-4` 到 `3e-4`，但合理范围仍取决于模型规模、数据量、目标模块和批量大小。这个示例只有四条数据，只用于展示完整流程，不能得到具备实际泛化能力的机器人策略。

训练后，`save_pretrained` 默认只保存 LoRA 配置和增量参数，因此文件通常远小于完整模型。部署时需要同时拥有底座模型和适配器。

最后，`merge_and_unload()` 将 LoRA 增量合并到原权重，并移除 LoRA 包装层。得到的 `merged_model` 可以像普通 Transformers 模型一样保存和推理。需要注意，合并后的完整模型文件会恢复到底座模型对应的体积，而不再只是一个轻量适配器。

## Common Misconceptions

**误区一：LoRA 训练后得到的是一个完全独立的新模型。**

默认保存的是相对于底座模型的增量适配器，而不是完整模型。加载时仍需要原底座及兼容的模型版本。只有执行权重合并并保存之后，才会得到可以独立加载的完整模型权重。

**误区二：`r` 越大，效果一定越好。**

更大的 `r` 提供更强的表达能力，但也会增加可训练参数、显存需求和过拟合风险。如果任务所需更新本来就处于很低维的子空间，更大的秩可能没有明显收益。通常应从 `r=8` 或 `r=16` 等较小值开始，通过验证集和跨场景评估决定是否增加。

**误区三：LoRA 可以无条件达到全量微调的能力上限。**

LoRA 对“调整已有能力”非常有效，但低秩约束限制了更新空间。面对巨大的领域差异、新模态输入或需要深度重建表征的任务，全量微调可能具有更高上限。LoRA 的优势是成本、存储和能力保持之间的平衡，而不是在所有情况下都严格优于全量微调。

**误区四：使用 LoRA 后，推理一定会变慢。**

如果保留独立 LoRA 分支，推理时确实会增加少量矩阵运算。但如果将 LoRA 权重合并到底座权重，通常不会产生额外推理延迟。是否合并取决于部署时更看重吞吐量，还是更看重多个适配器的动态切换能力。

**误区五：LoRA、Adapter 和 Prefix-tuning 是同一种方法。**

它们都属于参数高效微调，但修改位置不同。LoRA 为已有线性层的权重更新建立低秩分解；传统 Adapter 通常在网络层之间插入小型神经网络；Prefix-tuning 则学习可注入注意力计算的连续前缀。它们的训练成本、推理开销和适用场景并不完全相同。

## Socratic Check

**问题一：一个 VLA 模型能正确识别桌面物体，也能理解语言指令，但经常生成错误的动作序列。你会优先给视觉塔还是 LLM 添加 LoRA？**

提示：先判断错误发生在哪个环节。既然视觉识别已经可靠，主要瓶颈更可能是跨模态决策和动作生成，因此可以先冻结视觉塔，为 LLM 添加 LoRA，并训练视觉投影层或动作头。如果后续发现换相机或光照后连物体都识别不准，再考虑适配视觉塔。

**问题二：某个 `4096 × 4096` 权重矩阵使用 `r=16` 的 LoRA，需要训练多少个低秩参数？**

参考方向：`A` 有 `16 × 4096` 个参数，`B` 有 `4096 × 16` 个参数，因此总数为 `131,072`。这只计算 LoRA 参数，不包括被冻结的原始矩阵。

## Quick Summary

- LoRA 冻结预训练权重 `W`，通过低秩矩阵 `A` 和 `B` 学习增量，使 `W' = W + (α/r)·B·A`。
- 它显著减少了可训练参数、梯度和优化器状态的显存开销，但仍需要完成底座模型的前向与反向计算。
- `r`、`lora_alpha`、`lora_dropout` 和 `target_modules` 是核心超参数，常见起点是在 `q_proj` 和 `v_proj` 上添加 LoRA。
- LoRA 适配器可以独立保存和切换，也可以合并到底座权重，从而避免额外的推理分支。
- 在 VLM/VLA 中，应根据错误来源决定适配 LLM、视觉塔或两者；LoRA 有助于保留通用能力，但不能保证完全避免灾难性遗忘。

## Next Steps

🔍 **QLoRA 与量化微调**

继续理解 4 bit 底座量化、计算数据类型和量化误差，可以帮助你在单张消费级 GPU 上微调更大的模型，同时判断节省显存是否会影响具身任务表现。

🔍 **LoRA 超参数调优与秩的选择**

深入研究 `r`、`lora_alpha`、学习率和目标模块之间的关系，可以避免简单地把更大秩等同于更好效果，并建立基于验证集和错误分析的调参方法。

🔍 **多 LoRA 适配器的管理与合并**

不同任务、机器人平台或环境可以共享一个底座并维护多个适配器。理解适配器热切换、权重合并和多适配器组合，有助于构建模块化的机器人能力系统。

🔍 **VLA 模型中的参数高效微调策略**

进一步比较只适配 LLM、只适配视觉塔、联合适配以及训练动作头的方案，并分别评估感知、指令理解、动作预测和跨场景泛化能力。
