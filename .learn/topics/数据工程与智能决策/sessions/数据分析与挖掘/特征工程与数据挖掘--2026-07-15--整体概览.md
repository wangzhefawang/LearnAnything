## Analogy

前一节可以把多源数据看成送进中央厨房、已经完成验货和清洗的食材喵。蔬菜已经去掉烂叶，肉类已经核对保质期，不同包装也贴上了统一标签，但这些食材还不是一道可以端上桌的菜喵。

特征工程就像厨房里的切配和调味喵。同一根胡萝卜可以切丝、切片或切丁，选择哪种形态取决于准备做汤、炒菜还是馅料喵。

同样，一条「2026 年 7 月 15 日 08:00，G01 某路段交通量为 5300 辆，车辙深度为 24 mm」的记录，也可以加工成「早高峰」「高交通量」「车辙接近处治阈值」「近 3 天新增病害 7 处」等特征喵。这些表达更接近模型需要回答的养护问题喵。

数据挖掘则像厨师根据切配好的食材完成烹饪和判断喵。有时要判断一道菜属于清淡还是重口，这类似分类喵。

有时要估计还需烹煮多少分钟，这类似回归喵。有时没有预先规定菜系，而是让厨师根据味道把菜自动分组，这类似聚类喵。

因此，验货、切配和烹饪是连续环节喵。清洗保证食材可信，特征工程把食材加工成适合任务的形态，数据挖掘再利用这些特征发现模式或作出判断喵。

## Core Mechanism

### 1．什么是特征喵

特征是对一个分析对象进行描述、并能被计算方法使用的变量喵。在公路养护场景中，分析对象可以是一条路段、一个时间窗口或一次病害事件，特征则可以是平均交通量、最大车辙深度、道路等级、降雨后 24 小时新增裂缝数等喵。

原始数据通常不能原封不动地直接交给模型喵。原因不是原始数据一定错误，而是它的表达方式往往与计算任务不匹配喵。

例如，`高速公路` 是文字，`2026-07-15T08:00:00` 是时间戳，交通量和车辙深度的数值尺度相差很大，而单条巡查记录又没有直接表达一条路段最近一周的病害趋势喵。特征工程就是把这些原始字段转成含义明确、尺度合适、粒度一致的输入喵。

### 2．常见的特征工程手段喵

| 手段 | 它解决什么问题 | 公路养护示例 |
| --- | --- | --- |
| 数值标准化 | 标准化通常把数值转成以均值为中心、以标准差衡量距离的尺度，适合比较不同量纲的变量喵 | 把交通量和车辙深度标准化后，模型不会仅因交通量数字更大就误以为它更重要喵 |
| 数值归一化 | 归一化常把数值压缩到 `0～1` 等固定范围，便于形成可比较的评分喵 | 把各路段交通量按最小值和最大值缩放到 `0～1`，再与病害特征组合计算养护优先级喵 |
| 数值分箱 | 分箱把连续数值转换成少量有业务含义的区间，降低细小波动带来的干扰喵 | 把小时分成夜间、早高峰、平峰和晚高峰，或把车辙深度分成正常、关注和严重喵 |
| 类别 one-hot 编码 | one-hot 编码为每个无天然顺序的类别建立一个 `0/1` 开关，避免模型把类别名称当数字大小比较喵 | 道路等级为高速公路时令 `道路等级_高速公路=1`，同时令国道和县道对应特征为 `0`喵 |
| 类别序数编码 | 序数编码把确实存在先后或强弱关系的类别映射为有序数字喵 | 可以把病害严重度的轻度、中度、重度依次编码为 `1、2、3`，但不能随意把不同路线名称编码成大小关系喵 |
| 时间特征 | 时间特征从时间戳中提取模型更容易使用的周期和业务阶段喵 | 从采集时间提取小时、星期、月份、是否节假日、汛期标记和早晚高峰标记喵 |
| 聚合与窗口特征 | 聚合特征把多条记录汇总为对象级指标，窗口特征则只观察当前时点之前的一段范围喵 | 计算某路段近 3 天新增病害数、近 7 天平均交通量、近 30 天最大车辙深度和连续超限天数喵 |
| 交叉特征 | 交叉特征把两个或多个条件组合起来，表达单个字段无法体现的共同作用喵 | 「重载交通占比 × 路面服役年限」可以表达老旧路面在重载作用下的风险，而不仅是分别观察两个数字喵 |
| 缺失标记 | 缺失标记单独记录某个值原本是否缺失，避免填补后丢掉重要信息喵 | 车辙深度用中位数填补时，再增加 `车辙深度_原始缺失=1`，提醒模型和业务人员该值不是实测结果喵 |

标准化、归一化和分箱不是互相替代的固定步骤，而是针对不同任务选择的工具喵。例如，养护评分可能偏好容易解释的 `0～1` 归一化，某些距离计算则可能更适合标准化，而面向业务规则的报表常使用分箱喵。

窗口特征尤其需要明确观察边界喵。如果要在 7 月 15 日早晨判断某路段是否应优先养护，那么「近 3 天病害数」只能使用 7 月 15 日当时以及更早的数据，不能偷看 7 月 16 日才发生的病害喵。

### 3．什么是数据挖掘喵

数据挖掘不是单一算法，而是一组从数据中发现模式、结构、关系和异常的任务喵。它关注的问题类型不同，得到的结果形式也不同喵。

| 任务族 | 要回答的问题 | 公路场景示例 |
| --- | --- | --- |
| 分类 | 一个对象属于哪个离散类别喵 | 根据交通量、车辙、裂缝和道路等级，把路段判断为高、中、低养护优先级喵 |
| 回归 | 一个对象对应的连续数值是多少喵 | 预测某路段未来 30 天的车辙深度，或估计下一年度养护费用喵 |
| 聚类 | 在没有预先标签时，哪些对象彼此相似喵 | 按交通负荷、病害组合和路龄把路段自动分成重载老化型、排水敏感型和整体稳定型等群组喵 |
| 关联规则 | 哪些事件或条件经常一起出现喵 | 发现「连续强降雨 + 排水能力较弱」经常与「坑槽和路基水损害同时增加」共同出现喵 |
| 异常检测 | 哪些对象明显偏离通常模式喵 | 识别某传感器突然卡值、某路段病害数量异常激增，或视频 AI 在短时间内重复上报同一事件喵 |

分类和回归通常需要历史答案作为标签，例如过去哪些路段被判定为高优先级、实际车辙增长了多少喵。聚类、关联规则和异常检测则可以在没有完整标签时帮助寻找结构，但它们发现的模式仍需结合公路业务知识验证喵。

### 4．特征工程与数据挖掘的关系喵

特征工程决定模型看见什么，数据挖掘方法决定模型如何利用所看见的信息喵。如果输入只有某一天的一条车辙记录，模型很难理解长期恶化趋势；如果增加近 30 天斜率、连续超限天数和重载交通占比，它才可能识别更有意义的养护风险喵。

因此，特征质量往往决定数据挖掘效果的上限喵。所谓「garbage in, garbage out」，就是输入若含义错误、粒度混乱或与目标无关，再复杂的算法也只会更快地产生不可靠结果喵。

不过，特征越多并不一定越好喵。重复、噪声过大或无法在实际决策时获得的特征，都会让结果更难解释，甚至造成虚假的高准确率喵。

### 5．警惕特征泄漏喵

特征泄漏（data leakage）是指构造特征时，不小心使用了预测时本来不可能知道的信息喵。例如，用「养护完成后的复检结果」去预测养护前的优先级，或者用未来 7 天病害数量去判断今天是否高风险，都会让离线结果看起来很好，却无法在真实业务中复现喵。

一个实用检查方法是为每个特征追问：「在作出这次养护决策的那个时刻，这个值是否已经产生、是否已经入库、是否真的可用喵」只要答案是否定的，就不应把它作为当时的模型输入喵。

## Code Example

下面的脚本只使用 Python 标准库，模拟 3 条公路路段连续 4 天的交通量和病害记录喵。它会构造时段分箱、道路等级 one-hot、近 3 天病害数、交通量与车辙深度归一化、车辙缺失标记，再用一条透明的手写评分规则完成一次小型数据挖掘，为路段排列养护优先级喵。

```python
from copy import deepcopy
from datetime import date
from statistics import median

def time_bucket(hour):
    # 把原始小时转换成有业务含义的时段分箱喵
    if 0 <= hour < 7 or 21 <= hour <= 23:
        return "夜间"
    if 7 <= hour < 10:
        return "早高峰"
    if 17 <= hour < 20:
        return "晚高峰"
    return "平峰"


def min_max(value, minimum, maximum):
    # 最小值与最大值相同时返回零，避免除以零喵
    if maximum == minimum:
        return 0.0
    return (value - minimum) / (maximum - minimum)


def build_features(raw_records, window_days=3):
    # 使用深拷贝，保证特征加工不会覆盖原始记录喵
    records = deepcopy(raw_records)

    # 用已有车辙实测值的中位数填补缺失值喵
    measured_ruts = [
        row["rut_depth_mm"]
        for row in records
        if row["rut_depth_mm"] is not None
    ]
    rut_fill_value = median(measured_ruts)

    for row in records:
        row["rut_was_missing"] = int(row["rut_depth_mm"] is None)
        if row["rut_depth_mm"] is None:
            row["rut_depth_mm"] = rut_fill_value
        row["time_bucket"] = time_bucket(row["hour"])
        row["parsed_day"] = date.fromisoformat(row["day"])

    # 为无天然顺序的道路等级构造 one-hot 特征喵
    road_grades = ["高速公路", "国道", "县道"]
    for row in records:
        for grade in road_grades:
            row[f"road_grade_{grade}"] = int(row["road_grade"] == grade)

    # 按路段和日期排序，保证窗口只看当前及过去记录喵
    records.sort(key=lambda row: (row["segment"], row["parsed_day"]))
    histories = defaultdict(lambda: deque(maxlen=window_days))
    for row in records:
        history = histories[row["segment"]]
        history.append(row["new_distress_count"])
        row[f"distress_count_{window_days}d"] = sum(history)

    # 在全部示例记录的范围内计算归一化上下界喵
    traffic_values = [row["traffic_volume"] for row in records]
    rut_values = [row["rut_depth_mm"] for row in records]
    distress_key = f"distress_count_{window_days}d"
    distress_values = [row[distress_key] for row in records]

    for row in records:
        row["traffic_norm"] = min_max(
            row["traffic_volume"], min(traffic_values), max(traffic_values)
        )
        row["rut_norm"] = min_max(
            row["rut_depth_mm"], min(rut_values), max(rut_values)
        )
        row["distress_norm"] = min_max(
            row[distress_key], min(distress_values), max(distress_values)
        )

    return records, rut_fill_value


def mine_maintenance_priority(feature_rows):
    # 每个路段只取决策时点能够看到的最新一条记录喵
    latest_by_segment = {}
    for row in feature_rows:
        latest_by_segment[row["segment"]] = row

    ranked = []
    for row in latest_by_segment.values():
        # 评分权重用于演示透明规则，不代表真实行业标准喵
        score = (
            0.35 * row["rut_norm"]
            + 0.30 * row["traffic_norm"]
            + 0.25 * row["distress_norm"]
            + 0.10 * row["road_grade_高速公路"]
        )
        priority = "高" if score >= 0.65 else "中" if score >= 0.35 else "低"
        ranked.append({
            "segment": row["segment"],
            "day": row["day"],
            "time_bucket": row["time_bucket"],
            "road_grade_one_hot": {
                "高速公路": row["road_grade_高速公路"],
                "国道": row["road_grade_国道"],
                "县道": row["road_grade_县道"],
            },
            "distress_count_3d": row["distress_count_3d"],
            "traffic_norm": round(row["traffic_norm"], 3),
            "rut_norm": round(row["rut_norm"], 3),
            "rut_was_missing": row["rut_was_missing"],
            "score": round(score, 3),
            "priority": priority,
        })

    return sorted(ranked, key=lambda item: item["score"], reverse=True)


features, fill_value = build_features(RAW_RECORDS)
ranking = mine_maintenance_priority(features)

# 断言证明原始数据里的缺失值没有被原地改写喵
assert RAW_RECORDS[2]["rut_depth_mm"] is None

print(f"车辙缺失值使用中位数 {fill_value} mm 填补喵")
print("养护优先级挖掘结果如下喵")
for index, item in enumerate(ranking, start=1):
    print(
        f"{index}. {item['segment']}：优先级={item['priority']}，"
        f"评分={item['score']}，近 3 天病害数={item['distress_count_3d']}，"
        f"时段={item['time_bucket']}，道路等级 one-hot={item['road_grade_one_hot']}喵"
    )
```

`RAW_RECORDS` 是已经完成基础清洗的模拟标准层数据，每条记录都落在「路段 + 日期」粒度上喵。其中故意保留一处车辙深度缺失，以展示填补值与缺失标记如何同时存在喵。

`time_bucket` 把原始小时分成夜间、早高峰、晚高峰和平峰喵。这一步没有改变时间本身，而是把模型较难直接利用的时钟数字转成与交通运行更相关的业务阶段喵。

`build_features` 先使用车辙实测值的中位数填补缺失值，同时建立 `rut_was_missing` 标记喵。随后，它为高速公路、国道和县道建立 one-hot 特征，使道路等级不被误解为任意的数字大小关系喵。

窗口计算先按路段和日期排序，再用长度为 3 的队列累计当前及过去记录喵。因此，7 月 15 日的 `distress_count_3d` 只包含 7 月 13 日至 15 日的信息，不会偷看未来数据喵。

脚本再把交通量、车辙深度和近 3 天病害数归一化到 `0～1`喵。示例为了保持自包含而使用全部记录计算上下界；在正式预测项目中，应只用训练期数据确定填补值和归一化参数，再原样应用到未来数据，以避免特征泄漏喵。

`mine_maintenance_priority` 取每个路段在决策日可见的最新记录，并按车辙、交通量、近期病害和道路等级计算透明评分喵。这相当于一次非常小的分类与排序任务，输出高、中、低优先级，但真实养护决策还需结合预算、路面结构、现场复核和行业规范喵。

## Common Misconceptions

**误区一：特征就是原始表里的每一列喵**

原始字段只是构造特征的材料，不一定能直接表达分析问题喵。时间戳可以拆成高峰标记，逐条病害记录可以聚合成近 7 天数量，两个字段还可以交叉成新的风险指标喵。

**误区二：归一化以后，数据就变得更准确了喵**

归一化只改变数值尺度，不会修复错误值，也不会增加新的事实喵。把错误的车辙深度从 `999 mm` 缩放到 `1.0`，它仍然是错误数据，所以质量清洗必须发生在特征加工之前喵。

**误区三：特征越多，挖掘结果一定越好喵**

无关、重复和噪声特征会稀释真正有用的信号，并增加解释难度喵。选择特征时应围绕明确的养护问题，优先保留在实际决策时可获得、含义稳定且有业务依据的变量喵。

**误区四：模型离线准确率很高，就说明特征工程成功了喵**

高准确率可能来自特征泄漏，例如使用了养护后的复检数据或未来病害记录喵。必须按真实时间顺序划分数据，并逐项确认每个特征在决策时刻已经可用喵。

## Socratic Check

**问题一：要在 7 月 15 日 08:00 判断某路段当天是否应优先养护，「7 月 16 日新增病害数」和「7 月 15 日 08:00 之前近 3 天病害数」中，哪个可以作为特征，为什么喵**

**参考答案方向：**只能使用截至 7 月 15 日 08:00 已经产生并可获得的近 3 天病害数喵。7 月 16 日的数据属于未来信息，用它训练会形成特征泄漏，使离线效果虚高却无法真实部署喵。

**问题二：道路等级有高速公路、国道和县道，为什么通常不能简单编码为 `1、2、3`，而病害严重度的轻度、中度、重度却可以考虑这样编码喵**

**参考答案方向：**道路等级虽然有管理分类，但数字间距并不天然表示模型应采用的风险大小关系，one-hot 更能避免错误顺序假设喵。轻度、中度、重度具有明确的严重程度顺序，因此可以在业务定义清楚时使用序数编码喵。

## Quick Summary

- 特征是对路段、时间窗口或病害事件的可计算描述，特征工程负责把干净的原始数据转成适合任务的输入喵。

- 常见手段包括数值标准化、归一化与分箱，类别 one-hot 与序数编码，时间提取、窗口聚合、交叉构造和缺失标记喵。

- 数据挖掘是一组任务族，包括分类、回归、聚类、关联规则和异常检测，每类任务回答的问题不同喵。

- 特征质量决定挖掘效果的上限，错误、混乱或无关的输入会导致「garbage in, garbage out」喵。

- 所有特征都必须符合真实决策时点的信息边界，避免用未来数据或事后结果造成特征泄漏喵。

## Next Steps

🔍 **探索性数据分析**喵

探索性数据分析（Exploratory Data Analysis，EDA）是在正式建模前，通过统计摘要和图表了解数据分布、异常、趋势与变量关系的过程喵。这个概念尚未学习，补上它能帮助你判断应该构造哪些特征，而不是凭感觉加工数据喵。

🔍 **预测建模与时序分析**喵

进一步学习如何使用历史交通量、病害和养护记录预测未来状态，并理解时间顺序、趋势、季节性和滚动验证喵。这个方向能把本节的特征从透明评分规则推进到可评估的预测模型喵。

🔍 **窗口设计与特征泄漏防控**喵

深入比较近 3 天、近 7 天和近 30 天窗口如何表达短期冲击与长期恶化，并学习用特征可用时间检查泄漏喵。这个方向直接关系到公路养护模型能否从离线实验可靠地走向真实决策喵。
