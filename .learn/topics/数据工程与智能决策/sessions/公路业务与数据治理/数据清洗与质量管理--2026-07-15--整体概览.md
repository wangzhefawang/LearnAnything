## Positioning

前两节已经回答了两个基础问题：公路部门需要管理哪些业务，以及检测车、物联网传感器、视频 AI、交通量设备、ETC 门架、巡查 APP、卫星和无人机如何采集数据喵。采集阶段还建立了一个关键认识：多源数据需要统一到「路段 + 时间窗口」，并分成原始层、标准层和融合层喵。

本节沿着数据链路继续向下，解决「数据采回来以后能不能放心使用」的问题喵。数据清洗负责发现和处理脏数据，数据质量管理则把这种能力变成持续运行、可以监控和追责的工程闭环喵。

## Analogy

可以把公路多源数据想象成从不同产地运到中央厨房的一批食材喵。检测车数据像成箱蔬菜，路侧传感器数据像冷链肉类，视频 AI 识别结果像预加工食品，人工巡查记录则像手写的采购单喵。

食材进入厨房后不能立刻下锅，因为其中可能混有烂叶、重复送货、错误标签和超过保质期的产品喵。有些箱子写着千克，有些写着克，还有些只写了一个数字，这就像传感器单位不统一、桩号格式混乱或时间戳缺失喵。

数据清洗相当于验货、称重、分类和贴统一标签喵。工作人员会剔除确认变质的食材、合并重复订单、把重量换算成统一单位，并把暂时无法判断的食材放进隔离区，而不是随手扔掉喵。

数据质量管理则相当于厨房长期执行的食品安全制度喵。它不仅规定验货规则，还要持续监控合格率、生成质量报告、追查问题供应商，并保存原始进货记录，使每一次处理都能解释和重放喵。

## Core Mechanism

数据质量描述的是数据满足业务使用要求的程度喵。同一份数据可能适合做月度趋势统计，却不适合做分钟级交通预警，因此质量不能脱离具体业务目标来判断喵。

常见的数据质量维度包括以下六类喵。

| 质量维度 | 白话解释 | 公路数据示例 |
| --- | --- | --- |
| 完整性 | 应该存在的数据是否缺失喵 | 某路段每 5 分钟应有一条流量记录，但连续 30 分钟没有上报喵 |
| 准确性 | 数据与真实世界是否接近喵 | 检测车记录的病害位置与现场实际位置相差 200 米喵 |
| 一致性 | 不同字段、系统和时间段的表达是否相互协调喵 | 养护系统使用 `K12+300`，巡查 APP 却把同一位置写成 `12.3 km`喵 |
| 及时性 | 数据能否在业务需要的时间内到达喵 | 拥堵预警所需数据延迟 20 分钟，即使数值正确也失去价值喵 |
| 唯一性 | 同一个业务事实是否只记录一次喵 | 设备网络重连后把同一时间戳的数据重复上报了 3 次喵 |
| 有效性 | 数据是否符合格式、类型、字典和物理范围等规则喵 | 车速为负数、桩号写成无法解析的文字，或者车道编号超出实际车道数喵 |

完整性与准确性并不是一回事喵。一条记录的字段全部填写完整，只能说明它没有明显缺项，不能证明填写的内容是真实值喵。

公路数据中的脏数据通常具有明显的行业特征喵。

| 脏数据形态 | 可能原因 | 直接风险 |
| --- | --- | --- |
| GPS 漂移 | 高架遮挡、隧道失锁、设备定位误差或坐标系混用喵 | 病害可能被关联到错误路段或错误行车方向喵 |
| 传感器漂移或卡值 | 元件老化、校准失效、通信异常或程序故障喵 | 数值缓慢偏离真实值，或者长时间保持完全不变喵 |
| 重复上报 | 网络重试、消息重复消费或任务重复执行喵 | 流量、事件数量和统计指标被重复计算喵 |
| 时间戳缺失或时钟不准 | 设备未校时、时区混乱或程序漏填喵 | 多源数据无法正确对齐到同一时间窗口喵 |
| 单位不统一 | 不同厂商分别使用米、厘米、毫米或不同速度单位喵 | 合并计算后出现数量级错误喵 |
| 桩号格式混乱 | 系统标准不同或人工录入习惯不同喵 | `K12+300`、`12.3 km` 和 `K012+0300` 无法直接关联喵 |
| 人工填写不规范 | 自由文本、简称、错别字或必填约束不足喵 | 同一种病害被拆成多个类别，难以统计和检索喵 |
| AI 误识别 | 光照、遮挡、模型泛化不足或置信度阈值不合理喵 | 阴影可能被识别为裂缝，车辆可能被重复计数喵 |

清洗的第一类手段是规则校验，也就是把明确的业务要求写成机器可以执行的判断条件喵。例如，车速应位于合理范围内，时间戳必须能解析，桩号必须落在路线起终点之间，车道编号必须存在于道路基础设施台账中喵。

第二类手段是去重，其关键不是简单删除完全相同的行，而是定义「同一个业务事实」的唯一键喵。例如，传感器数据可以使用「设备编号 + 指标类型 + 采集时间」作为唯一键，巡查事件则可能需要「路线 + 桩号范围 + 事件类型 + 发现时间窗口」的组合键喵。

第三类手段是缺失值处理喵。可以根据业务语义选择删除、保留为空、使用前后值插值、使用同类数据的中位数填补，或者标记后交由人工确认，但任何填补值都必须附带「这是估算值」的标记喵。

第四类手段是异常值检测喵。物理量程校验适合识别绝不可能的数值，3σ 方法通过均值上下 3 个标准差寻找偏离点，四分位距（IQR）方法则把低于 `Q1 - 1.5 × IQR` 或高于 `Q3 + 1.5 × IQR` 的数据视为可疑值喵。

3σ 方法更适合近似正态分布的数据，而 IQR 对偏态分布和少量极端值通常更稳健喵。不过，统计异常不等于业务错误，例如暴雨期间突增的积水深度可能是真实事件，所以系统通常应先标记和隔离，再结合相邻设备、历史序列或现场记录判断喵。

第五类手段是标准化与字典映射喵。标准化负责统一时间格式、坐标系、桩号、单位和字段类型，字典映射则把「横裂」「横向裂缝」和厂商代码 `CRACK_T` 等表达映射为统一的标准分类喵。

清洗完成并不代表质量管理结束喵。成熟的数据质量管理是「质量规则 → 质量监控 → 质量报告 → 问题溯源 → 规则改进」的持续闭环喵。

质量规则定义什么算合格，质量监控持续计算缺失率、重复率、异常率和延迟等指标，质量报告展示问题分布及变化趋势，问题溯源则定位到具体设备、批次、接口或处理任务喵。修复原因后还应评估是否需要更新采集配置、数据字典或清洗规则，从源头减少同类问题喵。

原始数据必须保留且原则上不可覆盖，因为错误的清洗规则也可能误伤真实数据喵。标准层记录应保存来源记录标识、规则版本、处理时间、处理动作和异常原因，使清洗过程可追溯、可审计，并能在规则变化后重新播放原始数据喵。

## Code Example

下面的代码模拟了一批路面车辙深度记录，其中包含混乱桩号、重复上报、缺失值、单位不统一、时间戳缺失和超物理量程等问题喵。程序保留原始层副本，并通过确定性的规则管道生成标准层、处理血缘和质量报告喵。

```python
from copy import deepcopy
from datetime import datetime
from statistics import median
import json
import re


SOURCE_DATA = [
    {
        "device_id": "S01",
        "timestamp": "2026-07-15T08:00:00",
        "stake": "K12+300",
        "metric": "rut_depth",
        "value": 12,
        "unit": "mm",
    },
    {
        "device_id": "S01",
        "timestamp": "2026-07-15T08:00:00",
        "stake": "K12+300",
        "metric": "rut_depth",
        "value": 12,
        "unit": "mm",
    },
    {
        "device_id": "S02",
        "timestamp": "2026-07-15T08:00:00",
        "stake": "12.300km",
        "metric": "rut_depth",
        "value": 1.2,
        "unit": "cm",
    },
    {
        "device_id": "S03",
        "timestamp": "2026-07-15T08:00:00",
        "stake": "K12+500",
        "metric": "rut_depth",
        "value": None,
        "unit": "mm",
    },
    {
        "device_id": "S04",
        "timestamp": "2026-07-15T08:00:00",
        "stake": "K12+600",
        "metric": "rut_depth",

    },
    {

        "timestamp": "2026-07-15T08:00:00",
        "stake": "十二公里+700",
        "metric": "rut_depth",

        "unit": "mm",
    },
    {
        "device_id": "S06",
        "timestamp": "2026-07-15T08:00:00",
        "stake": "K12+800",
        "metric": "rut_depth",
        "value": 0.018,
        "unit": "m",
    },
    {
        "device_id": "S07",
        "timestamp": None,
        "stake": "K12+900",
        "metric": "rut_depth",
        "value": 15,
        "unit": "mm",
    },
]


def normalize_stake(raw_stake):
    text = str(raw_stake).strip().upper()

    match = re.fullmatch(r"K\s*(\d+)\s*\+\s*(\d+(?:\.\d+)?)", text)
    if match:
        total_meters = int(round(int(match.group(1)) * 1000 + float(match.group(2))))
    else:
        match = re.fullmatch(r"(\d+(?:\.\d+)?)\s*KM", text)
        if not match:
            raise ValueError("无法解析的桩号")
        total_meters = int(round(float(match.group(1)) * 1000))

    kilometers, meters = divmod(total_meters, 1000)
    return f"K{kilometers:03d}+{meters:03d}"


def normalize_depth(value, unit):
    if value is None or value == "":
        return None

    unit_factors = {"mm": 1.0, "cm": 10.0, "m": 1000.0}
    normalized_unit = str(unit).strip().lower()
    if normalized_unit not in unit_factors:
        raise ValueError("不支持的车辙深度单位")

    return float(value) * unit_factors[normalized_unit]


def run_cleaning_pipeline(raw_records):
    report = {
        "input_records": len(raw_records),
        "duplicate_records": 0,
        "invalid_stake_records": 0,
        "invalid_timestamp_records": 0,
        "invalid_unit_records": 0,
        "missing_values": 0,
        "imputed_values": 0,
        "out_of_range_records": 0,
        "unit_conversions": 0,
        "rejected_records": 0,
        "output_records": 0,
    }

    clean_layer = []
    lineage = []
    seen_keys = set()

    for source_index, raw in enumerate(raw_records, start=1):
        operations = []
        unique_key = (
            raw.get("device_id"),
            raw.get("metric"),
            raw.get("timestamp"),
        )

        # 组合业务键用于识别重复上报喵
        if unique_key in seen_keys:
            report["duplicate_records"] += 1
            report["rejected_records"] += 1
            lineage.append({
                "source_index": source_index,
                "status": "rejected",
                "reason": "重复业务键",
                "operations": operations,
            })
            continue
        seen_keys.add(unique_key)

        # 时间戳必须存在，并且能够按 ISO 8601 格式解析喵
        try:
            timestamp = raw.get("timestamp")
            if not timestamp:
                raise ValueError("时间戳缺失")
            datetime.fromisoformat(timestamp)
        except (TypeError, ValueError):
            report["invalid_timestamp_records"] += 1
            report["rejected_records"] += 1
            lineage.append({
                "source_index": source_index,
                "status": "rejected",
                "reason": "时间戳缺失或格式错误",
                "operations": operations,
            })
            continue

        # 不同桩号写法统一转换为标准的 Kxxx+xxx 格式喵
        try:
            normalized_stake = normalize_stake(raw.get("stake"))
            if normalized_stake != raw.get("stake"):
                operations.append(
                    f"桩号标准化：{raw.get('stake')} -> {normalized_stake}"
                )
        except ValueError:
            report["invalid_stake_records"] += 1
            report["rejected_records"] += 1
            lineage.append({
                "source_index": source_index,
                "status": "rejected",
                "reason": "桩号无法解析",
                "operations": operations,
            })
            continue

        # 所有车辙深度统一换算成毫米喵
        try:
            value_mm = normalize_depth(raw.get("value"), raw.get("unit"))
            if value_mm is not None and str(raw.get("unit")).lower() != "mm":
                report["unit_conversions"] += 1
                operations.append(
                    f"单位换算：{raw.get('value')} {raw.get('unit')} -> {value_mm} mm"
                )
        except (TypeError, ValueError):
            report["invalid_unit_records"] += 1
            report["rejected_records"] += 1
            lineage.append({
                "source_index": source_index,
                "status": "rejected",
                "reason": "数值或单位无法解析",
                "operations": operations,
            })
            continue

        # 示例规则规定车辙深度的物理有效范围为 0 至 100 mm 喵
        if value_mm is not None and not 0 <= value_mm <= 100:
            report["out_of_range_records"] += 1
            report["rejected_records"] += 1
            lineage.append({
                "source_index": source_index,
                "status": "rejected",
                "reason": "超出物理量程",
                "operations": operations,
            })
            continue

        if value_mm is None:
            report["missing_values"] += 1

        clean_layer.append({
            "source_index": source_index,
            "device_id": raw["device_id"],
            "timestamp": timestamp,
            "stake": normalized_stake,
            "metric": raw["metric"],
            "value_mm": value_mm,
            "imputed": False,
            "operations": operations,
        })

    # 使用已通过规则校验的数据中位数填补缺失值，并明确记录填补标记喵
    reference_values = [
        record["value_mm"]
        for record in clean_layer
        if record["value_mm"] is not None
    ]

    if reference_values:
        fill_value = median(reference_values)
        for record in clean_layer:
            if record["value_mm"] is None:
                record["value_mm"] = fill_value
                record["imputed"] = True
                record["operations"].append(f"缺失值填补：中位数 {fill_value} mm")
                report["imputed_values"] += 1

    # 标准层记录通过 source_index 回指原始层，并保存每一步处理动作喵
    for record in clean_layer:
        lineage.append({
            "source_index": record["source_index"],
            "status": "accepted",
            "reason": None,
            "operations": record["operations"],
        })

    report["output_records"] = len(clean_layer)
    report["acceptance_rate"] = round(
        len(clean_layer) / len(raw_records), 4
    ) if raw_records else 0.0

    return clean_layer, lineage, report


# 原始层使用深拷贝保存，清洗管道不会直接修改原始采集数据喵
raw_layer = deepcopy(SOURCE_DATA)
raw_snapshot = deepcopy(raw_layer)

clean_layer, lineage, quality_report = run_cleaning_pipeline(raw_layer)

# 该断言用于验证原始层在清洗前后保持不变喵
assert raw_layer == raw_snapshot

print("清洗后的标准层记录如下喵")
print(json.dumps(clean_layer, ensure_ascii=False, indent=2))

print("\n数据质量报告如下喵")
print(json.dumps(quality_report, ensure_ascii=False, indent=2))

print("\n处理血缘记录如下喵")
print(json.dumps(lineage, ensure_ascii=False, indent=2))
```

`SOURCE_DATA` 模拟原始采集层，故意加入了重复记录、混合桩号、厘米和米单位、缺失值、异常大值以及缺失时间戳喵。程序先用深拷贝保存原始层，并在最后通过断言确认原始记录没有被清洗过程覆盖喵。

`normalize_stake` 把 `K12+300` 和 `12.300km` 等写法统一为 `K012+300`喵。`normalize_depth` 则根据单位换算系数，把厘米和米统一转换为毫米，使后续规则使用同一语义基准喵。

主清洗管道依次执行唯一键去重、时间戳校验、桩号标准化、单位换算和物理量程校验喵。无法可靠修复的记录进入拒绝结果并写明原因，不会悄悄混入标准层喵。

对于缺失的车辙深度，示例使用已通过校验数据的中位数填补喵。中位数不容易被少量极端值拉偏，但程序仍通过 `imputed` 和 `operations` 字段说明该值并非设备实测值喵。

`quality_report` 汇总输入量、重复量、异常量、填补量、拒绝量和接收率喵。`lineage` 使用 `source_index` 连接原始层和标准层，并记录每条数据经历的操作，因此同一套函数可以在规则更新后重新运行喵。

## Common Misconceptions

**误区一：发现异常值就应该立即删除喵**

异常值可能来自设备故障，也可能代表事故、暴雨、拥堵或路面突然恶化等真实事件喵。更稳妥的做法是先标记或隔离，再结合物理规则、相邻设备、历史序列和人工记录判断喵。

**误区二：缺失值统一用平均数填补就可以喵**

平均数会掩盖不同路段、时段和设备之间的差异，还可能制造并不存在的观测结果喵。应先分析缺失原因和业务用途，再选择保留为空、删除、插值、中位数填补或人工补录，并保存填补标记喵。

**误区三：通过格式和量程校验的数据就是准确数据喵**

规则校验只能证明数据符合已定义的形式要求，不能证明它与真实世界完全一致喵。一个传感器可能始终输出量程内的数值，却因为长期漂移而持续偏离真实值喵。

**误区四：数据清洗是模型训练前执行一次的临时工作喵**

数据源、设备状态、接口版本和业务字典都会持续变化，一次性脚本无法长期保证质量喵。数据清洗应进入正式管道，并与质量监控、报告、告警、溯源和规则版本管理组成闭环喵。

## Socratic Check

**问题一：某交通量设备连续 2 小时每分钟都上报完全相同的数值，而且数值仍在合法量程内，你会把它归为哪类质量问题，又会怎样确认喵**

**参考答案方向：**它首先表现为疑似传感器卡值，涉及准确性，而不是简单的有效性问题喵。可以检查数值变化率、连续相同值时长、相邻设备趋势、设备心跳和现场交通视频，确认后再标记、隔离或补算喵。

**问题二：两条记录的桩号分别为 `K12+300` 和 `12.300km`，时间与指标也相同，为什么不能在标准化桩号之前直接按整行去重喵**

**参考答案方向：**原始字符串不同并不代表业务事实不同，直接按整行去重会漏掉语义重复记录喵。应先统一桩号、时间、单位和字典，再根据业务唯一键判断它们是否代表同一次观测喵。

## Quick Summary

- 数据质量需要从完整性、准确性、一致性、及时性、唯一性和有效性等维度综合判断喵。

- 公路脏数据常见于 GPS、传感器、时间戳、单位、桩号、人工填写和 AI 识别等环节喵。

- 清洗手段包括规则校验、去重、缺失值处理、异常值检测、标准化和字典映射喵。

- 统计异常只是需要调查的信号，不能在缺少业务证据时直接等同于错误数据喵。

- 质量管理是持续闭环，必须保留原始数据，并让规则版本、处理动作和结果可追溯、可重放喵。
