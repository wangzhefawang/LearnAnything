## 定位

「公路交通业务与养护场景」位于「公路业务与数据治理」子域的起点，它先回答公路由谁管理、如何运行、为什么养护以及业务人员怎样作出决策，随后才能准确理解公路多源数据采集、数据清洗与质量管理、ETL（Extract、Transform、Load，抽取、转换、加载）管道、分析挖掘和智能决策中的指标含义与数据用途喵。

## 类比

可以把一条公路想象成一个需要终身管理的人，而公路建设、运营和养护就是这个人的出生、生活、体检、治疗与健康管理过程喵。

公路建成通车相当于一个人完成出生登记，设计图纸、施工记录和交工验收数据构成它的「初始健康档案」喵。

日常巡查类似每天观察体温、精神状态和外观，巡查人员会记录路面是否出现坑槽、护栏是否损坏、边沟是否堵塞以及桥梁是否存在异常喵。

检测车进行的路面检测类似定期体检，通过激光、图像、惯性传感器等设备测量平整度、车辙、裂缝和抗滑性能，并将肉眼难以连续判断的问题转化为可比较的指标喵。

预防性养护类似体检发现血脂升高后调整饮食和运动，此时道路结构尚未严重损坏，但需要通过灌缝、雾封层、微表处或薄层罩面延缓性能衰减喵。

坑槽修补和局部结构修复类似门诊治疗，主要解决已经发生且范围有限的问题喵。

大中修则类似住院手术，通常意味着路面性能明显下降、结构承载能力不足或病害范围较大，需要铣刨重铺、补强基层甚至重建部分结构喵。

桥梁和隧道还像人体中的心脏与重要器官，一处严重缺陷即使占整条路线的比例很小，也可能造成重大安全风险，因此不能仅依靠全路线平均分判断是否安全喵。

数据工程师在这个类比中不是只负责保管体检报告的档案员，而是要把来自不同医院、设备和时间的记录统一起来，识别错误数据，追踪健康变化，并为医生的诊断和治疗决策提供可信依据喵。

## 核心机制

### 公路业务的完整生命周期

中国公路交通业务可以从建设、养护、运营和路网监测四个相互连接的环节理解喵。

这四个环节并不是通车后依次结束的流水线，而是在公路服役期间持续并行、不断反馈的闭环喵。

1. **建设阶段：** 包括规划、勘察、设计、施工、质量监督、交工验收和竣工验收，主要形成路线、路基、路面、桥涵、隧道、交通安全设施和机电设施的基础档案喵。

2. **养护阶段：** 通过巡查、检测、评定、方案设计、工程实施和效果验收，使道路保持安全、畅通和规定的服务水平喵。

3. **运营阶段：** 包括收费、路产管理、清障救援、服务区管理、交通组织、事件处置和公众出行服务，高速公路的运营业务通常尤其复杂喵。

4. **路网监测阶段：** 汇聚交通流、气象、视频、事件、施工占道和基础设施状态数据，掌握路网运行态势并支持预警、调度与信息发布喵。

从数据闭环看，建设数据提供资产底账，巡查和检测数据描述当前状态，运营与交通数据说明设施承受的荷载和外部环境，养护工单记录采取了什么措施，后续复测数据则用于判断措施是否有效喵。

### 中国公路管理体制

交通运输部主要负责国家层面的行业政策、法规制度、技术标准、全国路网规划与行业监督指导喵。

省级交通运输主管部门通常负责本行政区域内的行业管理、规划衔接、资金政策、监督考核和重大项目协调喵。

公路局、公路事业发展中心或类似机构通常承担普通国省干线的具体行业管理、技术管理或养护管理职责，但机构名称和职能边界会因地区改革情况而不同喵。

市县级交通运输部门、公路机构和乡镇相关单位还会分别承担地方道路、农村公路或辖区设施的管理职责喵。

高速公路通常由高速公路集团、项目公司或其他运营单位负责具体运营和养护，并接受交通运输主管部门及相关监管机构监督喵。

实际项目中不能仅根据机构名称推断数据责任，而应先确认谁是资产所有者、谁负责日常管理、谁实施检测、谁下达养护任务、谁验收工程以及谁对数据质量负责喵。

### 高速公路与普通国省干线的区别

| 对比维度 | 高速公路 | 普通国省干线 |
| --- | --- | --- |
| 通行方式 | 封闭或控制出入，通常设互通和收费设施 | 多为开放式道路，与沿线道路、村镇和单位出入口联系紧密 |
| 交通特征 | 速度高、连续通行、重载车辆比例可能较高 | 客货混行、非机动车和行人干扰较多，路段环境差异明显 |
| 管理主体 | 常由高速公路集团或项目运营公司负责 | 常由各级交通主管部门、公路机构或养护单位负责 |
| 设施构成 | 收费、监控、通信、隧道机电和服务区等系统较完整 | 平交口、穿村镇路段、路侧设施和沿线环境问题更突出 |
| 养护组织 | 作业窗口短，对封道、导改和安全布控要求高 | 需要协调沿线生产生活、平交交通和地方施工影响 |
| 数据特点 | 门架、收费、视频、事件和机电设备数据较丰富 | 数据来源更分散，路线里程、行政区划和养护责任边界更复杂 |

高速公路与普通国省干线可以使用相同类型的技术状况指标，但它们的风险容忍度、服务水平要求、交通组织成本、数据密度和养护优先级并不相同喵。

例如，同样大小的坑槽出现在高速公路主车道和低交通量普通公路上，其安全风险、处置时限和封闭施工成本会有明显差异喵。

### 养护业务的基本闭环

一个典型养护业务闭环可以概括为「发现问题—确认问题—评定状态—制定方案—安排资金—组织施工—验收复测—持续跟踪」喵。

问题可能来自人工巡查、公众上报、视频识别、检测车、桥隧检查、结构传感器或历史趋势分析喵。

发现异常后，业务人员需要确认病害类型、位置、范围、严重程度、发展速度和安全影响，并区分数据异常、表面病害与结构性问题喵。

随后，管理人员综合技术状况、交通量、重载比例、道路等级、事故风险、剩余寿命、施工条件和预算约束，确定是否养护、何时养护以及采用什么措施喵。

工程完成后还要记录材料、工程量、费用、施工时间、交通组织和验收结果，并通过后续检测评估养护效果喵。

因此，养护不是简单的「分数低就修」，而是一个同时考虑技术、安全、经济和资源约束的多目标决策过程喵。

### 日常养护、预防性养护、专项养护与大中修

| 类型 | 主要目标 | 典型触发条件 | 常见措施 |
| --- | --- | --- | --- |
| 日常养护 | 保持道路整洁、完整和基本通行能力 | 高频巡查发现的小范围、轻微或突发问题 | 清扫保洁、疏通排水、灌缝、坑槽小修、护栏修复、标志扶正 |
| 预防性养护 | 在结构基本良好时延缓性能衰减 | 早期病害出现，整体性能尚可，预计预防措施具有经济性 | 雾封层、裂缝封闭、微表处、稀浆封层、薄层罩面 |
| 专项养护 | 集中解决某类设施缺陷、风险或功能短板 | 某类病害突出、专项检查发现隐患或政策任务要求 | 排水整治、边坡加固、桥梁加固、隧道渗漏治理、交安设施提升 |
| 大中修 | 恢复路面整体功能或结构承载能力 | 病害范围广、性能明显下降、结构强度不足或剩余寿命过短 | 大面积铣刨重铺、基层补强、结构层重建、系统性修复 |

日常养护通常具有高频、小额、快速处置的特点，重点是防止小问题演变为安全事件或大范围损坏喵。

预防性养护强调「在恰当的道路上、恰当的时间、采用恰当的措施」，如果结构已经严重损坏，再做表面封层通常只能暂时遮盖问题喵。

专项养护更多是按问题类型或设施对象组织项目，它与大中修不是完全互斥的分类轴，某个桥梁专项加固工程也可能达到较大的工程规模喵。

大中修是传统养护管理中按工程性质和规模形成的常用说法，当前项目还可能使用预防养护、修复养护、专项养护和应急养护等分类，因此数据建模时必须保留原始业务分类及其适用标准喵。

### 养护措施的决策逻辑

养护决策首先要区分功能性损坏和结构性损坏喵。

功能性损坏主要影响平整度、抗滑、噪声或行驶舒适性，道路结构可能仍具备足够承载能力，通常可以考虑预防性养护或表层修复喵。

结构性损坏意味着面层、基层、路基或桥梁构件的承载能力已经不足，只处理表面往往会使病害很快复发，因此需要进一步检测并考虑补强或重建喵。

业务人员通常会综合以下因素确定养护优先级喵。

- **技术状况：** 当前得分、病害类型、严重程度、影响范围和劣化速度喵。

- **安全风险：** 是否可能引起车辆失控、构造物失效、落物、积水或交通中断喵。

- **交通重要性：** 道路等级、交通量、重载比例、应急通道属性和绕行条件喵。

- **全寿命成本：** 当前投入能否减少未来更昂贵的修复费用喵。

- **施工条件：** 气候、材料、工期、作业窗口、交通导改和施工安全条件喵。

- **资源约束：** 年度预算、人员设备、项目储备和资金使用规则喵。

对于数据工程师，这些因素说明养护标签不是天然客观的单一答案喵。

历史上某一路段没有实施养护，既可能表示路况良好，也可能表示预算不足、审批尚未完成或施工窗口不合适喵。

### 常见路面病害

裂缝是最常见的病害类型之一，包括横向裂缝、纵向裂缝、块状裂缝和龟裂等形态喵。

单条轻微裂缝可能来自温度收缩，而密集龟裂往往提示疲劳破坏或结构承载问题，因此「裂缝数量」不能替代裂缝形态、面积和严重程度喵。

车辙是轮迹带形成的纵向凹槽，可能由沥青混合料高温稳定性不足、压实问题、基层变形或长期重载交通造成喵。

车辙不仅影响舒适性，还可能导致雨天积水和车辆横向稳定性下降喵。

坑槽是路面材料脱落形成的局部凹坑，通常需要快速处置，因为其对行车安全和病害扩展都有直接影响喵。

沉陷表现为局部路面下沉，可能与路基压实不足、地基变形、排水失效、地下空洞或结构层破坏有关喵。

其他常见病害还包括松散、剥落、拥包、泛油、波浪、错台和修补损坏等喵。

同一种表面现象可能存在不同成因，因此图像识别模型可以帮助发现病害，却不能在缺少结构、气象、交通和历史维修信息时自动完成全部工程诊断喵。

### 桥梁与隧道养护

桥梁养护对象包括桥面系、上部结构、下部结构、支座、伸缩缝、排水设施和附属设施喵。

常见问题包括混凝土裂缝与剥落、钢筋锈蚀、支座异常、伸缩缝损坏、桥面铺装破损和基础冲刷喵。

桥梁检查通常具有经常检查、定期检查和特殊检查等不同层次，严重缺陷需要进一步检测、限载、封闭或加固处置喵。

隧道养护既涉及衬砌裂缝、渗漏水、变形和路面病害，也涉及照明、通风、消防、供配电、通信和监控等机电系统喵。

桥隧数据具有明显的对象层级，例如「桥梁—桥跨—构件—测点」或「隧道—区段—设备—部件」喵。

如果数据仓库只保留路线和里程而不保留构造物及构件标识，就难以追踪缺陷演化、维修历史和传感器异常喵。

桥隧安全还具有明显的短板效应，一处关键构件的严重缺陷不能被大量正常构件的平均值掩盖喵。

### JTG 5210-2018 技术状况评定体系

JTG 5210-2018《公路技术状况评定标准》用于把巡查和检测结果转化为统一的技术状况指标，使不同路线、路段和时期能够进行比较喵。

MQI（Maintenance Quality Index，公路技术状况指数）是反映公路总体技术状况的综合指标，通常取值为 0 至 100，数值越大表示技术状况越好喵。

MQI 由多个分项指标按权重综合形成，其基本思想可以表示为以下公式喵。

```
MQI = w_PQI × PQI + w_SCI × SCI + w_BCI × BCI + w_TCI × TCI
```

其中各权重之和为 1，具体权重需要按照标准规定的公路等级和评定条件选取喵。

对于高速公路和一级公路，常见的综合权重为 PQI、SCI、BCI、TCI 分别占 0.70、0.08、0.12 和 0.10；其他等级公路常见权重分别为 0.60、0.10、0.15 和 0.15，正式项目必须以适用标准条文和项目评定口径为准喵。

| 指标 | 英文全称 | 中文含义 | 主要数据来源 |
| --- | --- | --- | --- |
| MQI | Maintenance Quality Index | 公路技术状况指数 | 各分项指标综合计算 |
| PQI | Pavement Quality Index | 路面使用性能指数 | 路面检测、病害调查 |
| SCI | Subgrade Condition Index | 路基技术状况指数 | 路基、边坡、排水等检查 |
| BCI | Bridge, Tunnel and Culvert Condition Index | 桥隧构造物技术状况指数 | 桥梁、隧道、涵洞检查 |
| TCI | Traffic-facility Condition Index | 沿线设施技术状况指数 | 标志、标线、护栏等检查 |

PQI 不是一个传感器直接测出的数值，而是由路面损坏、平整度、车辙、抗滑和结构强度等评价结果组成的指标体系喵。

| 指标 | 英文全称 | 中文含义 | 主要反映的问题 |
| --- | --- | --- | --- |
| PCI | Pavement Surface Condition Index | 路面损坏状况指数 | 裂缝、坑槽、松散等表面病害 |
| RQI | Riding Quality Index | 路面行驶质量指数 | 平整度与行驶舒适性 |
| RDI | Rutting Depth Index | 路面车辙深度指数 | 沥青路面车辙状况 |
| SRI | Skidding Resistance Index | 路面抗滑性能指数 | 路面抗滑能力 |
| PSSI | Pavement Structural Strength Index | 路面结构强度指数 | 路面结构承载能力 |

不同路面类型、公路等级和检测条件对应的指标组合及权重可能不同，不能把某一种沥青路面的计算公式直接套用于所有道路喵。

在常见的沥青路面评定思路中，PQI 可以表示为多个适用分项指标的加权和喵。

```
PQI = Σ (w_i × I_i)    其中 i = 1..n
```

这里的 I_i 可以包括 PCI、RQI、RDI 和 SRI 等适用指标，权重总和应为 1 喵。

PSSI 通常用于补充判断结构承载能力，其检测频率、适用范围和在综合评定中的使用方式需要按标准及专项检测要求确定，不能因为表面 PQI 较高就忽略较低的结构强度喵。

评定等级通常按照分数划分为优、良、中、次、差五级喵。

| 等级 | 分数范围 |
| --- | --- |
| 优 | 90 ≤ 分数 ≤ 100 |
| 良 | 80 ≤ 分数 < 90 |
| 中 | 70 ≤ 分数 < 80 |
| 次 | 60 ≤ 分数 < 70 |
| 差 | 分数 < 60 |

综合分数适合路线排序和总体评价，但工程决策还必须检查分项指标、原始病害和重大安全缺陷喵。

例如，PQI 为良并不表示每一个分项都良好，较高的 PCI 可能掩盖较低的 RDI，导致总体分数看起来尚可但车辙风险已经需要处理喵。

同样，桥隧构造物中的严重缺陷具有安全控制属性，不能只通过提高其他分项得分来抵消喵。

### 业务场景产生的数据

| 数据类别 | 典型字段 | 主要业务用途 | 常见质量问题 |
| --- | --- | --- | --- |
| 检测车数据 | 路线、方向、车道、桩号、时间、IRI、车辙深度、病害图像 | 技术状况评定、衰减分析、养护候选路段识别 | 定位漂移、漏测、重复测量、设备标定差异 |
| 巡查记录 | 设施、位置、病害类型、严重程度、照片、上报人、时间 | 日常处置、安全隐患闭环 | 描述不统一、照片与位置不匹配、重复上报 |
| 养护工单 | 问题来源、处置措施、工程量、材料、班组、费用、状态 | 任务调度、成本核算、绩效评价 | 状态回填滞后、措施编码混乱、缺少验收结果 |
| 交通调查数据 | 断面、车型、流量、速度、轴载、方向、时间 | 交通需求、荷载分析、拥堵与劣化关系分析 | 设备离线、车型误判、异常零值、时间不同步 |
| 结构监测数据 | 构件、测点、应变、位移、振动、温度、采样频率 | 桥隧预警、趋势分析、异常诊断 | 漂移、噪声、断点、测点更换后口径变化 |
| 路网运行数据 | 视频、事件、气象、施工占道、救援记录 | 路网监测、应急调度、风险预警 | 多源事件重复、时间延迟、空间范围不一致 |
| 基础资产数据 | 路线编码、桩号范围、桥隧编号、设施类型、建成年份 | 数据关联、资产盘点、全寿命管理 | 编码变更、里程调整、对象重复、历史版本缺失 |

这些数据具有时间、空间和资产对象三类关键维度喵。

时间维度用于区分检测时间、发现时间、派单时间、施工时间和验收时间，空间维度用于描述路线、上下行、车道和起止桩号，资产维度用于定位桥梁、隧道、涵洞、边坡或具体构件喵。

如果三类维度没有统一编码，同一处病害可能无法与交通量、历史维修和后续复测数据正确关联喵。

### 为什么数据工程师必须先懂业务

数据字段只有放入业务流程后才具有稳定含义喵。

例如，巡查记录中的「已完成」可能表示现场已经临时处置，也可能表示工单已经关闭，但不一定表示永久修复已经通过验收喵。

检测车记录的桩号不是普通坐标，它还受到路线改线、里程重排、上下行方向和断链的影响喵。

交通流量中的零值可能表示真实无车、设备离线、通信中断或数据尚未汇总，不能不加区分地填充为平均值喵。

在数据采集阶段，业务理解决定采什么、在哪里采、以什么频率采以及怎样记录设备和环境条件喵。

在数据清洗阶段，业务理解决定哪些值不可能、哪些异常必须保留、哪些记录可以合并以及如何处理里程和资产编码变化喵。

在建模阶段，业务理解决定预测目标、观察窗口和标签定义，并帮助识别预算审批、施工选择和检测频率造成的数据偏差喵。

在智能决策阶段，业务理解决定优化目标不能只有平均分，还要同时考虑安全风险、交通影响、预算、施工窗口、公平性和全寿命成本喵。

因此，懂业务不是要求数据工程师取代道路工程师，而是让数据工程师能够提出正确的问题、建立正确的数据关系，并避免生成技术上精确但业务上无效的结果喵。

## 代码示例

下面的 Python 程序演示如何把 PCI、RQI、RDI 和 SRI 加权计算为简化的 PQI，并结合 PSSI 给出评定等级与初步养护建议喵。

示例采用 0.35、0.40、0.15 和 0.10 的教学权重，体现 JTG 5210-2018 按适用分项指标进行加权聚合的思路，但不能替代正式评定计算喵。

```python
from dataclasses import dataclass


@dataclass(frozen=True)
class PavementIndicators:
    """所有指标均按 0～100 计分，分数越高表示状况越好。"""

    pci: float
    rqi: float
    rdi: float
    sri: float
    pssi: float


PQI_WEIGHTS = {
    "pci": 0.35,
    "rqi": 0.40,
    "rdi": 0.15,
    "sri": 0.10,
}


def validate(indicators: PavementIndicators) -> None:
    for name, value in vars(indicators).items():
        if not 0 <= value <= 100:
            raise ValueError(f"{name.upper()} 必须位于 0～100，当前值为 {value}")


def calculate_pqi(indicators: PavementIndicators) -> float:
    """按适用分项指标加权计算教学版 PQI。"""
    validate(indicators)

    pqi = sum(
        getattr(indicators, name) * weight
        for name, weight in PQI_WEIGHTS.items()
    )
    return round(pqi, 2)


def evaluate_grade(score: float) -> str:
    if score >= 90:
        return "优"
    if score >= 80:
        return "良"
    if score >= 70:
        return "中"
    if score >= 60:
        return "次"
    return "差"


def recommend_maintenance(
    indicators: PavementIndicators,
    pqi: float,
) -> list[str]:
    suggestions: list[str] = []

    # PSSI 不直接放入本示例的 PQI，而是作为结构性风险控制条件。
    if indicators.pssi < 60:
        suggestions.append("结构强度偏低：开展结构专项检测，论证补强或大中修")
    elif indicators.pssi < 70:
        suggestions.append("结构强度需关注：缩短复测周期并核查弯沉等原始数据")

    if indicators.pci < 70:
        suggestions.append("表面病害较多：复核裂缝、坑槽及修补损坏的范围和成因")

    if indicators.rdi < 70:
        suggestions.append("车辙状况较差：调查重载交通与结构层变形，考虑车辙专项处治")

    if indicators.sri < 70:
        suggestions.append("抗滑性能不足：优先评估雨天安全风险并安排抗滑处治")

    if suggestions:
        return suggestions

    if pqi >= 90:
        return ["保持日常养护和常规检测"]
    if pqi >= 80:
        return ["总体状况良好，可结合病害发展趋势安排预防性养护"]
    if pqi >= 70:
        return ["进入养护候选范围，开展现场复核和全寿命成本比较"]
    return ["优先开展详细调查，论证修复养护或大中修方案"]


def main() -> None:
    indicators = PavementIndicators(
        pci=82.0,
        rqi=88.0,
        rdi=64.0,
        sri=76.0,
        pssi=68.0,
    )

    pqi = calculate_pqi(indicators)
    grade = evaluate_grade(pqi)
    suggestions = recommend_maintenance(indicators, pqi)

    print(f"PQI：{pqi}")
    print(f"评定等级：{grade}")
    print("养护建议：")
    for suggestion in suggestions:
        print(f"- {suggestion}")


if __name__ == "__main__":
    main()
```

`PavementIndicators` 把一次路面评价使用的指标组织成结构化对象，并明确所有指标均采用 0 至 100 的高分优制喵。

`validate` 函数先检查输入范围，避免把负数、百分制以外的数值或设备原始量纲误当成评价指数喵。

`calculate_pqi` 使用加权和计算 PQI，其中 RQI 权重为 0.40、PCI 权重为 0.35、RDI 权重为 0.15、SRI 权重为 0.10，四项权重之和为 1 喵。

`evaluate_grade` 根据优、良、中、次、差的分界值输出等级喵。

`recommend_maintenance` 没有只依据 PQI 给出结论，而是先检查 PSSI、RDI 和 SRI 等关键分项喵。

示例中的 PSSI 没有直接参与 PQI 加权，而是作为结构性风险控制条件，这体现了「综合评价用于概览，关键分项用于诊断」的思路喵。

给定示例中，PQI 的计算结果可能仍处于良好区间，但 RDI 较低且 PSSI 需要关注，因此程序不会简单输出「无需养护」喵。

真实项目还需要根据公路等级、路面类型、适用指标、原始检测数据、交通荷载、病害成因和标准条文选择正式公式，并由具备相应专业能力的人员完成工程判断喵。

## 常见误区

1. **认为养护就是发现坑槽后修补喵。**
   养护还包括状态检测、预防性处治、结构修复、桥隧检查、风险控制、工程验收和效果评估等完整闭环喵。

2. **认为 MQI 或 PQI 越高就代表所有设施都安全喵。**
   综合分数可能掩盖较差的单项指标，而桥隧关键构件、抗滑不足或严重坑槽等问题还具有安全控制属性喵。

3. **认为所有低分路段都应该立即大修喵。**
   养护方式取决于病害成因、结构强度、交通风险、劣化趋势、预算和施工条件，低分既可能需要大修，也可能适合局部处治喵。

4. **认为检测设备输出可以直接进入模型喵。**
   设备数据还可能存在定位漂移、漏测、重复、标定差异和异常零值，必须结合路线、方向、车道、桩号、时间和设备状态进行治理喵。

5. **认为历史养护记录可以直接作为最佳决策标签喵。**
   历史决策受到预算、审批、材料、施工窗口和管理政策影响，模型如果忽略这些因素，可能只会复制过去的资源约束喵。

## 苏格拉底式检查

**问题 1：某高速公路路段的 PQI 为 86，属于良，但 RDI 只有 62，能否直接得出「暂时不需要养护」的结论喵？**

参考思路：不能，因为 PQI 是多个分项的加权结果，较高的 PCI 或 RQI 可能掩盖较低的 RDI，而较差的车辙状况可能带来积水和行车稳定性风险喵。

还需要查看车辙深度原始数据、车道分布、重载交通、发展趋势和现场情况，再决定采用监测、预防性处治还是专项修复喵。

**问题 2：为什么同一个坑槽不能只用经纬度和照片完成数据管理喵？**

参考思路：养护业务还需要知道坑槽属于哪条路线、哪个方向、哪条车道、哪个桩号区间、何时发现、是否重复上报、由谁处置以及处置后是否通过验收喵。

只有把空间位置、资产标识、业务状态和时间过程关联起来，才能形成可追踪的养护闭环，并用于成本分析、效果评估和模型训练喵。

## 快速总结

- 公路业务由建设、养护、运营和路网监测构成持续反馈的全生命周期闭环喵。

- 日常养护、预防性养护、专项养护和大中修的目标、触发条件及工程规模不同，选择措施前必须区分功能性问题与结构性问题喵。

- JTG 5210-2018 使用 MQI 描述总体技术状况，并通过 PQI、SCI、BCI 和 TCI 等分项指标组织评价喵。

- PQI、PCI、RQI、RDI、SRI 和 PSSI 分别描述不同维度，综合得分不能替代关键分项和原始病害诊断喵。

- 懂业务场景是正确采集、清洗、关联和建模公路数据的前提，也是把分析结果转化为可执行养护决策的基础喵。
