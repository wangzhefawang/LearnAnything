## Positioning

ROS 节点与话题通信是导航、机械臂、Gazebo 仿真、传感器感知和控制指令之间的主干连接方式，`/cmd_vel`、`/scan`、`/odom`、`/joint_states` 这些实习中常见的话题本质上都依赖同一套发布订阅机制喵。

## Analogy

如果之前把 ROS 话题理解成「邮政系统」，这次可以把它升级成「广播电台加订阅频道」喵。

发布者像电台，只负责按固定频道播出节目，例如 `/scan` 频道持续播激光雷达数据，`/cmd_vel` 频道持续播速度指令喵。

订阅者像收音机，只要调到同一个频道并且能听懂节目格式，就可以接收内容喵。

这个类比比普通邮政更接近 ROS 话题通信，因为一个频道可以同时有多个电台发布，也可以同时有多个收音机收听，双方不需要知道彼此的名字、代码位置或启动顺序喵。

`queue_size` 可以理解成收音机旁边的临时录音缓存，如果听众处理节目太慢，缓存满了之后旧节目或新节目就会被丢掉，系统不会无限堆积消息等你慢慢处理喵。

这也解释了为什么机器人控制里经常宁愿丢掉旧消息，也不要把旧速度、旧激光或旧里程计排队执行，因为机器人需要的是「现在的世界」，不是几秒前的世界喵。

## Core Mechanism

话题通信的核心是发布订阅（pub/sub）解耦，发布者只声明「我在某个话题上发布某种消息类型」，订阅者只声明「我订阅某个话题并用某个回调函数处理消息」，两者不直接写死对方是谁喵。

这种解耦让机器人系统可以拆成很多小节点，例如 Gazebo 发布 `/scan`，导航节点订阅 `/scan` 和 `/odom` 后发布 `/cmd_vel`，底盘驱动订阅 `/cmd_vel` 并控制轮子，机械臂驱动发布 `/joint_states` 供 RViz 和规划器读取喵。

话题天然支持多对多通信，一个 `/joint_states` 可以同时被 RViz、MoveIt 和日志记录节点订阅，一个 `/cmd_vel` 也可能被键盘遥控节点、导航节点或安全停止节点发布，但多发布者会带来控制权冲突，所以实际工程里要通过 mux、优先级或状态机管理谁有权发命令喵。

在 ROS1 中，`roscore` 里的 master 负责「牵线」，节点启动后会向 master 注册自己发布或订阅了哪些话题，master 把匹配到的发布者地址告诉订阅者，然后真正的消息通常通过节点之间的点对点 TCPROS 连接传输喵。

因此，ROS1 的消息不是经过 master 中转的，master 更像电台目录或电话号码簿，真正的音频流是在电台和收音机之间直接传输喵。

在 ROS2 中，没有 ROS1 那种中心化 master，节点通过 DDS 做自动发现，发布者和订阅者能在网络中找到彼此，并且通信行为还可以通过 QoS 控制，例如队列深度、可靠性、持久性和实时性取舍喵。

所以可以粗略理解为 ROS1 是「先查中央频道目录，再建立点对点连接」，ROS2 是「频道设备自己在网络里发现彼此，并带着更明确的通信服务质量规则」喵。

消息类型是话题能否匹配的第二个关键条件，因为同名话题必须使用一致的消息定义，订阅者才能正确反序列化字节流喵。

常见消息包里，`std_msgs` 放简单标量和字符串，例如 `String`、`Bool`、`Float64`，适合状态量或调试信息喵。

`geometry_msgs` 放几何和运动相关结构，例如 `Twist` 表示线速度与角速度，`Pose` 表示位置和姿态，`Transform` 表示坐标变换喵。

`sensor_msgs` 放传感器数据，例如 `LaserScan` 表示二维激光，`Image` 表示图像，`PointCloud2` 表示点云，`JointState` 表示关节位置、速度和力矩喵。

自定义 `.msg` 文件用于定义项目自己的消息结构，例如把检测目标的类别、置信度和三维坐标打包成一个消息喵。

在 ROS1 中，消息定义会计算出 md5 校验值，发布者和订阅者只有在消息类型及字段定义兼容时才能连接，字段名、字段顺序或嵌套类型变化都可能导致 md5 不一致喵。

这就是为什么改了 `.msg` 后必须重新编译工作空间，并确保发布端和订阅端使用的是同一版消息包，否则看起来话题名一样，实际却连不上或报类型不匹配喵。

`queue_size` 是发布者或订阅者侧的缓冲队列大小，它决定当生产速度和消费速度不匹配时最多积压多少条消息喵。

`queue_size=1` 常用于控制和传感器的实时近似场景，因为它倾向于保留最新消息，旧消息处理不完就不要了喵。

但 `queue_size=1` 也可能让短时突发消息更容易丢失，例如低频事件通知、一次性状态切换或日志类消息就不适合随便设成 1 喵。

很大的 `queue_size` 可以缓解短时拥塞，但如果回调长期处理不过来，它会制造延迟，把机器人带入「还在处理旧世界」的危险状态喵。

订阅回调是异步触发的，消息到达后并不是你在主循环里手动 `read`，而是 ROS 通信层在合适时机调用你注册的 callback 喵。

在 `rospy` 中，`rospy.spin()` 用来保持节点运行并处理回调，如果节点本身还有发布循环，就常见写法是 `while not rospy.is_shutdown()` 配合 `Rate.sleep()`，订阅回调在后台接收消息并更新共享状态喵。

频率和延迟观测是理解话题通信的第三个关键能力，因为机器人问题经常不是「代码有没有发」，而是「发得够不够快、有没有积压、有没有丢、订阅者实际收到的频率是多少」喵。

`rostopic hz /scan` 用来观察订阅端看到的消息频率，`rostopic echo /cmd_vel` 用来查看具体消息内容，`rostopic info /odom` 用来查看发布者、订阅者和消息类型，`rqt_graph` 用来把节点和话题画成图喵。

如果 `rostopic hz` 看到频率很低，不一定是发布端慢，也可能是网络拥塞、仿真卡顿、订阅端处理过慢、消息太大、时间戳异常、队列太小或 QoS/连接配置不匹配喵。

## Code Example

下面这个 `rospy` 示例模拟实习中常见的「订阅 `/scan` 激光数据，发布 `/cmd_vel` 速度指令」避障节点喵。

```python
#!/usr/bin/env python
# -*- coding: utf-8 -*-

import math

import rospy
from geometry_msgs.msg import Twist
from sensor_msgs.msg import LaserScan


class SimpleAvoider(object):
    def __init__(self):
        rospy.init_node("simple_scan_avoider")

        self.front_distance = float("inf")

        self.cmd_pub = rospy.Publisher(
            "/cmd_vel",
            Twist,
            queue_size=1
        )

        self.scan_sub = rospy.Subscriber(
            "/scan",
            LaserScan,
            self.on_scan,
            queue_size=1
        )

    def on_scan(self, msg):
        # 取激光正前方 ±30°（约 60° 扇区）内的有效距离
        front_ranges = []
        total = len(msg.ranges)

        if total == 0:
            self.front_distance = float("inf")
            return

        center = total // 2
        window = max(1, int(total * 30.0 / 360.0))

        for distance in msg.ranges[center - window:center + window + 1]:
            if math.isfinite(distance) and msg.range_min <= distance <= msg.range_max:
                front_ranges.append(distance)

        if front_ranges:
            self.front_distance = min(front_ranges)
        else:
            self.front_distance = float("inf")

    def run(self):
        rate = rospy.Rate(10)

        while not rospy.is_shutdown():
            cmd = Twist()

            if self.front_distance < 0.8:
                cmd.linear.x = 0.0
                cmd.angular.z = 0.6
                rospy.loginfo_throttle(1.0, "Obstacle ahead: %.2f m, turning", self.front_distance)
            else:
                cmd.linear.x = 0.2
                cmd.angular.z = 0.0
                rospy.loginfo_throttle(1.0, "Path clear: %.2f m, moving forward", self.front_distance)

            self.cmd_pub.publish(cmd)
            rate.sleep()


if __name__ == "__main__":
    try:
        node = SimpleAvoider()
        node.run()
    except rospy.ROSInterruptException:
        pass
```

第一段导入 `geometry_msgs/Twist` 和 `sensor_msgs/LaserScan`，这两个类型正好对应控制输出和激光输入喵。

`rospy.Publisher("/cmd_vel", Twist, queue_size=1)` 表示节点会向 `/cmd_vel` 频道发布底盘速度命令，并且只保留很短的发送缓存喵。

这里设 `queue_size=1` 是因为速度命令强时效，旧的速度命令不应该排队等待执行喵。

`rospy.Subscriber("/scan", LaserScan, self.on_scan, queue_size=1)` 表示节点订阅激光数据，每来一帧 `LaserScan` 就异步调用 `on_scan` 喵。

`on_scan` 并不直接控制机器人，而是只更新 `front_distance`，这种写法把「感知输入更新」和「控制输出循环」分开，结构更接近真实机器人节点喵。

`run()` 里的 10 Hz 循环持续发布速度，如果前方 0.8 m 内有障碍物就原地转向，否则低速前进喵。

这个例子里，发布循环和订阅回调是并行协作关系，主循环不需要主动等待 `/scan`，回调也不应该做太慢的计算，否则会导致消息积压或丢弃喵。

调试这类节点时，可以先运行下面这些命令观察系统是否真的连起来喵。

```bash
rostopic info /scan
rostopic hz /scan
rostopic echo -n 1 /scan
rostopic info /cmd_vel
rostopic echo /cmd_vel
rqt_graph
```

`rostopic info /scan` 重点看消息类型是不是 `sensor_msgs/LaserScan`，以及你的避障节点是否出现在订阅者列表里喵。

`rostopic hz /scan` 重点看频率是否稳定，例如仿真激光可能是 5 Hz、10 Hz 或 20 Hz，频率抖动很大时要怀疑 Gazebo 性能、仿真时间或传感器插件配置喵。

`rostopic echo /cmd_vel` 重点看速度命令是否符合预期，例如遇到障碍时 `linear.x` 是否变成 0，`angular.z` 是否开始转向喵。

`rqt_graph` 重点看图里是不是 Gazebo 或传感器节点发布 `/scan`，避障节点订阅 `/scan` 并发布 `/cmd_vel`，底盘或仿真控制器再订阅 `/cmd_vel` 喵。

如果发现丢消息或延迟，可以按顺序排查发布频率、消息大小、网络状态、回调耗时、`queue_size`、CPU 占用、仿真是否卡顿，以及同一话题是否有多个发布者互相覆盖喵。

## Common Misconceptions

第一个误区是以为 `publish()` 调用后消息就同步送达所有订阅者喵。

实际情况是，`publish()` 更像把消息交给通信层排队发送，订阅者什么时候收到、是否收到、是否来得及处理，还受到连接、队列、网络和回调速度影响喵。

第二个误区是以为 ROS1 的消息会经过 master 中转喵。

ROS1 master 只负责注册和发现，真正的数据通常在发布者和订阅者节点之间通过 TCPROS 点对点传输，所以 master 挂掉后已有连接可能还能短暂工作，但新节点发现和新连接会出问题喵。

第三个误区是以为 `queue_size` 越大越安全喵。

队列大确实能缓冲突发流量，但控制和感知场景里过大的队列会把旧消息变成延迟，机器人可能根据几秒前的 `/scan` 或 `/odom` 做现在的决策喵。

第四个误区是看到 `rostopic hz` 低就认定发布端有 bug 喵。

`rostopic hz` 测到的是当前观测者收到的频率，它会受订阅端机器负载、网络、消息序列化、仿真时间、话题重映射和统计窗口影响，所以要结合 `rostopic info`、发布端日志、CPU 占用和 `rqt_graph` 一起判断喵。

## Socratic Check

如果 Gazebo 里 `/scan` 是 10 Hz，但你的避障节点回调每次要做 300 ms 的点云或图像处理，那么 `queue_size=1` 和 `queue_size=100` 会分别带来什么现象喵。

如果键盘遥控节点和导航节点同时向 `/cmd_vel` 发布速度，底盘控制器会天然知道谁优先吗，还是需要额外的仲裁节点来管理控制权喵。

## Quick Summary

- 话题通信的价值不是「能发消息」这么简单，而是让传感器、导航、控制、仿真和可视化节点在弱耦合结构里协作喵。
- ROS1 由 master 负责注册和牵线，实际数据通常走节点间 TCPROS，ROS2 则依赖 DDS 自动发现和 QoS 管理通信行为喵。
- 消息类型是通信契约，常用包包括 `std_msgs`、`geometry_msgs` 和 `sensor_msgs`，自定义 `.msg` 在 ROS1 中还要注意 md5 一致性喵。
- `queue_size` 是实时性和完整性之间的取舍，控制类话题常保留最新消息，事件类或日志类话题则要避免过早丢弃喵。
- `rostopic hz`、`rostopic echo`、`rostopic info` 和 `rqt_graph` 是排查频率、内容、连接关系和丢消息问题的基本工具喵。
