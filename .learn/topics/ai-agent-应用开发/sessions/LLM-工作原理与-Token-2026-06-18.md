## 类比：一个超级会接话的"文字接龙大师"

想象有这么一个人，他这辈子读完了人类几乎所有的书、网页、对话。你跟他玩文字接龙——你说半句话，他来接下一个字。

但他接话有两个特点：

1. **他一次只猜一个字（词）**，猜完之后，把这个新字也当成"题目的一部分"，再去猜下一个字，循环往复。
2. **他不是死记硬背原文**，而是凭"语感概率"来接——他脑子里有个直觉："在这个上下文之后，接『天空』的可能性 70%，接『地面』的可能性 5%……"，然后挑一个接上。

这就是 LLM（大语言模型，Large Language Model）的本质：**它不是数据库，不是搜索引擎，它是一个"预测下一个词"的概率机器**。它说出的每一句话，都是一个字一个字"接龙"接出来的。

## 核心机制

### 1. Token —— 模型眼里的"文字积木"

模型并不像我们一样按"字"或"词"来看文本，它看的是 **Token（词元）**。

Token 是文本被切分后的最小单位，可能是一个词、半个词、一个汉字，甚至一个标点。例如：

- 英文 `"unbelievable"` 可能被切成 `un` + `believ` + `able` 三个 Token。
- 中文「你好世界」大致是 4 个 Token（中文常常一个字 ≈ 1～2 个 Token）。
- 经验法则：**1 个英文 Token ≈ 4 个字符 ≈ 0.75 个英文单词；1 个中文字 ≈ 1～2 个 Token**。

为什么要切成 Token 而不是直接用单词或字母？因为这样既能控制"词表"大小（不用记下世界上所有单词），又能处理没见过的新词（拆成认识的碎片）。

> **关键认知**：模型读不到"文字"，它只认识一串 **Token 编号**（比如 `天`=15922）。你发给它的所有内容，进去之前都会先被"分词器"（Tokenizer）切成 Token 编号，输出再被还原成文字。

为什么对 AI 应用工程师极其重要：

- **API 按 Token 计费**，不是按字数。
- **上下文窗口的容量是按 Token 算的**（比如"128K 上下文"= 12.8 万个 Token，不是 12.8 万个字）。
- 做 RAG、做上下文裁剪、算成本，全都要会"数 Token"。

### 2. 自回归生成 —— 一个字一个字"滚雪球"

LLM 的生成方式叫 **自回归（Autoregressive）**，就是上面文字接龙的正式名字：

```
输入：今天天气真
        ↓ 模型预测下一个 Token 的概率分布
        好 (65%) / 不错 (20%) / 糟糕 (8%) / ...
        ↓ 选中"好"
输入：今天天气真好
        ↓ 再预测
        ，(40%) / ！(30%) / 啊 (15%) / ...
        ↓ 选中"，"
输入：今天天气真好，
        ↓ ……不断循环，直到生成"结束符"
```

每生成一个新 Token，就把它拼回输入末尾，再预测下一个。所以模型的输出是**串行**的、一个一个蹦出来的——这也正是 ChatGPT 回答呈现"打字机效果"逐字冒出来的原因（这叫**流式输出**）。

### 3. 上下文窗口 + 采样参数 —— 模型的"视野"和"性格"

**上下文窗口（Context Window）**：模型一次能"看到"的 Token 总量上限，**包括你的输入 + 它的输出**。好比一张固定大小的桌子，你放的资料 + 它写的答案都得摆在这张桌子上，超出桌面的就掉下去（被遗忘）了。这就是为什么长对话到后面模型会"忘记"开头说过的话。

**采样参数**——决定模型"接龙时怎么挑字"，最常用的是 **温度（Temperature）**：

- **温度低（如 0.2）**：模型很"保守"，每次都挑概率最高的那个字 → 输出稳定、可复现，适合**事实问答、代码、数据抽取**。
- **温度高（如 1.0+）**：模型很"放飞"，会给低概率的字也一些机会 → 输出多样、有创意，适合**写故事、头脑风暴**。

一句话记住：**温度 = 创造力旋钮**。做 Agent 和 RAG 时通常调低（要准确），做创意写作时调高。

## 代码示例：亲手"看见" Token

用 OpenAI 的分词库 `tiktoken`，直观看到一句话是怎么被切成 Token 的：

```python
# pip install tiktoken
import tiktoken

# 加载 GPT-4 系列用的分词器
enc = tiktoken.encoding_for_model("gpt-4")

text = "Hello, 你好世界!"

# 编码：文字 → Token 编号列表
tokens = enc.encode(text)
print("Token 编号：", tokens)
print("Token 数量：", len(tokens))   # 这就是计费/占上下文的数量

# 解码：把每个 Token 编号单独还原，看看它到底切成了啥
for t in tokens:
    print(f"  {t:>6}  ->  {enc.decode([t])!r}")
```

可能的输出：

```
Token 编号： [9906, 11, 220, 57668, 53901, 99489, 0]
Token 数量： 7
    9906  ->  'Hello'
      11  ->  ','
     220  ->  ' '
   57668  ->  '你'
   53901  ->  '好'
   99489  ->  '世界'
       0  ->  '!'
```

**怎么读这段输出**：
- `"Hello"` 整个是 1 个 Token（常见英文词），但中文「你」「好」各自 1 个，「世界」又合成 1 个——可见**切分没有固定规则，取决于训练时的统计**。
- 总共 7 个 Token：这就是这次调用"输入侧"要付费、要占上下文的真实数量，**和直觉里的"11 个字符"对不上**。
- 学会数 Token，以后估算 API 成本、设计上下文裁剪策略时就有了"标尺"。

## 常见误区

1. **「LLM 是在数据库里查答案」** ❌ —— 不是。它不存原文，它是在"预测下一个最可能的 Token"，所以它会**一本正经地编造**（这就是"幻觉"的根源）。理解这点，才会明白为什么需要 RAG 给它"喂真实资料"。

2. **「1 个汉字 = 1 个 Token」** ❌ —— 不一定。不同模型的分词器不同，中文有时 1 字 1 Token，有时 1 字 2 Token，英文一个长单词反而可能是好几个 Token。**算成本必须用对应模型的分词器实测，不能拍脑袋**。

3. **「上下文窗口只算我发的内容」** ❌ —— 错。窗口 = 输入 + 输出 一起算。给的资料太长，留给模型回答的空间就被挤没了。

4. **「调高温度模型会更聪明」** ❌ —— 温度只影响"挑字的随机性"，不改变模型的知识和能力。高温只是更"敢说"，不是更"会说"。

## 苏格拉底式小思考

> 假设你在做一个客服 Agent，每次对话都要把"公司的 50 页产品手册"全部塞进 Prompt 里发给模型。基于今天讲的 **Token 计费** 和 **上下文窗口** 两点，这种做法会带来哪两个现实问题？

提示：一个跟"钱"有关（每次都把整本手册当输入 Token，成本爆炸），一个跟"桌子大小"有关（手册太长可能直接超出上下文窗口，或挤占回答空间）。而这两个问题，正是后面要学的 **RAG 知识库** 想解决的核心痛点——只检索手册里相关的几段塞进去，而不是全文。

## 快速总结

- LLM 本质是"预测下一个 Token"的概率机器，不是数据库；它的"幻觉"正源于此。
- Token 是模型处理文本的最小单位（≠ 字/词），API 计费与上下文窗口都按 Token 算。
- 生成是自回归的：一个 Token 接一个 Token 串行产出，这就是流式输出的由来。
- 上下文窗口 = 输入 + 输出 的 Token 总上限，是一张有限大小的"桌子"。
- 温度是"创造力旋钮"：低温稳定准确，高温多样有创意。
