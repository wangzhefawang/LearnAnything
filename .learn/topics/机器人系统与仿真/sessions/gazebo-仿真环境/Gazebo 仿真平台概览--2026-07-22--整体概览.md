## Positioning

Gazebo 位于机器人软件栈的虚拟验证层，它用可计算的世界、机器人和传感器替代一部分真实场地与硬件，让感知、导航和控制系统先在安全环境中运行喵。

它与 ROS 经常一起出现，但二者职责不同，Gazebo 负责生成物理过程和虚拟观测，ROS 负责把这些数据送到算法节点并把控制命令送回仿真器喵。

## Analogy

可以把 Gazebo 想象成“机器人驾校的模拟训练场 + 可重复的物理实验室”喵。

训练场里可以布置墙壁、坡道、桌椅和光照，机器人撞墙不会损坏昂贵硬件，失败后也能立刻把场景恢复到同一个起点喵。

物理实验室则负责按设定计算重力、摩擦、碰撞和关节运动，同时让虚拟激光雷达、相机与 IMU 像真机设备一样持续产生数据喵。

仿真并不是现实的完美复制品，而是一台参数可控的实验机器，价值来自安全、低成本、可复现和可自动化，而不是来自“看起来像真的”这一点喵。

## Core Mechanism

Gazebo 是面向机器人的三维物理仿真平台，它把物理引擎、传感器仿真和 3D 渲染组合在同一运行环境中喵。

物理引擎根据质量、惯性、关节约束、碰撞形状、摩擦和外力推进世界状态，传感器系统从这个状态生成激光、图像、深度、IMU 等观测，渲染系统则把场景显示出来喵。

机器人开发需要仿真，首先是因为真实设备具有安全与成本约束，错误速度指令可能撞坏底盘、机械臂或周围物体喵。

其次，仿真能够固定世界文件、初始位姿、参数和随机种子，使回归测试反复运行同一场景，这在真机和开放环境中很难做到喵。

仿真还可以并行执行大量测试、制造罕见故障并收集训练数据，从而缩短算法迭代周期喵。

不过最终目标通常是 sim-to-real，也就是让在仿真中验证或训练的系统迁移到真实机器人，因此模型误差、传感器噪声、延迟和执行器特性必须被认真对待喵。

Gazebo 名称背后有两条容易混淆的产品线喵。

Gazebo Classic 是早期广泛使用的系列，典型命令是 `gazebo`，ROS 集成通常依赖 `gazebo_ros`，大量 ROS1 教程和旧项目仍以 Gazebo 11 为基础喵。

新版 Gazebo 曾以 Ignition Gazebo 为名，后来统一回到 Gazebo 品牌，软件包与命令常使用 `gz` 前缀，仿真服务器的核心组件称为 `gz-sim` 喵。

新建 ROS2 项目通常优先选择与当前 ROS 发行版兼容的新版 Gazebo 和 `ros_gz`，维护旧 ROS1 工程或依赖 Classic 专用插件时才保留 Gazebo Classic 喵。

版本选择不能只看名字的新旧，还要核对 ROS 发行版、Gazebo 发行版、操作系统和桥接包的官方兼容组合，否则可能在插件 ABI 或消息桥接处遇到问题喵。

Gazebo 有自己的进程、消息和传输机制，并不是 ROS 的一个可视化窗口喵。

在 Classic 中，`gazebo_ros` 插件把传感器与控制接口接入 ROS，在新版 Gazebo 中，`ros_gz_bridge` 可以在 Gazebo Transport 消息与 ROS2 消息之间建立桥接喵。

桥接后，虚拟激光雷达可以表现为 ROS 的 `/scan`，相机可以表现为图像 Topic，控制命令可以从 `/cmd_vel` 进入仿真，仿真时钟则通过 `/clock` 提供给使用 `use_sim_time` 的节点喵。

一个 Gazebo 场景通常由 World（世界）和 Model（模型）组成，世界描述重力、光源、物理参数和其中的模型喵。

SDF（Simulation Description Format，仿真描述格式）是 Gazebo 原生的世界与模型描述格式，它可以表达多个模型、传感器、插件、碰撞参数和复杂环境喵。

机器人项目常用 URDF 描述机器人连杆与关节，再通过生成实体、转换或启动工具把 URDF 机器人放进 Gazebo 世界喵。

URDF 适合表达一棵机器人运动学树，而 SDF 对闭环结构、世界元素、传感器和仿真专用属性的表达更完整，因此两者常常协同出现喵。

Link（连杆）是具有位姿和物理属性的刚体，Joint（关节）约束两个连杆之间允许的相对运动喵。

Visual 决定屏幕上看到的外观，Collision 决定碰撞检测所用的几何形状，Inertial 则用质量、质心与惯性张量决定物体受力后的运动反应喵。

漂亮的网格不等于正确的碰撞体，缺失或错误的惯性参数甚至会让模型抖动、穿透、弹飞或导致求解器不稳定喵。

插件体系让 Gazebo 在基本刚体仿真之外连接具体机器人能力喵。

传感器插件可以生成激光雷达、RGB 相机、深度相机和 IMU 数据，控制插件可以实现差速驱动、关节控制或与 `ros2_control` 对接喵。

插件通常读取 SDF 或 URDF 中的配置，再在 Gazebo 内部更新周期中计算输出或施加力，因此更新率、话题名、坐标系和噪声参数都会直接影响上层算法喵。

仿真时间是 Gazebo 内部推进的逻辑时间，墙钟时间是真实世界流逝的时间，两者不一定相等喵。

实时因子（Real Time Factor，RTF）可以近似理解为“仿真时间增量 ÷ 墙钟时间增量”，RTF 为 1 表示仿真大致与现实同速，低于 1 表示仿真跑得更慢，高于 1 表示仿真跑得更快喵。

`max_step_size` 决定单次物理步长，`real_time_update_rate` 决定期望的物理更新频率，两者与场景复杂度、传感器负载和计算性能共同影响 RTF 喵。

更小的步长通常能提高高速接触或控制计算的稳定性，但也需要更多计算，不能简单理解为越小越好喵。

## Code Example

下面的 `minimal_world.sdf` 是一个自包含的最小世界，它包含物理参数、光源、地面和一个会受重力影响的立方体喵。

```xml
<?xml version="1.0"?>
<sdf version="1.9">
  <world name="minimal_world">
    <gravity>0 0 -9.8</gravity>

    <physics name="default_physics" type="ode">
      <max_step_size>0.001</max_step_size>
      <real_time_update_rate>1000</real_time_update_rate>
    </physics>

    <light name="sun" type="directional">
      <pose>0 0 10 0 0 0</pose>
      <diffuse>0.8 0.8 0.8 1</diffuse>
      <direction>-0.5 0.1 -0.9</direction>
    </light>

    <model name="ground">
      <static>true</static>
      <link name="ground_link">
        <collision name="ground_collision">
          <geometry>
            <box><size>20 20 0.1</size></box>
          </geometry>
        </collision>
        <visual name="ground_visual">
          <geometry>
            <box><size>20 20 0.1</size></box>
          </geometry>
        </visual>
      </link>
    </model>

    <model name="falling_box">
      <pose>0 0 1 0 0 0</pose>
      <link name="box_link">
        <inertial>
          <mass>1.0</mass>
          <inertia>
            <ixx>0.04167</ixx>
            <iyy>0.04167</iyy>
            <izz>0.04167</izz>
          </inertia>
        </inertial>
        <collision name="box_collision">
          <geometry>
            <box><size>0.5 0.5 0.5</size></box>
          </geometry>
        </collision>
        <visual name="box_visual">
          <geometry>
            <box><size>0.5 0.5 0.5</size></box>
          </geometry>
        </visual>
      </link>
    </model>
  </world>
</sdf>
```

`gravity` 指定世界重力，立方体初始位于地面上方 1 m，因此仿真开始后会下落并与地面碰撞喵。

物理步长是 0.001 s，期望更新率是 1000 Hz，两者的理想乘积对应 RTF 约为 1，但实际 RTF 仍取决于计算机能否及时完成每一步喵。

地面模型设置为 `static`，表示它不会被碰撞推动，立方体没有设置 `static`，因此会参与动力学计算喵。

地面和立方体都分别定义了 Collision 与 Visual，前者参与接触求解，后者用于显示喵。

立方体的质量为 1 kg，三个主轴惯量约为边长 0.5 m 的均匀实心立方体绕质心的惯量，这让物理引擎拥有完整的动力学信息喵。

在安装了新版 Gazebo 的 PowerShell 终端中，可以进入文件所在目录并启动世界喵。

```powershell
gz sim -r .\minimal_world.sdf
```

`gz sim` 启动新版 Gazebo，`-r` 表示加载完成后立即运行物理仿真，而不是保持暂停喵。

如果 ROS2 节点需要使用 Gazebo 的仿真时钟，可以在另一个已加载 ROS2 与 `ros_gz` 环境的 PowerShell 终端中建立单向桥接喵。

```powershell
ros2 run ros_gz_bridge parameter_bridge '/clock@rosgraph_msgs/msg/Clock[gz.msgs.Clock'
ros2 topic echo /clock
```

桥接参数中的 `[` 表示把 Gazebo 消息单向转换到 ROS2，左侧声明 ROS2 的 `Clock` 类型，右侧声明 Gazebo 的 `Clock` 类型喵。

桥接成功后，`ros2 topic echo /clock` 能看到仿真时间，导航、控制或记录节点再把 `use_sim_time` 设为 `true`，就会以仿真时钟而不是墙钟时间驱动定时与时间戳判断喵。

如果使用 Gazebo Classic，启动命令、插件名称和桥接方式会不同，不能把 `gz sim`、`ros_gz` 与 Classic 的 `gazebo_ros` 配置直接混用喵。

## Common Misconceptions

第一个误区是把 Gazebo 当作 RViz 的加强版，实际上 RViz 主要显示 ROS 数据，Gazebo 会主动推进物理世界并生成传感器数据喵。

第二个误区是认为仿真成功就等于真机成功，仿真中的摩擦、惯性、噪声、延迟和接触模型都可能与现实不同，sim-to-real 仍需要标定、随机化与真机验证喵。

第三个误区是认为 URDF 中模型看起来正确就足够，Visual 正常并不能证明 Collision、Inertial、关节限制和坐标系正确喵。

第四个误区是把 Gazebo Classic、Ignition Gazebo 和新版 Gazebo 当成可以任意替换的同一套命令，它们的包名、插件 API、传输系统和 ROS 集成方式存在代际差异喵。

第五个误区是认为 RTF 低只代表画面卡顿，实际上 RTF 会改变仿真推进速度，并可能暴露控制周期、传感器更新率和超时逻辑之间的不匹配喵。

第六个误区是认为把 `/clock` 桥接出来就自动启用了仿真时间，每个 ROS 节点仍需要正确设置 `use_sim_time` 才会读取该时钟喵。

## Socratic Check

问题 1：为什么立方体需要同时设置 Visual、Collision 和 Inertial，而不是只给一个三维网格喵？

答案：Visual 只决定外观，Collision 决定接触几何，Inertial 决定受力后的运动，三者分别服务渲染、碰撞和动力学，缺少任何一类都会让仿真信息不完整喵。

问题 2：如果 Gazebo 的 RTF 只有 0.5，但 ROS 节点都使用 `/clock`，从节点视角看 10 s 的仿真过程需要多少墙钟时间喵？

答案：RTF 为 0.5 表示仿真时间以墙钟时间一半的速度推进，因此 10 s 仿真时间大约需要 20 s 墙钟时间喵。

## Quick Summary

- Gazebo 把物理引擎、传感器仿真和 3D 渲染组合成可重复的机器人虚拟实验环境喵。
- Gazebo Classic 主要服务旧项目，新版 Gazebo 由 Ignition 演进而来并使用 `gz-sim` 与 `ros_gz` 生态，版本选择必须匹配 ROS 发行版喵。
- SDF 描述世界与仿真属性，URDF 常描述机器人结构，Link、Joint、Collision、Visual 与 Inertial 共同决定模型怎样显示和运动喵。
- 插件与 ROS 桥接把虚拟传感器、控制接口和 `/clock` 暴露给上层节点，而物理步长与 RTF 决定仿真时间怎样推进喵。

- **SDF 建模与惯性参数**：深入学习模型嵌套、关节限制、碰撞几何和惯性张量，能够解决模型抖动、穿透与运动不真实等基础问题喵。练习入口：`/learn:practice SDF 建模与惯性参数` 喵。
- **ros_gz 桥接与仿真时钟**：掌握消息方向、类型映射、`/clock` 和 `use_sim_time`，能让导航栈、rosbag2 与仿真器共享一致的时间和数据接口喵。练习入口：`/learn:practice ros_gz 桥接与仿真时钟` 喵。
- **Gazebo 传感器与控制插件**：通过激光雷达、相机、IMU、差速驱动和 `ros2_control` 插件，可以把静态模型变成可被真实算法驱动的虚拟机器人喵。练习入口：`/learn:practice Gazebo 传感器与控制插件` 喵。
- **Sim-to-Real 测试设计**：学习噪声注入、域随机化、延迟建模和真机对照指标，能够识别“只在仿真中有效”的脆弱假设喵。练习入口：`/learn:practice Sim-to-Real 测试设计` 喵。
