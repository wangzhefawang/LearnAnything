## Positioning
ROS 服务与动作通信是在昨天的 Topic 发布 / 订阅基础上，继续学习「一问一答」和「长任务可跟踪」这两类更有控制意图的通信方式喵。

## Analogy
可以把一个 ROS 系统想成一家餐厅，Topic、Service、Action 分别对应三种不同的沟通方式喵。
Topic 像餐厅大屏广播：「3 号桌需要水」「厨房当前出餐速度」「大厅温度 26°C」会持续发布，谁关心谁就订阅喵。
这种广播适合传递连续状态，例如 `/scan`、`/odom`、`/camera/image`、`/cmd_vel`、`/joint_states` 喵。
Service 像去柜台问一句明确的问题：「请帮我查一下 3 号桌账单是多少」或「请清空一下本桌的餐盘」喵。
柜台收到请求后给出一次明确答复，问答结束，这就是同步请求响应喵。
Action 像外卖下单：「请把一份套餐送到 A 楼 12 层」不是一秒内能完成的事，所以你需要看到骑手已接单、正在取餐、距离还剩多少、最后是否送达，也可能中途取消订单喵。
导航到某个目标点、机械臂规划并执行一段轨迹、自动泊车或长时间检测流程，都更像 Action，而不是一次 Service 调用喵。
这三个机制不是谁替代谁，而是分别回答三个问题：我要持续听状态吗，我要问一个短问题吗，我要发起一个可跟踪的长任务吗喵。

## Core Mechanism

### 1. Service：同步请求响应
Service 的核心模型是 request → response，也就是客户端发出一次请求，服务端处理后返回一次结果喵。
在 ROS 1 中，Python 常用 `rospy.Service` 创建服务端，用 `rospy.ServiceProxy` 创建客户端喵。
在 ROS 2 中，Python 常用 `node.create_service()` 创建服务端，用 `node.create_client()` 创建客户端，并经常用 `call_async()` 发起异步调用喵。
Service 的接口文件通常是 `.srv`，中间用一行 `---` 分隔请求字段和响应字段喵。
例如一个加法服务可以写成这样喵：
```srv
int64 a
int64 b
---
int64 sum
```
`---` 上方是客户端必须提供的输入，下方是服务端返回的输出喵。
这和昨天学过的 `.msg` 不一样，`.msg` 描述单向消息结构，`.srv` 同时描述请求和响应两半喵。
Service 适合处理短、小、确定的问题喵。
典型例子包括清空导航代价地图、查询某个参数、重置仿真世界、生成一个仿真实体、切换控制器、触发一次标定喵。
实习场景里常见的例子有 ROS 1 `move_base` 的 `/move_base/clear_costmaps`，Gazebo 的 `/gazebo/spawn_urdf_model` 或 `/gazebo/spawn_sdf_model`，以及控制器管理器里的加载、卸载、切换控制器服务喵。
Service 不适合拿来做长任务喵。
原因不是「不能写」，而是它的交互模型天生只有一次请求和一次响应喵。
如果你用 Service 做「让机器人走到 10 米外」这种任务，客户端很可能在等待期间阻塞，看不到中间进度，也不好取消任务喵。
一旦服务端执行很久，调用方会不知道是机器人还在执行、网络卡住、服务端崩了，还是任务已经失败但没有返回喵。
ROS 1 的 `rospy.wait_for_service(timeout=...)` 只能给「等待服务出现」设超时，不等于给真正的服务调用设了完整超时喵。
因此初学阶段要记住：Service 是短问答，不是长流程控制器喵。

### 2. Action：目标、反馈、结果
Action 面向的是会持续一段时间的目标型任务喵。
Action client 发送 goal，Action server 接收 goal 后执行，并在执行过程中持续发 feedback，最后返回 result 喵。
这三段式可以记成 goal → feedback → result 喵。
Action 通常还支持 cancel，也就是客户端可以取消一个尚未完成的目标喵。
Action server 也可以 preempt，也就是用新目标抢占旧目标，这在导航和机械臂任务里非常常见喵。
例如给移动机器人发送一个导航目标，机器人先规划全局路径，再边走边避障，期间可能不断反馈当前位置或剩余距离，最后返回成功、失败、被取消或超时喵。
如果使用 Service 做同一件事，你只能发出请求后干等结果，无法自然地观察中间状态喵。
在 ROS 1 中，Action 的主流实现是 `actionlib` 喵。
ROS 1 的一个 Action 在底层通常展开为 5 个 Topic：`goal`、`cancel`、`status`、`feedback`、`result` 喵。
这说明 Action 不是一条神秘的新链路，而是在 Topic 机制上组合出「发目标、取消、查状态、收反馈、收结果」这一套协议喵。
在 ROS 2 中，Action 是 ROS 2 通信图里的一级概念，客户端常用 `rclcpp_action` 或 `rclpy.action` 喵。
ROS 2 Action 的底层仍由多条通信通道组合完成，常见结构可以理解为发送 goal、获取 result、取消 goal 的服务，以及 feedback、status 等话题喵。
所以 ROS 2 的 Action 使用体验比 ROS 1 更统一，但思想仍然是「长任务要有进度、结果和取消能力」喵。
Action 的接口文件是 `.action`，通常分成三段喵。
第一段是 goal，第二段是 result，第三段是 feedback，中间也用 `---` 分隔喵。
一个简化的动作接口可以这样理解喵：
```action
# Goal
geometry_msgs/PoseStamped target_pose
---
# Result
bool success
string message
---
# Feedback
geometry_msgs/PoseStamped current_pose
float32 distance_remaining
```
`.action` 和 `.srv` 的区别很重要喵。
`.srv` 是 request 和 response 两段，适合一次问答喵。
`.action` 是 goal、result、feedback 三段，适合持续执行并反馈进度喵。

### 3. Topic、Service、Action 的选型口诀
如果数据会持续变化，而且接收方不止一个，就优先考虑 Topic 喵。
如果任务很短，只需要一次请求和一次回答，就优先考虑 Service 喵。
如果任务会持续一段时间，需要反馈、结果、取消或抢占，就优先考虑 Action 喵。
更口语化地说：看状态用 Topic，问短事用 Service，派长活用 Action 喵。
机器人开发中最常犯的设计错误，是把所有东西都当 Topic，或者把长任务硬塞进 Service 喵。
例如 `/odom` 用 Topic 合理，因为机器人位姿持续变化喵。
例如 `/clear_costmaps` 用 Service 合理，因为清空代价地图是一个短触发动作喵。
例如「导航到会议室门口」用 Action 合理，因为它有目标、有过程、有失败、有取消喵。

### 4. ROS 1 与 ROS 2 的关键差异
ROS 1 实习环境里，你大概率会看到 `rospy.Service`、`rospy.ServiceProxy`、`actionlib.SimpleActionClient` 和 `actionlib.SimpleActionServer` 喵。
ROS 1 的 Service 调用在 Python 里常以阻塞方式使用，写起来简单，但需要自己处理等待、异常和卡住的风险喵。
ROS 1 的 Action 常依赖 `actionlib`，导航栈 `move_base` 就是最典型的例子喵。
ROS 2 中，Service 客户端更常见的写法是异步调用，例如 `client.call_async(request)`，然后通过 future 等待结果喵。
ROS 2 的 Action 客户端使用 `rclpy.action.ActionClient` 或 C++ 的 `rclcpp_action::Client` 喵。
ROS 2 的 Nav2 不再使用 ROS 1 的 `move_base`，而是提供 `nav2_msgs/action/NavigateToPose` 等 Action 接口喵。
ROS 2 的工具命令也更统一，例如 `ros2 service list`、`ros2 service call`、`ros2 action list`、`ros2 action send_goal --feedback` 喵。
对实习来说，建议先把 ROS 1 的 `move_base` 和 `actionlib` 看懂，因为很多车企和研究院的仿真、测试遗留项目仍以 ROS 1 为主喵。
同时也要知道 ROS 2 的 Nav2 和 ActionClient 写法，因为新项目、仿真平台升级和招聘面试里会频繁问到 ROS 2 差异喵。

### 5. 与实习强相关的任务接口
ROS 1 的 `move_base` 接收导航目标时，核心接口是 `move_base_msgs/MoveBaseAction` 喵。
你在 RViz 里点 `2D Nav Goal`，本质上就是向导航系统发送目标位姿，底层可以对应到 Action 目标喵。
ROS 2 Nav2 的常见导航接口是 `nav2_msgs/action/NavigateToPose` 喵。
它表示「导航到某个位姿」这个长任务，天然需要反馈、结果和取消喵。
MoveIt 里，规划和执行机械臂轨迹通常也是长任务喵。
ROS 1 MoveIt 常见的 Action 包括 `moveit_msgs/MoveGroupAction` 和 `moveit_msgs/ExecuteTrajectoryAction` 喵。
机械臂规划可能失败，执行可能被碰撞检测或控制器错误打断，用户也可能取消或改目标，所以它不适合用普通 Service 承载完整执行流程喵。
不过 MoveIt 和导航系统里也会有 Service，例如查询规划场景、清理状态、切换参数或做一次短触发喵。
Gazebo 里 `/gazebo/spawn_sdf_model`、`/gazebo/spawn_urdf_model` 常作为 Service 使用，因为「生成一个模型并返回是否成功」是一次短请求喵。
`/move_base/clear_costmaps` 也是 Service，因为它只是触发清空局部和全局代价地图，不需要持续反馈喵。
所以同一个系统里 Topic、Service、Action 会同时存在，关键是看任务语义，而不是看模块名字喵。

## Code Example
下面给两个 ROS 1 Python 例子，因为实习环境通常 ROS 1 居多喵。
第一个例子展示一个最小 Service server 和 client，第二个例子展示如何用 Action client 给 `move_base` 发送导航目标并接收 feedback 喵。

### 例子 A：ROS 1 Service server + client
假设你的功能包名为 `my_robot_msgs`，并定义了 `srv/AddTwoInts.srv` 喵。
```srv
int64 a
int64 b
---
int64 sum
```
服务端文件可以命名为 `add_two_ints_server.py` 喵。
```python
#!/usr/bin/env python3
# -*- coding: utf-8 -*-

import rospy
from my_robot_msgs.srv import AddTwoInts, AddTwoIntsResponse


def handle_add_two_ints(req):
    """处理一次加法请求喵。"""
    result = req.a + req.b
    rospy.loginfo("收到请求：%d + %d = %d", req.a, req.b, result)
    return AddTwoIntsResponse(sum=result)


def main():
    rospy.init_node("add_two_ints_server")

    # 创建名为 add_two_ints 的 Service，类型是 AddTwoInts 喵。
    service = rospy.Service("add_two_ints", AddTwoInts, handle_add_two_ints)

    rospy.loginfo("Service 已启动：/add_two_ints")
    rospy.spin()


if __name__ == "__main__":
    main()
```
客户端文件可以命名为 `add_two_ints_client.py` 喵。
```python
#!/usr/bin/env python3
# -*- coding: utf-8 -*-

import sys
import rospy
from my_robot_msgs.srv import AddTwoInts


def call_add_two_ints(a, b):
    # 等待服务出现，避免服务端还没启动就调用喵。
    rospy.wait_for_service("add_two_ints", timeout=5.0)

    try:
        # ServiceProxy 是客户端侧的调用入口喵。
        add_two_ints = rospy.ServiceProxy("add_two_ints", AddTwoInts)
        response = add_two_ints(a, b)
        return response.sum
    except rospy.ServiceException as exc:
        rospy.logerr("Service 调用失败：%s", exc)
        raise


def main():
    rospy.init_node("add_two_ints_client")

    if len(sys.argv) != 3:
        rospy.logerr("用法：rosrun your_pkg add_two_ints_client.py 3 5")
        return

    a = int(sys.argv[1])
    b = int(sys.argv[2])
    result = call_add_two_ints(a, b)
    rospy.loginfo("结果：%d + %d = %d", a, b, result)


if __name__ == "__main__":
    main()
```
这段 Service 代码里，`.srv` 文件定义了请求字段 `a`、`b` 和响应字段 `sum` 喵。
服务端通过 `rospy.Service()` 把服务名、服务类型和回调函数绑定在一起喵。
每次客户端调用 `/add_two_ints`，ROS 都会把请求交给 `handle_add_two_ints(req)`，然后把返回值作为响应发回客户端喵。
客户端通过 `rospy.wait_for_service()` 等服务出现，再用 `rospy.ServiceProxy()` 发起调用喵。
这个例子适合 Service，因为加法是一个短任务，客户端只关心最终答案，不需要中间进度喵。
如果把「机器人导航到目标点」也写成这种结构，客户端就只能卡在函数调用里等最终结果，这就是设计上的不合适喵。

### 例子 B：ROS 1 Action client 调用 move_base
下面的例子会向 ROS 1 `move_base` 发送一个地图坐标系下的导航目标，并在执行过程中打印 feedback 喵。
运行前通常需要已经启动地图、定位、代价地图和 `move_base` 节点喵。
```python
#!/usr/bin/env python3
# -*- coding: utf-8 -*-

import rospy
import actionlib
from actionlib_msgs.msg import GoalStatus
from move_base_msgs.msg import MoveBaseAction, MoveBaseGoal


def feedback_cb(feedback):
    # feedback 里会带有机器人当前的 base_position 喵。
    pose = feedback.base_position.pose
    rospy.loginfo(
        "导航反馈：当前 x=%.2f, y=%.2f",
        pose.position.x,
        pose.position.y,
    )


def main():
    rospy.init_node("send_move_base_goal")

    # move_base 是 Action server 名称，MoveBaseAction 是动作类型喵。
    client = actionlib.SimpleActionClient("move_base", MoveBaseAction)

    rospy.loginfo("等待 move_base Action server...")
    if not client.wait_for_server(rospy.Duration(10.0)):
        rospy.logerr("等待 move_base 超时，请检查导航栈是否启动")
        return

    goal = MoveBaseGoal()
    goal.target_pose.header.frame_id = "map"
    goal.target_pose.header.stamp = rospy.Time.now()

    # 设置一个简单目标点，实际项目里要换成安全可达的位置喵。
    goal.target_pose.pose.position.x = 1.0
    goal.target_pose.pose.position.y = 0.5
    goal.target_pose.pose.orientation.w = 1.0

    rospy.loginfo("发送导航目标")
    client.send_goal(goal, feedback_cb=feedback_cb)

    # 最多等待 60 秒，避免测试脚本无限卡住喵。
    finished = client.wait_for_result(rospy.Duration(60.0))

    if not finished:
        rospy.logwarn("导航超时，取消当前目标")
        client.cancel_goal()
        return

    state = client.get_state()
    result = client.get_result()

    if state == GoalStatus.SUCCEEDED:
        rospy.loginfo("导航成功")
    else:
        rospy.logwarn("导航未成功，状态码：%d，结果：%s", state, result)


if __name__ == "__main__":
    main()
```
这段 Action 代码里，`SimpleActionClient("move_base", MoveBaseAction)` 表示连接名为 `move_base` 的 Action server 喵。
`MoveBaseGoal` 里最关键的是 `target_pose`，它说明机器人要到哪个坐标系下的哪个目标位姿喵。
`send_goal()` 不只是发出目标，还可以注册 `feedback_cb`，让客户端在任务执行过程中持续收到反馈喵。
`wait_for_result()` 用来等待最终结果，但这里加了 60 秒上限，避免测试脚本在异常情况下长时间挂起喵。
如果超时，`cancel_goal()` 会向 Action server 发送取消请求，这正是 Action 比 Service 更适合长任务的地方喵。
在真实项目里，还会根据 `GoalStatus` 区分 `ABORTED`、`PREEMPTED`、`REJECTED` 等状态，然后把它们转成测试报告或故障定位线索喵。
如果换成 ROS 2 Nav2，思路仍是 Action client 发送目标、订阅 feedback、等待 result，但接口会变成 `nav2_msgs/action/NavigateToPose` 和 `rclpy.action.ActionClient` 喵。

## Common Misconceptions
1. 误区一：Service 也能等很久，所以长任务用 Service 也没问题喵。
Service 技术上可以让服务端执行很久再返回，但这会让调用方缺少进度、取消和状态表达，调试体验也很差喵。
2. 误区二：Action 是 Topic 和 Service 之外的一种完全独立网络协议喵。
Action 更准确的理解是 ROS 在 Topic 和 Service 等通信机制上封装出的长任务协议喵。
ROS 1 的 `actionlib` 特别直观，因为一个 Action 会展开成 `goal`、`cancel`、`status`、`feedback`、`result` 这 5 个 Topic 喵。
3. 误区三：`.msg`、`.srv`、`.action` 只是后缀不同，随便选一个都能表达接口喵。
`.msg` 表达单条消息，`.srv` 表达一次请求和一次响应，`.action` 表达目标、结果和反馈三段语义喵。
接口文件的选择会影响调用方式、工具命令和系统可观测性，不能只看字段是否能塞进去喵。
4. 误区四：Service 调用失败后 ROS 会自动重试喵。
通常不会自动按你的业务意图重试，客户端需要自己处理服务不存在、连接失败、服务端异常和返回错误等情况喵。
5. 误区五：`wait_for_service(timeout=...)` 就等于整个 Service 调用都有超时保护喵。
在 ROS 1 Python 里，它主要保护等待服务可用这个阶段，真正调用后如果服务端长时间不返回，仍可能让调用方卡住喵。

## Socratic Check
问题 1：如果你要写一个「清空 costmap」按钮，它应该用 Topic、Service 还是 Action，为什么喵？
参考答案：更适合用 Service，因为它是一个短触发任务，客户端发出清空请求后只需要知道是否调用成功，不需要持续反馈喵。
问题 2：如果你要写一个「让机器人从 A 点导航到 B 点，并且中途能看到进度、必要时取消」的接口，它应该用什么通信方式，为什么喵？
参考答案：更适合用 Action，因为导航是长任务，需要 goal、feedback、result 和 cancel 语义喵。

## Quick Summary
- Topic 适合持续广播状态，Service 适合短请求响应，Action 适合可跟踪、可取消的长任务喵。
- Service 的 `.srv` 文件分为 request 和 response 两段，不能自然表达中间进度喵。
- Action 的 `.action` 文件分为 goal、result、feedback 三段，ROS 1 常用 `actionlib`，ROS 2 常用 `rclpy.action` 或 `rclcpp_action` 喵。
- `move_base`、Nav2 的 `NavigateToPose`、MoveIt 的规划执行接口都体现了 Action 在机器人任务层的价值喵。
- `/clear_costmaps`、Gazebo spawn model、控制器切换这类短触发操作通常更适合 Service 喵。
