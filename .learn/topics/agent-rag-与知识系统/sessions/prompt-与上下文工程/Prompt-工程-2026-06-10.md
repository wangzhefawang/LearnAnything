## 定位

Prompt 工程位于知识树的 **LLM 与 Prompt 基础 → Prompt 工程** 分支，它是你和大模型打交道的「第一语言」——后面要学的 Agent、RAG、工作流，本质上都是把 Prompt 工程系统化、工程化的产物喵。

## 类比

把 LLM 想象成一位**刚入职的天才实习生**：他读过全世界的书、什么都懂一点，但他**不了解你的公司、不知道你的目标、也猜不到你心里的标准**。

- 你说「帮我写个报告」→ 他只能瞎猜格式、长度、受众，交上来的东西多半不能用。
- 你说「你是部门数据分析师，给 CTO 写一份 300 字的周报，分『进展/风险/下周计划』三段，用要点列表」→ 他立刻交出接近完美的成果。

**Prompt 工程就是「给天才实习生写任务说明书」的艺术**：模型能力是固定的，你说明任务的方式决定了你能调动出它几成功力。

## 核心机制

LLM 本质是「根据前文预测下一个词」的概率机器，所以你给它的 Prompt 就是在**塑造它的概率分布**。四个最核心的技巧：

**1. 角色设定（Role Prompting）**
告诉模型「你是谁」，相当于把它的概率分布拉向某个专业领域。「你是一位资深 Python 代码审查员」会让输出更挑剔、更专业，因为它在模仿训练数据里这类角色的说话方式。

**2. Few-shot 示例（少样本学习）**
与其描述规则，不如直接给 2-3 个「输入 → 输出」的例子。模型非常擅长模仿模式——例子就是最精确的规格说明，比一大段文字描述更有效。

**3. 思维链（Chain-of-Thought, CoT）**
让模型「先一步步思考，再给答案」。因为模型是逐词生成的，中间推理过程就是它的「草稿纸」——直接要答案容易出错，先推理再回答准确率显著提升。

**4. 结构化输出**
要求模型按 JSON、Markdown 表格等固定格式输出。这是**工程落地的关键**：格式固定了，下游代码才能解析，AI 才能接进自动化流程——这正是「不只会玩 Demo」和「会做工程」的分水岭。

## 代码示例

真实工作场景：把杂乱的会议记录自动转成结构化纪要：

```python
import json
from openai import OpenAI

client = OpenAI()  # 任何 OpenAI 兼容接口都适用

# 一个综合运用四大技巧的 Prompt
system_prompt = """
你是一位专业的会议纪要整理助手。              # ← 技巧1：角色设定

请一步步分析会议记录：先识别参与者，          # ← 技巧3：思维链
再提取决议事项，最后整理待办任务。

严格按以下 JSON 格式输出：                    # ← 技巧4：结构化输出
{
  "participants": ["姓名"],
  "decisions": ["决议内容"],
  "todos": [{"task": "任务", "owner": "负责人", "deadline": "期限"}]
}

示例：                                        # ← 技巧2：Few-shot 示例
输入：小王说周五前出方案，老李拍板用方案A。
输出：{"participants": ["小王", "老李"],
      "decisions": ["采用方案A"],
      "todos": [{"task": "输出方案", "owner": "小王", "deadline": "周五"}]}
"""

meeting_notes = "今天小张提了三个风险点，最后大家同意延期一周发布，小刘负责周三前更新测试报告。"

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "system", "content": system_prompt},
        {"role": "user", "content": meeting_notes},
    ],
    response_format={"type": "json_object"},  # 强制 JSON，保证可解析
)

result = json.loads(response.choices[0].message.content)
print(result["todos"])
# [{'task': '更新测试报告', 'owner': '小刘', 'deadline': '周三'}]
```

代码走读：`system_prompt` 把四个技巧打包在一起——角色定基调、思维链保证提取质量、JSON 格式让结果能直接 `json.loads()` 进数据库或推送到飞书，Few-shot 例子消除格式歧义。这一个模式吃透了，「会议纪要整理」「周报生成」「任务排期」全是它的变体。

## 常见误区

1. **「Prompt 工程就是会说漂亮话」** —— 错。核心是**精确的任务规格**：输入是什么、输出长什么样、边界情况怎么处理，更像写需求文档而不是写作文。
2. **把所有要求堆进一个超长 Prompt** —— 模型对中间部分的指令遵循度会下降（lost in the middle）。复杂任务应该**拆成多步流水线**，每步一个小 Prompt——这其实就是工作流编排的雏形。
3. **写完一版就以为完事了** —— Prompt 是需要**迭代测试**的工程产物：准备一批测试用例，改一版跑一遍，像调参一样调 Prompt。
4. **忽略「负面指令失效」问题** —— 「不要输出多余解释」常常没用，更可靠的是正面规定「只输出 JSON，第一个字符必须是 `{`」。

## 苏格拉底式检查

假设你用上面的代码处理会议记录，但模型偶尔会在 JSON 前面加一句「好的，以下是整理结果：」，导致 `json.loads()` 报错。结合刚才讲的技巧，你猜有哪些办法能解决这个问题？（提示：至少有三种不同层面的方案）

## 快速总结

- Prompt 工程 = 给「天才实习生」写精确的任务说明书，塑造模型的概率分布
- 四大核心技巧：角色设定、Few-shot 示例、思维链 CoT、结构化输出
- 结构化输出（JSON）是 AI 接入自动化流程的关键，是 Demo 和工程的分水岭
- Prompt 是工程产物：要拆分流水线、要迭代测试、要用正面指令
