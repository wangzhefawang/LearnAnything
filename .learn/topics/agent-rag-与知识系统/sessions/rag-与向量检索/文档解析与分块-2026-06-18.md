## 定位

它在知识图谱里位于 **RAG 知识库 → 文档解析与分块**，是整条 RAG 流水线的**第一道工序**喵。

整条链路串起来是这样：

```
原始文档（PDF/Word/网页）
   │  ← ① 文档解析（Parsing）：把它变成纯文本
   ▼
一大段纯文本
   │  ← ② 分块（Chunking）：切成一小段一小段
   ▼
多个文本块（chunks）
   │  ← ③ 向量化 Embedding
   ▼
向量 → ④ 存进向量数据库 → ⑤ 检索
```

「向量化」和「向量数据库」是第 ③④ 步，而本节是喂给它们的**原料加工厂** —— 原料切得好不好，直接决定后面检索准不准。

## 类比

想象准备一本**考试用的活页笔记**：

- **文档解析**：把厚厚的纸质教材（PDF、扫描件、网页）一页页抄成可编辑文字，去掉图片、页眉页脚、乱排版，只留干净正文。
- **分块**：把这些文字剪成一张张小卡片，每张记一个相对完整的知识点，方便快速翻找。

关键：卡片不能剪太大（信息太杂），也不能剪太碎（意思被切断）。这个「剪多大」的取舍就是分块的核心艺术。

## 核心机制

### （1）文档解析 Parsing —— 为什么不能直接用原文件？

LLM 和 Embedding 模型只认**纯文本**，但现实文档格式繁多：
- **PDF** —— 可能是文字版（能直接抽取）或扫描图片版（需要 OCR）；常有分栏、表格、页眉页脚干扰。
- **Word / Markdown / HTML** —— 带标签和样式，需要剥离。
- **表格、图片** —— RAG 里最头疼，纯文本会丢掉二维结构。

目标：**任意格式 → 干净、有结构的纯文本**，尽量保留标题层级等结构信息。

### （2）分块 Chunking —— 为什么必须切？

两个硬约束：

1. **Embedding 模型有输入长度上限**（如 512 / 8192 token），整篇塞不进去。
2. **检索精度** —— 块太大 → 向量变成「大杂烩」、语义被平均 → 检索不准、还把无关内容塞给 LLM，浪费上下文且误导。

**三个关键参数：**

| 参数 | 含义 | 经验值 |
|------|------|--------|
| `chunk_size` | 每块多大（字符 / token） | 中文常用 300–500 字 / 英文 500–1000 token |
| `chunk_overlap` | 相邻块重叠多少 | chunk_size 的 10%–20% |
| 切分依据 | 在哪里下刀 | 优先按段落/句子边界，而非硬切 |

**为什么需要 overlap？** 硬切会把一句完整的话切成两半分落两块，两块都看不懂。让相邻块重叠一小段，保证语义不被切断。

**常见分块策略（由粗到精）：**
- **固定长度切分**：每 N 字符硬切，最简单但易切断句子。
- **递归字符切分（最常用）**：按「段落→句子→词」顺序找分隔符下刀，LangChain 的 `RecursiveCharacterTextSplitter`。
- **按文档结构切分**：按 Markdown 标题/章节分，保留天然层级。
- **语义切分**：用 Embedding 判断相邻句语义突变，在话题转折点下刀，最智能也最贵。

## 代码示例

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

# 假设这是文档解析后得到的纯文本
text = """RAG 全称是检索增强生成。它的核心思想是：在让大模型回答问题之前，
先从外部知识库里检索出相关资料，再把资料和问题一起喂给模型。
这样做的好处是模型能引用最新的、私有的知识，而不必依赖训练时记住的内容。
分块是 RAG 的第一步，切得好不好直接影响检索质量。"""

splitter = RecursiveCharacterTextSplitter(
    chunk_size=50,        # 每块最多约 50 个字符
    chunk_overlap=10,     # 相邻块重叠 10 个字符，防止语义被切断
    separators=["\n\n", "\n", "。", "，", " ", ""],  # 下刀优先级：段落>换行>句号>逗号>...
)

chunks = splitter.split_text(text)

for i, chunk in enumerate(chunks):
    print(f"--- 块 {i}（长度 {len(chunk)}）---")
    print(chunk)
```

**逐段讲解：**
- `separators` 是核心：先用 `\n\n`（段落）切，若仍超过 `chunk_size` 就递归降级用 `\n`、`。`……一路降到按字符切，保证「能在自然边界断开就绝不硬断」。
- `chunk_overlap=10`：第 1 块结尾 10 字符会重复出现在第 2 块开头，形成滑动窗口，跨块句子也能被完整理解。
- 输出为若干长度 ≤ 50 的小块，每块尽量在句号/逗号处干净断开。

## 常见误区

1. **❌「chunk_size 越大越好」** —— 块太大→向量语义被稀释→检索召回一堆不相关内容→LLM 被噪声淹没。大小要按「单个知识点的粒度」调。
2. **❌「直接把 PDF 文字一股脑抽出来就行」** —— 不清洗的话页眉页脚、页码、目录、参考文献会污染知识库。**解析质量是 RAG 的天花板**，垃圾进垃圾出。
3. **❌「overlap 没用，浪费存储」** —— 没 overlap，边界上被切断的关键句会变成「两边都答不上」的盲区，是漏检常见元凶。
4. **❌「表格也当普通文本切」** —— 按行硬切会让列名和数据分家，片段失去意义。表格需特殊处理（转 Markdown 表格、或整表作为一块）。

## 苏格拉底式检查

> 有一份《公司员工手册》，含「请假流程」「报销流程」等很多独立小节，每节约 200 字。用户会问「年假怎么请？」这类很具体的问题。
> `chunk_size` 该设大一点（如 800，塞好几节）还是小一点（如 250，约一节一块）？为什么？

参考思路：这种「问题指向单个独立小节」的场景，**倾向小一点、贴合一节的粒度**，让每块语义聚焦、检索精准命中那一节，不把无关小节捞进来。

## 速记摘要

- 文档解析 = 任意格式→干净纯文本（PDF 可能要 OCR，要清洗页眉页脚）；分块 = 把长文本切成语义完整的小块。
- 必须切块的两个原因：Embedding 有长度上限 + 检索精度（块太大语义被稀释）。
- 三个旋钮：`chunk_size`（块大小）、`chunk_overlap`（重叠，防切断语义）、切分依据（优先自然边界）。
- 最常用策略是递归字符切分（按段落→句子→词降级），表格等特殊结构要特殊处理。
- 解析质量是 RAG 效果的天花板：垃圾进、垃圾出。
