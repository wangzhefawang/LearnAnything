## 定位

记忆机制位于 **Agent 智能体 → 记忆机制** 分支，分为短期对话记忆和长期持久化记忆两层。它是上下文管理的直接升级：上下文管理解决「一次会话内记什么」，记忆机制解决「**跨会话、跨任务**还能记得什么」——没有它，Agent 每次重启都是失忆的新员工。

## 类比

还是那位没有记忆的天才实习生。上一讲的对策是「每次递给他一沓会议纪要」——但纪要只在当天有效，他明天来上班又是一张白纸。

真正的解决方案是给他配**一套办公系统**：

- **便签纸**（短期记忆）：今天这个任务里的临时信息，下班就扔。
- **个人笔记本**（长期记忆-事实）：「老板姓张、报销走 OA、预算上限 5000」——随时翻阅、持续补充。
- **档案柜**（长期记忆-经验）：过去每个项目的完整卷宗，太多了不可能全背下来，但需要时能**按关键词检索**出相关那几份。

实习生每天早上的例行动作：翻笔记本 → 按今天的任务去档案柜抽几份相关卷宗 → 开始干活。**记忆机制 = 便签纸 + 笔记本 + 档案柜 + 这套「早晨例行动作」**。

## 核心机制

**1. 短期记忆（Working Memory）**

当前会话的 `history` + `summary`，生命周期到会话结束为止。Agent 的短期记忆还包括**工具调用的中间结果**——「我刚查过数据库，结果是 X」也是记忆，丢了就会重复调用。

**2. 长期记忆（Persistent Memory）——核心是「写什么、存哪里、怎么取」**

**写什么**：不是把聊天记录全存下来（那叫日志），而是提取**值得记的结论**，通常分三类：

| 类型 | 例子 | 类比 |
|------|------|------|
| 事实 Facts | 「用户偏好用 PowerShell」「项目部署在阿里云」 | 笔记本里的条目 |
| 经验 Episodes | 「上次任务 X 用方案 A 失败了，换 B 才成功」 | 档案柜的卷宗 |
| 技能 Procedures | 「生成周报的标准流程是 1→2→3」 | 员工手册 |

**写入时机**：每轮对话后让 LLM 自动判断「有没有值得长期记住的信息」（自动提取），或由用户/Agent 显式触发「记住这一点」（手动写入）。

**存哪里 & 怎么取**——两条主流路线：

- **文件/结构化存储**：记成 Markdown 或 JSON，启动时全量加载进系统提示词。简单、可读、可手改，适合记忆量小的场景——Claude Code 的 `CLAUDE.md` 和记忆目录就是这条路。
- **向量数据库**：每条记忆转成向量存起来，用时按**语义相似度**检索 Top-K 条注入上下文。适合记忆量大的场景——档案柜模式，这就是 RAG 技术在记忆上的应用。

实践中常见混合：高频核心事实走文件（全量加载），海量历史经验走向量库（按需检索）。

**3. 记忆的完整生命周期**

```
对话进行 → 提取(LLM判断值得记什么) → 写入(文件/向量库)
   ↑                                        ↓
回答时注入 ← 检索(全量加载 或 相似度Top-K) ← 下次会话启动
```

外加常被忽略的环节：**遗忘与更新**——「用户偏好 Python」后来变成「偏好 Go」，旧记忆必须被覆盖而不是并存，否则 Agent 会精神分裂。

## 代码示例

文件型长期记忆的最小可用实现——`pinned_facts` 的跨会话版：

```python
import json
from pathlib import Path
from openai import OpenAI

client = OpenAI()
MODEL = "gpt-4o"
MEMORY_FILE = Path("agent_memory.json")   # 持久化到磁盘，重启不丢

class MemoryStore:
    def __init__(self):
        self.facts: list[str] = (
            json.loads(MEMORY_FILE.read_text(encoding="utf-8"))
            if MEMORY_FILE.exists() else []
        )

    def extract_and_save(self, user_msg: str, assistant_msg: str):
        """每轮对话后：让 LLM 判断有没有值得长期记住的信息"""
        resp = client.chat.completions.create(
            model=MODEL,
            messages=[{
                "role": "user",
                "content": (
                    "从下面的对话中提取值得长期记住的事实（用户偏好、约束、决定）。\n"
                    "每条一句话；如果与已有记忆矛盾，输出更新后的版本；没有则输出 []。\n"
                    f"已有记忆：{json.dumps(self.facts, ensure_ascii=False)}\n"
                    f"用户：{user_msg}\n助手：{assistant_msg}\n"
                    '只输出 JSON 数组，如 ["用户预算上限为5000元"]'
                ),
            }],
            response_format={"type": "json_object"},  # API 层约束
        )
        new_facts = json.loads(resp.choices[0].message.content)
        if new_facts:
            self.facts = self._merge(new_facts)       # 更新而非无脑追加
            MEMORY_FILE.write_text(
                json.dumps(self.facts, ensure_ascii=False, indent=2),
                encoding="utf-8",
            )

    def _merge(self, new_facts: list[str]) -> list[str]:
        """简化版冲突处理：这里偷懒直接追加；生产中应让 LLM 合并矛盾条目"""
        return list(dict.fromkeys(self.facts + new_facts))

    def inject(self) -> str:
        """会话启动时：全量注入系统提示词"""
        return "关于用户的已知信息：\n" + "\n".join(f"- {f}" for f in self.facts)

# ===== 使用 =====
memory = MemoryStore()           # 启动：从磁盘加载（翻开笔记本）
system_prompt = f"你是私人助理。\n{memory.inject()}"   # 注入记忆
# ... 正常对话（复用上一讲的 ChatSession）...
# 每轮结束后：
memory.extract_and_save("我们预算定在5000以内", "好的，已记住预算约束")
# → agent_memory.json 中多了一条 "用户预算上限为5000元"，下次启动依然在
```

代码走读：`MemoryStore` 把记忆生命周期四步全走了一遍——启动加载（检索）、`inject()` 进系统提示词（注入）、`extract_and_save()` 用 LLM 自动提取（提取+写入），提取 Prompt 里特意写了「与已有记忆矛盾则输出更新版」（遗忘与更新）。它复用了之前学过的全部技巧：结构化输出、API 层 JSON 约束、正面指令。

## 常见误区

1. **「把全部聊天记录存下来就是长期记忆」** —— 那是日志不是记忆。记忆的本质是**提取过的结论**；不加提炼全量存储，检索时噪音淹没信号，注入时白白烧 Token。
2. **只追加、从不更新** —— 「偏好 Python」和「偏好 Go」并存时，Agent 的行为取决于模型今天心情。写入前必须做冲突检测，矛盾的旧记忆要被覆盖或标记过期。
3. **什么都记** —— 「用户今天说了声谢谢」不值得进长期记忆。记忆质量 > 记忆数量，写入门槛要高：只记**会影响未来行为**的信息。
4. **把记忆当成绝对真相** —— 记忆反映的是「写入时」的状态，环境可能已经变了。高风险操作前应该验证关键记忆是否仍然成立。

## 苏格拉底式检查

文件型记忆是「全量加载」：启动时把 `agent_memory.json` 整个塞进系统提示词。假设 Agent 在公司用了一年，积累了 3000 条记忆——这套方案会在哪里先撑不住？分界点出现后该换成什么方案，新方案又会引入什么新问题？（提示：档案柜的类比，以及「Token = 成本 + 延迟」）

## 快速总结

- 短期记忆 = 会话内的 history/summary/工具结果；长期记忆 = 跨会话持久化的事实/经验/技能
- 记忆生命周期：提取 → 写入 → 检索 → 注入，外加「遗忘与更新」处理矛盾
- 两条存储路线：文件型（全量加载，简单可读，量小适用）vs 向量库（语义检索 Top-K，量大适用），实践常混合
- 记忆是提取过的结论而非日志；写入门槛要高，只记会影响未来行为的信息
