## Positioning

ADAS/NOA 功能定义是「自动驾驶功能与算法链路」这一域的起点，它把"车要在什么场景下帮用户做什么、做到什么边界、失效时如何退出"说清楚，连接用户体验、法规责任、系统需求、算法能力和量产安全闭环喵。

## Analogy

可以把智驾功能定义理解成"餐厅菜单加服务边界"喵。

算法像厨师能力，传感器像厨房设备，整车执行器像上菜流程，但功能定义先回答菜单上到底卖什么菜、什么时间供应、哪些食材缺货时停售、顾客过敏时如何提醒，以及服务员什么时候必须把决定权交还给顾客喵。

所以量产智驾不是只问"模型能不能规划出轨迹"，还要问"这个能力在什么道路、天气、车速、法规和驾驶责任边界内可以稳定交付给用户"喵。

## Core Mechanism

**1. ADAS 是一组可被清楚定义边界的驾驶辅助功能喵。**

ADAS 是 Advanced Driver Assistance Systems 的缩写，通常指帮助驾驶员完成部分感知、决策或控制任务的辅助驾驶功能集合喵。

功能定义不是算法模块清单，而是面向用户、整车和安全目标描述"触发条件、工作范围、控制对象、退出方式、报警方式、驾驶员责任"的工程语言喵。

典型功能包括 ACC、LCC、AEB、APA 等喵。

- **ACC** 是 Adaptive Cruise Control，自适应巡航控制，核心是**纵向控制**，也就是在设定车速范围内跟随前车、保持时距、自动加减速喵。
- **LCC** 是 Lane Centering Control，车道居中控制，核心是**横向控制**，也就是在车道线或可识别道路边界有效时，让车辆保持在本车道中心附近行驶喵。
- **AEB** 是 Automatic Emergency Braking，自动紧急制动，核心是**碰撞风险缓解**，也就是在驾驶员未及时制动且系统判断存在紧急碰撞风险时自动制动喵。
- **APA** 是 Automatic Parking Assist，自动泊车辅助，核心是**低速泊车场景控制**，也就是在可识别车位和周边障碍物条件满足时辅助完成泊入或泊出喵。

这些功能看起来都与"自动驾驶"有关，但功能定义的粒度不同喵。

ACC 通常只承诺纵向跟车，不承诺识别所有静止障碍物喵。LCC 通常只承诺车道内横向保持，不承诺自动换道或处理复杂路口喵。AEB 通常是安全兜底功能，不是舒适驾驶功能，也不等价于系统能在所有碰撞场景中避免事故喵。APA 通常工作在低速、近距离、结构化泊车环境中，它的 ODD 与高速巡航完全不同喵。

**2. NOA 是把导航意图和组合辅助驾驶能力连接起来的功能形态喵。**

NOA 一词源自特斯拉的 Navigate on Autopilot，如今在行业里泛指导航辅助驾驶或领航辅助驾驶这一类功能喵。

它通常在导航路线约束下，把 ACC、LCC、自动变道、匝道通行、车道选择、限速响应、路口或交通参与者处理等能力组合成更连续的驾驶辅助体验喵。

但 NOA 本身不自动意味着 SAE L3，也不自动意味着车辆承担驾驶责任喵。在大多数乘用车量产语境里，NOA 常以 SAE L2 或 L2+ 的组合辅助驾驶形式出现，驾驶员仍需持续监督并承担动态驾驶任务的责任喵。

高速 NOA 和城市 NOA 的差异，核心不是"算法名字不同"，而是 **ODD 和交互复杂度完全不同**喵。

高速 NOA 面对的是相对结构化的道路环境，包括车道线稳定、交通流方向一致、行人和非机动车较少、路口冲突较少、道路规则较统一等特点喵。它的典型难点包括匝道进出、变道决策、施工区域、静止或低速异常目标、地图与现实不一致、遮挡后的风险预判等喵。

城市 NOA 面对的是更开放、更高熵的道路环境，包括红绿灯、无保护左转、复杂路口、行人横穿、非机动车混行、临停车辆、窄路绕行、博弈式交互和长尾交通规则喵。它的难点不只是感知更多目标，而是要把目标意图预测、交通规则理解、局部博弈、舒适性、安全冗余和用户可理解性同时放进功能边界喵。

小鹏 XNGP 这类"全场景 NOA"语境，可以理解为厂商希望把高速、城市、泊车等多个驾驶辅助场景用更统一的体验和技术栈连接起来喵。面试或工程讨论时，可以提"全场景"强调功能覆盖范围和体验连续性，但不要把它直接说成没有 ODD、没有边界、或天然达到 L3/L4 喵。量产系统越强调全场景，越需要清楚定义可用条件、退出条件、接管提示、降级策略和责任边界喵。

**3. ODD 是功能定义的边界盒子喵。**

ODD 是 Operational Design Domain，运行设计域，指某个自动化或辅助驾驶功能被设计为可以工作的具体条件集合喵。

ODD 可以包含道路类型、地理区域、车速范围、天气、光照、车道线质量、交通参与者类型、地图可用性、传感器状态、驾驶员状态、法规区域和车辆健康状态等维度喵。简单说，ODD 是功能的"营业时间、营业地点和营业条件"喵。

例如，一个高速 NOA 的 ODD 可能要求车辆位于支持的高速或快速路、导航路线有效、车道线或道路边界可靠、传感器无遮挡、驾驶员监控有效、车速处于允许范围内喵。

如果车辆进入城市无保护左转路口、遇到暴雨导致感知置信度下降、地图缺失、摄像头被遮挡、驾驶员长时间不响应提醒，那么系统就可能需要限制功能、降级到 ACC/LCC、发出接管请求，甚至退出辅助驾驶喵。

ODD 的价值在于把"模型在测试集上表现不错"转化为"功能在可控边界内可以被量产交付"喵。

**4. SAE L0–L5 分级描述的是自动化程度和责任边界，不是单纯技术先进程度喵。**

SAE J3016 把驾驶自动化分为 L0 到 L5 喵。

- **L0** 是 No Driving Automation，无驾驶自动化，系统可以警告或短暂辅助，但不持续执行动态驾驶任务喵。
- **L1** 是 Driver Assistance，驾驶辅助，系统持续控制横向或纵向中的**一类**，驾驶员负责其余任务并持续监督喵。
- **L2** 是 Partial Driving Automation，部分驾驶自动化，系统可以**同时**持续控制横向和纵向，但驾驶员必须持续监督环境并随时接管喵。
- **L3** 是 Conditional Driving Automation，有条件驾驶自动化，系统在其 ODD 内执行完整动态驾驶任务，驾驶员在系统请求时需要作为后备用户接管喵。
- **L4** 是 High Driving Automation，高度驾驶自动化，系统在特定 ODD 内不需要人类作为后备完成驾驶任务，超出 ODD 时系统也要能达到最小风险状态喵。
- **L5** 是 Full Driving Automation，完全驾驶自动化，系统可在所有人类驾驶员能处理的道路和环境中完成驾驶任务喵。

对功能定义来说，L2 和 L3 的关键差异不是"有没有激光雷达"或"模型是不是端到端"，而是**动态驾驶任务和驾驶责任如何分配**喵。

L2 组合辅助驾驶可以同时控方向和速度，但驾驶员仍是实时监督者喵。L3 在 ODD 内由系统承担动态驾驶任务，系统还必须定义接管请求、接管时间、最小风险策略、故障处理和法规合规边界喵。

因此，把高速 NOA 或城市 NOA 直接称为 L3 是常见错误，除非它的功能定义、法规责任、HMI、冗余设计、验证证据和产品声明都明确支持 L3 喵。

**5. 在量产开发中，功能定义会向下派生系统需求和验证闭环喵。**

一个智驾系统开发工程师理解功能定义，是因为后续所有工程工作都依赖这份"契约"喵。

功能定义首先描述用户可感知能力，例如"在支持道路上按导航路线辅助变道和上下匝道"喵。然后它会转化为系统需求，例如需要定位、地图、感知、预测、规划、控制、HMI、驾驶员监控和故障诊断共同满足哪些条件喵。接着它会明确 ODD 边界，例如支持道路类型、速度范围、天气限制、车道线质量、驾驶员状态、传感器健康状态等喵。最后它会定义降级策略，例如从城市 NOA 降级到 LCC、从 LCC 降级到 ACC、从 ACC 退出到人工驾驶，并通过声音、视觉、方向盘或制动反馈提醒用户喵。

这条链路可以概括为**"功能定义 → 系统需求 → ODD 边界 → 降级策略 → 测试验证 → 用户说明"**喵。

算法背景的同学如果只盯着模型指标，容易忽视功能定义里最量产的部分，也就是"什么时候不能用、不能用时怎么办、用户是否理解、系统是否可验证"喵。小鹏智驾系统开发工程师岗关注的不是只会写一个规划模型，而是能把算法能力放进整车功能、用户体验、安全约束和量产验证的框架里思考喵。

## Code Example

```python
from dataclasses import dataclass
from enum import Enum


class FunctionState(Enum):
    OFF = "off"
    ACC = "acc"
    LCC = "lcc"
    HIGHWAY_NOA = "highway_noa"
    CITY_NOA = "city_noa"
    TAKEOVER_REQUIRED = "takeover_required"


@dataclass
class VehicleContext:
    road_type: str
    speed_kph: float
    lane_quality: float
    weather: str
    navigation_active: bool
    map_available: bool
    sensors_healthy: bool
    driver_attentive: bool
    complex_intersection: bool


def odd_allows_highway_noa(ctx: VehicleContext) -> bool:
    # 高速 NOA 的功能定义要求道路、导航、地图、感知和驾驶员监督都满足边界喵。
    return (
        ctx.road_type in {"highway", "expressway"}
        and 0 <= ctx.speed_kph <= 130
        and ctx.lane_quality >= 0.7
        and ctx.weather in {"clear", "cloudy", "light_rain"}
        and ctx.navigation_active
        and ctx.map_available
        and ctx.sensors_healthy
        and ctx.driver_attentive
        and not ctx.complex_intersection
    )


def odd_allows_city_noa(ctx: VehicleContext) -> bool:
    # 城市 NOA 的 ODD 更复杂，这里只用极简条件表达功能边界思想喵。
    return (
        ctx.road_type == "urban"
        and 0 <= ctx.speed_kph <= 80
        and ctx.lane_quality >= 0.6
        and ctx.weather in {"clear", "cloudy"}
        and ctx.navigation_active
        and ctx.sensors_healthy
        and ctx.driver_attentive
    )


def choose_degraded_state(ctx: VehicleContext) -> FunctionState:
    # 降级策略把不可用原因转化为更低阶但仍可控的功能状态喵。
    if not ctx.sensors_healthy or not ctx.driver_attentive:
        return FunctionState.TAKEOVER_REQUIRED

    if ctx.lane_quality >= 0.7 and ctx.speed_kph <= 130:
        return FunctionState.LCC

    if ctx.speed_kph <= 150:
        return FunctionState.ACC

    return FunctionState.TAKEOVER_REQUIRED


def request_function(desired: FunctionState, ctx: VehicleContext) -> FunctionState:
    # 功能请求不能只看用户想开什么功能，还要检查该功能的 ODD 是否成立喵。
    if desired == FunctionState.HIGHWAY_NOA and odd_allows_highway_noa(ctx):
        return FunctionState.HIGHWAY_NOA

    if desired == FunctionState.CITY_NOA and odd_allows_city_noa(ctx):
        return FunctionState.CITY_NOA

    if desired in {FunctionState.HIGHWAY_NOA, FunctionState.CITY_NOA}:
        return choose_degraded_state(ctx)

    return desired


ctx = VehicleContext(
    road_type="urban",
    speed_kph=45,
    lane_quality=0.65,
    weather="clear",
    navigation_active=True,
    map_available=False,
    sensors_healthy=True,
    driver_attentive=True,
    complex_intersection=True,
)

state = request_function(FunctionState.HIGHWAY_NOA, ctx)
print(f"当前允许的功能状态是：{state.value}喵")
```

这段代码不是量产逻辑，而是用一个极简状态机表达功能定义的思路喵。

`VehicleContext` 表示车辆当前所处环境和系统健康状态，它对应 ODD 中的道路、速度、天气、地图、传感器和驾驶员状态等条件喵。

`odd_allows_highway_noa` 和 `odd_allows_city_noa` 分别表达高速 NOA 与城市 NOA 的可用边界，体现同样叫 NOA，ODD 也可能完全不同喵。

`request_function` 表示用户或系统想打开某个功能时，不能只检查按钮状态，而要先检查 ODD 是否成立喵。

`choose_degraded_state` 表示功能不可用时的降级策略，例如 NOA 不可用时可能退到 LCC 或 ACC，而不是无提示地继续宣称自己在 NOA 状态喵。示例里驾驶员请求高速 NOA，但当前在城市道路且车道线质量不足 0.7，于是系统降级到 ACC 喵。

这个例子对应量产开发中的主线，也就是先定义功能，再定义可用条件，再定义退出和降级行为喵。

## Common Misconceptions

1. **把 SAE L2、L3 当成纯技术能力排行榜喵。** 真正关键是责任边界和动态驾驶任务分配，而不是传感器数量、模型规模或是否端到端喵。

2. **把 NOA 直接等同于 L3 喵。** 很多 NOA 产品是 L2 或 L2+ 组合辅助驾驶，驾驶员仍需要持续监督，是否 L3 要看功能定义、法规声明、责任转移和系统安全设计喵。

3. **认为 ODD 是法规文档里的附属项喵。** ODD 实际上决定功能能不能开启、何时退出、如何测试、如何向用户解释，也是系统需求和安全分析的核心输入喵。

4. **只用算法指标理解功能可用性喵。** 开环规划误差、感知 mAP、预测 minADE 等指标很重要，但量产功能还要关心长尾场景、HMI、接管、故障诊断、舒适性、法规和用户误用喵。

5. **认为功能定义只是产品经理的事情喵。** 系统开发工程师如果不理解功能定义，就很难写出正确的需求、接口、状态机、诊断逻辑、降级策略和测试用例喵。

## Socratic Check

1. 如果一个系统能在高速上自动跟车、保持车道、按导航自动变道和上下匝道，它一定是 L3 吗喵？

   提示：答案是"不一定"，因为还要看驾驶员是否需要持续监督、系统是否承担 ODD 内完整动态驾驶任务、是否有明确接管和最小风险策略，以及产品和法规责任是否声明为 L3 喵。

2. 为什么城市 NOA 通常比高速 NOA 难很多，即使二者可以使用类似的感知、预测和规划模块喵？

   提示：城市 ODD 更开放，交通参与者更多，规则和交互更复杂，遮挡和长尾场景更多，系统既要会开车，也要更频繁地判断自己是否仍在可用边界内喵。

## Quick Summary

- ADAS/NOA 功能定义关注"系统承诺做什么、在什么条件下做、做不到时如何退出"，不是单纯算法模块介绍喵。
- ACC、LCC、AEB、APA 分别对应典型纵向控制、横向控制、紧急制动和泊车辅助场景，它们的功能边界和 ODD 不同喵。
- 高速 NOA 更结构化，城市 NOA 更复杂，核心差异在道路环境、交通交互、规则理解和长尾风险喵。
- ODD 是量产智驾功能的边界盒子，直接决定可用条件、系统需求、降级策略和验证范围喵。
- SAE L2 与 L3 的关键差异是驾驶责任和动态驾驶任务分配，NOA 不应被自动等同于 L3 喵。
