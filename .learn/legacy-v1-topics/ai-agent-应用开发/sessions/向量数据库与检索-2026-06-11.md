## 定位

向量数据库与检索位于 **RAG 知识库 → 向量数据库与检索** 分支，包括 Chroma/Milvus/FAISS、相似度检索、混合检索三块。上一讲理解了 Embedding 原理——向量数据库就是让「计算余弦相似度取 Top-K」这件事在百万级数据上也能毫秒级完成的基础设施。

## 类比

继续图书馆的比喻。上一讲学会了给每本书贴坐标贴纸——但图书馆有 100 万本书，不可能每次都走遍所有书架逐一比对坐标。

向量数据库 = **图书馆的分区编目系统**：

- 把书按坐标范围分成若干**区域**（索引结构），比如「科技类在东区」「文学类在西区」。
- 进来说「我要找跟预算相关的」，管理员先根据坐标判断「大概在东区第 3-5 排」，只在那几排里精确查找。
- 100 万本书 → 实际只比对了 1000 本 → 速度提升 1000 倍，代价是可能漏掉个别在边界上的书（近似搜索）。

**向量数据库 = 带索引的向量存储 + 近似最近邻（ANN）搜索算法**。

## 核心机制

**1. 为什么需要专门的「向量数据库」**

| 需求 | 普通数据库（MySQL/Postgres） | 向量数据库 |
|------|------|------|
| 存储向量 | 可以（存 JSON/BLOB） | 原生支持 |
| 精确比对 | 可以但巨慢（O(N)全表扫描） | 不做精确，做近似 |
| 近似搜索 | 不支持 | 核心能力（毫秒级） |
| 百万级规模 | 卡死 | 轻松 |
| 带元数据过滤 | 擅长 | 也支持 |

核心区别：向量数据库用**近似最近邻（ANN）**算法建索引，牺牲极小的精度（recall ~95-99%），换取几个数量级的速度提升。

**2. 主流方案对比**

| 方案 | 类型 | 特点 | 适合场景 |
|------|------|------|------|
| **FAISS** | 库（Meta 开源） | 纯内存、极快、无服务端 | 数据量 < 1000 万，嵌入本地应用 |
| **Chroma** | 轻量数据库 | Python 原生、零配置 | 原型开发、中小项目（< 100 万） |
| **Milvus** | 分布式数据库 | 高可用、可水平扩展 | 亿级数据、生产集群 |
| **Qdrant** | 独立服务 | Rust 实现，过滤性能强 | 生产中等规模、需复杂过滤 |
| **pgvector** | PG 插件 | 复用现有 Postgres | 已有 PG 基础设施 |

黄金法则：原型用 Chroma，生产看规模选 Milvus/Qdrant/pgvector。

**3. 混合检索（Hybrid Search）**

向量搜索对精确匹配不行，关键词对语义匹配不行。混合检索 = 两种方式同时跑，合并结果：

```
用户问题
  ├─→ 向量检索：语义相似 Top-K₁
  ├─→ 关键词检索（BM25）：词频匹配 Top-K₂
  └─→ 合并 + 重排序（RRF/交叉编码器） → 最终 Top-K
```

**Reciprocal Rank Fusion (RRF)**：
```
RRF_score(doc) = Σ  1 / (k + rank_in_source_i)
```

一篇文档如果在两种检索里都排名靠前，融合分数就高。

上一讲思考题的答案：「推荐餐厅」时「预算5000」检索不到——用混合检索 + 分类标签，给记忆打 `type: constraint`，检索时除了语义 Top-K 还强制注入所有 constraint 条目。

**4. 元数据过滤（Metadata Filtering）**

每条记录可带元数据，检索时加过滤条件：先过滤再向量搜索，既提升准确率又降低计算量。

## 代码示例

用 Chroma 实现完整的 RAG 检索器：

```python
import chromadb
from openai import OpenAI

client = OpenAI()
EMBED_MODEL = "text-embedding-3-small"

# ===== 初始化 Chroma（本地持久化） =====
chroma = chromadb.PersistentClient(path="./vector_db")
collection = chroma.get_or_create_collection(
    name="company_knowledge",
    metadata={"hnsw:space": "cosine"},
)

# ===== 建库：公司技术文档入库 =====
docs = [
    {"text": "部署流程：先打包Docker镜像，推送到阿里云ACR，再用K8s滚动更新", "type": "procedure"},
    {"text": "数据库密码存储在阿里云KMS中，禁止硬编码到代码里", "type": "constraint"},
    {"text": "每周五下午4点是代码冻结时间，不允许合并PR", "type": "constraint"},
    {"text": "项目预算上限为5000元/月，超出需审批", "type": "constraint"},
    {"text": "前端使用React18+TypeScript，后端使用FastAPI", "type": "fact"},
    {"text": "上次ES集群宕机是因为JVM堆内存设置过小，调到16G后稳定", "type": "experience"},
]

for i, doc in enumerate(docs):
    resp = client.embeddings.create(input=doc["text"], model=EMBED_MODEL)
    collection.add(
        ids=[f"doc_{i}"],
        embeddings=[resp.data[0].embedding],
        documents=[doc["text"]],
        metadatas=[{"type": doc["type"]}],
    )

# ===== 检索：回答用户问题 =====
question = "我想部署新版本，有什么要注意的？"
q_resp = client.embeddings.create(input=question, model=EMBED_MODEL)

# 纯语义搜索
results = collection.query(
    query_embeddings=[q_resp.data[0].embedding],
    n_results=3,
)

# 带过滤：确保约束不遗漏
constraints = collection.query(
    query_embeddings=[q_resp.data[0].embedding],
    where={"type": "constraint"},
    n_results=5,
)

# 组装进 LLM 上下文
context = "\n".join(results["documents"][0] + constraints["documents"][0])
messages = [
    {"role": "system", "content": f"根据以下公司知识回答问题：\n{context}"},
    {"role": "user", "content": question},
]
answer = client.chat.completions.create(model="gpt-4o", messages=messages)
print(answer.choices[0].message.content)
```

代码走读：`PersistentClient` 数据存磁盘重启不丢；`hnsw:space: cosine` 用余弦距离建 HNSW 索引；两次查询演示「纯语义 + 元数据过滤」的组合；最终两路检索结果拼成 context 注入 LLM——一个完整可工作的 RAG 系统。

## 常见误区

1. **「向量数据库就是存向量的数据库」** —— 核心价值是 ANN 索引 + 毫秒级近似搜索，不是存储本身。
2. **「Top-K 越大越好」** —— K 越大又回到上下文爆炸 + 注意力稀释。实践中 K=3~10 是甜点区。
3. **「入库之后就不用管了」** —— 文档更新/删除了旧版还在，必须设计增量更新/删除机制。
4. **「只用向量检索就够了」** —— 精确匹配是弱项，混合检索才是生产标配。面试答「加 BM25 做 hybrid + reranker 重排序」是加分项。

## 苏格拉底式检查

RAG 系统用来回答公司内部问题。用户问「上次服务器挂了是怎么回事？」，向量检索找到了「ES集群宕机因为JVM堆内存过小」。但如果公司有两次不同原因的宕机事件——模型只看到一条记录可能当成唯一答案。从哪些环节改进让系统在「有多条可能相关」时给出更可靠回答？（提示：分块策略、检索策略、回答策略各想一个）

## 快速总结

- 向量数据库 = 带 ANN 索引的向量存储，让百万级语义搜索在毫秒级完成
- 主流方案：原型 Chroma、生产 Milvus/Qdrant/pgvector、嵌入式 FAISS
- 混合检索（向量 + BM25 + RRF 融合）是生产标配，解决两种检索各自的盲区
- 元数据过滤 = 先缩范围再搜索，提升准确率又降计算量
- Top-K 甜点区 3~10，不是越大越好
