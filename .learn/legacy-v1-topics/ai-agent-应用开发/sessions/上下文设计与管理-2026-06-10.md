# 上下文设计与管理 — 学习会话

> **日期：** 2026-06-10
> **主题：** ai-agent-应用开发
> **路径：** LLM 与 Prompt 基础/上下文设计与管理
> **水平：** 初学者偏上

---

## 定位

上下文设计与管理位于 **LLM 与 Prompt 基础 → 上下文设计与管理** 分支，是 Prompt 工程的直接延伸：Prompt 工程解决「单次怎么问」，上下文管理解决「连续对话、长任务里，模型每次到底该看到什么」。它也是后面 Agent 记忆机制和 RAG 的共同地基。

## 类比

继续用「天才实习生」的比喻，但加一个残酷设定：**这位实习生没有记忆，每次找他都像第一次见面**。

你以为的多轮对话：「他记得我们聊过什么」。
实际上的多轮对话：每次你都递给他**一沓完整的会议纪要**——里面抄录了之前的全部对话——他从头读一遍，然后回答最新的问题。

这沓纪要就是**上下文（Context）**。于是三个工程问题自然冒出来：

1. 纪要最多只能装 N 页（**上下文窗口有限**）；
2. 纪要太厚，他读中间部分会走神（**lost in the middle**）；
3. 每页纸都要花钱（**Token 即成本**）。

**上下文管理 = 决定这沓纪要里装什么、扔什么、压缩什么**。

## 核心机制

**1. LLM 是无状态的，「记忆」是客户端伪造的**

API 不保存任何对话——所谓多轮对话，是你的代码把历史消息拼成数组重新发过去：

```python
messages = [
    {"role": "system", "content": "..."},   # 系统提示词
    {"role": "user", "content": "第1问"},
    {"role": "assistant", "content": "第1答"},
    {"role": "user", "content": "第2问"},    # 模型靠前面的内容"记住"了对话
]
```

理解了这一点，就明白上下文管理完全是**你的代码的责任**。

**2. 上下文窗口的分区设计**

一个工程化应用的上下文通常分四区，优先级从高到低：

| 分区 | 内容 | 策略 |
|------|------|------|
| 系统提示词 | 角色、规则、输出格式 | 永远保留，放最前 |
| 长期信息 | 用户偏好、任务目标、知识库检索结果 | 按需注入 |
| 对话历史 | 之前的问答 | **裁剪/压缩的主要对象** |
| 当前输入 | 最新问题 | 永远保留，放最后 |

**3. 历史裁剪的三种策略**

- **滑动窗口**：只保留最近 N 轮，最简单，但会忘记早期关键信息。
- **摘要压缩**：旧对话用 LLM 压缩成一段摘要放在前面，细节丢失但主线保留——Claude Code 在长对话时就是这么干的。
- **选择性保留**：固定保留「关键事实」（用户说过的需求、做过的决定），只裁剪闲聊——效果最好，实现最复杂。

**4. 位置很重要**

模型对上下文的**开头和结尾**最敏感，中间最容易被忽略。约定俗成：规则放系统提示词（最前），最新任务放最后，大段参考资料注入中间并用分隔符标明。

## 代码示例

一个带「摘要压缩」的对话管理器——客服机器人、内部助手的标准骨架：

```python
from openai import OpenAI

client = OpenAI()
MODEL = "gpt-4o"
MAX_HISTORY = 6          # 历史超过 6 条消息就触发压缩

class ChatSession:
    def __init__(self, system_prompt: str):
        self.system_prompt = system_prompt
        self.summary = ""        # 旧对话的摘要（长期记忆）
        self.history = []        # 最近的原文对话（短期记忆）

    def _compress(self):
        """把最旧的一半历史压缩进摘要"""
        old, self.history = self.history[:len(self.history)//2], self.history[len(self.history)//2:]
        old_text = "\n".join(f"{m['role']}: {m['content']}" for m in old)
        resp = client.chat.completions.create(
            model=MODEL,
            messages=[{
                "role": "user",
                "content": f"把以下对话压缩成简短摘要，保留关键事实、需求和决定：\n"
                           f"已有摘要：{self.summary}\n新增对话：\n{old_text}",
            }],
        )
        self.summary = resp.choices[0].message.content

    def ask(self, question: str) -> str:
        if len(self.history) > MAX_HISTORY:
            self._compress()                      # 超长就先压缩

        messages = [{"role": "system", "content": self.system_prompt}]
        if self.summary:                          # 摘要作为长期记忆注入
            messages.append({"role": "system",
                             "content": f"之前对话的摘要：{self.summary}"})
        messages += self.history                  # 最近对话原文
        messages.append({"role": "user", "content": question})  # 最新问题放最后

        resp = client.chat.completions.create(model=MODEL, messages=messages)
        answer = resp.choices[0].message.content
        self.history += [{"role": "user", "content": question},
                         {"role": "assistant", "content": answer}]
        return answer
```

代码走读：每次 `ask()` 都重新拼装完整上下文（印证「无状态」）；历史超过阈值时，旧的一半被 LLM 压缩成摘要——压缩 Prompt 里「保留关键事实、需求和决定」正是选择性保留的思想；拼装顺序严格遵守分区表：系统提示词最前、摘要次之、原文历史在中、新问题最后。

## 常见误区

1. **「上下文窗口越大越好，1M Token 还管什么上下文」** —— 错。窗口再大，长上下文的中间注意力衰减依然存在，而且 Token 数直接等于钱和延迟。大窗口给了你余地，不是免除管理的理由。
2. **把所有东西塞进系统提示词** —— 系统提示词应该只放**稳定不变**的规则；会变的信息（检索结果、用户资料）应该作为独立消息按需注入，否则每轮都为不相关内容付费。
3. **裁剪历史时把 user/assistant 拆散** —— 裁剪必须以「一问一答」为单位，只删一半会让模型看到没头没尾的对话，行为变得诡异。
4. **忘了工具调用结果也占上下文** —— 到了 Agent 阶段，工具返回的大段 JSON 往往比对话本身更吃 Token，同样需要裁剪和压缩策略。

## 苏格拉底式检查

用摘要压缩有个隐患：假如用户在第 2 轮说过「预算不超过 5000 元」，对话进行到第 30 轮时，这条信息经过了好几次「摘要的摘要」，可能已经面目全非或彻底丢失。结合三种裁剪策略，工程上可以怎么防止「关键约束被压缩丢失」？

## 快速总结

- LLM 无状态，多轮"记忆"是客户端每次重新拼装上下文伪造出来的
- 上下文四分区：系统提示词（最前）→ 长期信息 → 对话历史（裁剪对象）→ 当前输入（最后）
- 三种裁剪策略：滑动窗口（简单）、摘要压缩（平衡）、选择性保留（最好但复杂）
- 开头结尾最受关注，中间易被忽略；Token = 成本 + 延迟，大窗口不等于不用管理
