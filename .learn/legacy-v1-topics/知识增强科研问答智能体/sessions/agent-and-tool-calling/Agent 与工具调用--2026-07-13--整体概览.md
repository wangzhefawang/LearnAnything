# Agent 与工具调用 — 学习会话
> **日期：** 2026-07-13
> **主题：** 知识增强科研问答智能体
> **路径：** Agent 与工具调用（领域整体概览）
> **水平：** 中级（有项目实践）
---
## 定位
Agent 与工具调用处在科研问答系统的编排层喵。
它不直接替代 FAISS、Neo4j、RoBERTa 向量化或 RAG，而是决定什么时候调用这些组件、用什么参数调用、如何处理返回结果、以及何时停止并生成答案喵。
如果把昨天学过的 GraphRAG 看成「混合检索链路」，那么 Agent 就是这条链路上方的「调度大脑」喵。
GraphRAG 解决的是「怎么把向量召回、图扩展、上下文融合起来」的问题喵。
Agent 与工具调用解决的是「面对一个具体问题，应该走哪条路径、调用哪些工具、观察结果后是否继续查」的问题喵。
在你的科研问答智能体项目中，FAISS 检索适合找语义相近论文片段喵。
Neo4j 图查询适合沿着实体、方法、任务、指标、论文之间的关系做结构化扩展喵。
重排工具适合在候选证据较多时筛出更相关的段落喵。
摘要工具适合把长论文、多个片段或图谱扩展结果压缩成可用上下文喵。
LLM 最终负责理解问题、规划工具调用、融合证据、组织答案和表达不确定性喵。
因此，这个领域的核心价值不是「让模型更会聊天」，而是让问答流程从固定流水线升级为可控的动态决策流程喵。
面试中你可以把它概括为一句话：我用 Agent 编排 FAISS、Neo4j、重排和摘要工具，让系统能根据问题类型自主选择检索路径，并用工具返回的证据完成可追溯回答喵。
## 类比
可以把整个系统想象成一位实验室主管带着一组研究助理和仪器工作喵。
LLM 是实验室主管，负责理解研究问题、制定调查路线、判断证据是否足够，并最终写出报告喵。
FAISS 检索工具像文献库检索员，能根据语义快速找出相似论文片段喵。
Neo4j 图查询工具像知识图谱管理员，能回答某个方法关联了哪些任务、指标、数据集和论文喵。
重排工具像资深审稿助理，能把一堆候选材料按与问题的相关性重新排序喵。
摘要工具像速记员，能把长文献压缩成短证据摘要喵。
Function Calling 就像主管给每位助理发的标准工作单，规定输入字段、输出格式、失败时怎么报错喵。
ReAct 循环就像主管每做一步都先想一想、派一个助理行动、看回报结果、再决定下一步喵。
长文本问答策略就像主管面对不同研究任务时采用不同调查流程，查事实、写综述和做比较不会使用同一套路线喵。
这个类比的重点是：Agent 不是「一个更大的模型」，而是「会调度工具的研究主管」喵。
如果主管没有标准工作单，助理会收到含糊任务喵。
如果主管没有行动上限，调查可能一直绕圈喵。
如果主管把所有原始材料都塞进最终报告，读者会看不到重点喵。
## 核心机制
第一个概念是 Function Calling 工具接口喵。
Function Calling 的本质是让 LLM 以结构化方式提出「我要调用哪个工具、传什么参数」的请求喵。
它不是模型真的在自己执行 Python 函数喵。
真正执行函数的是你的 Agent 运行时、后端服务或编排框架喵。
在科研问答项目里，典型工具可以包括 `search_papers`、`query_graph`、`rerank`、`summarize` 和 `answer_with_citations` 喵。
`search_papers` 可以接收查询语句、召回数量、过滤条件，并返回论文片段、标题、年份、相似度和来源位置喵。
`query_graph` 可以接收实体名、关系类型、最大跳数或 Cypher 模板参数，并返回图谱路径、邻居节点和关系证据喵。
`rerank` 可以接收问题和候选证据列表，并返回排序后的证据 ID 与分数喵。
`summarize` 可以接收长文本、目标问题和摘要长度，并返回压缩后的要点喵。
工具 schema 的作用是把这些接口写清楚，让模型不能只说「帮我查一下」这种模糊请求喵。
一个好的 schema 至少要包含工具名称、工具用途、参数类型、必填字段、枚举值、默认值和返回结构喵。
例如 `search_papers` 的 `top_k` 应该是整数，并且可以限制在 1 到 20 之间喵。
例如 `query_graph` 的 `relation_types` 可以限制为 `method_of`、`evaluated_on`、`improves_metric`、`cites` 等枚举关系喵。
参数校验是 Function Calling 落地时非常关键的一步喵。
LLM 生成的参数可能类型错误、字段缺失、实体名歧义、检索数量过大，甚至试图构造危险的 Cypher 查询喵。
因此，Agent 运行时应该先验证参数，再决定是否调用真实工具喵。
如果参数不合法，系统可以把错误作为 observation 回填给 LLM，让它修正参数后重试喵。
异常返回也需要结构化喵。
工具失败时不应该只抛出一段栈信息给模型喵。
更好的做法是返回 `ok: false`、`error_type`、`message` 和 `retryable` 字段喵。
这样 LLM 能区分「网络临时失败」「实体不存在」「参数非法」「没有检索结果」等不同情况喵。
工具结果回填是 Function Calling 的最后一环喵。
回填不是把完整数据库结果无脑塞进上下文喵。
回填应该只给 LLM 继续推理所需的紧凑证据、来源 ID、分数和必要元数据喵。
例如 FAISS 返回 50 个片段时，可以先重排并只回填前 5 个候选证据喵。
例如 Neo4j 返回多跳路径时，可以把路径压缩成「论文 A 使用方法 B，在数据集 C 上提升指标 D」的证据句喵。
面试官可能会问：为什么不用普通 prompt 让模型自己决定怎么查喵。
你可以回答：Function Calling 把工具使用从自然语言意图变成可验证的结构化调用，便于类型校验、权限控制、日志追踪、异常处理和评估复现喵。
面试官也可能问：工具 schema 设计不好会怎样喵。
你可以回答：schema 太宽会导致乱传参数和滥用工具，schema 太窄会限制推理路径，所以我会根据科研问答任务把参数约束、返回字段和错误格式一起设计喵。
第二个概念是 ReAct 推理与行动循环喵。
ReAct 是 Reasoning and Acting 的组合思路喵。
它让 Agent 在「思考、行动、观察」之间循环，而不是一次性直接回答喵。
典型流程是 `Thought -> Action -> Observation -> Thought -> Action -> Observation -> Final Answer` 喵。
Thought 表示模型当前对问题的判断和下一步计划喵。
Action 表示模型选择某个工具并生成结构化参数喵。
Observation 表示工具执行后的返回结果喵。
Final Answer 表示证据足够后给出最终回答喵。
在你的项目中，一个事实型问题可能先触发 FAISS 检索喵。
如果检索结果提到关键方法但缺少关系背景，Agent 可以继续调用 Neo4j 查询该方法与数据集、指标和论文之间的关系喵。
如果候选证据过多，Agent 可以调用 rerank 喵。
如果证据片段太长，Agent 可以调用 summarize 喵。
这就是把昨天的 GraphRAG 链路放进一个可循环决策框架里喵。
ReAct 的优势是它允许系统根据中间观察调整路线喵。
固定 RAG 流水线通常是先检索、再拼上下文、再回答喵。
ReAct 可以在发现召回不充分时改写查询，在发现实体歧义时查询图谱，在发现证据冲突时追加检索喵。
但 ReAct 也有明显风险喵。
第一个风险是循环失控喵。
如果没有最大步数、超时限制或置信度终止条件，Agent 可能不断检索相似内容喵。
第二个风险是工具误用喵。
例如一个简单定义题本来只需要检索一两段，Agent 却反复查询图谱和摘要，导致延迟和成本上升喵。
第三个风险是无效推理链喵。
模型可能写出看似合理的 Thought，但后续 Action 与问题目标没有关系喵。
第四个风险是观察污染喵。
如果 Observation 中包含低质量或冲突证据，模型可能把它们当作事实吸收喵。
因此，ReAct 必须有明确的终止条件喵。
常见终止条件包括达到最大步数、找到足够数量的高置信证据、问题已被直接回答、连续两次工具调用没有新增信息、或工具返回不可恢复错误喵。
在科研问答里，还可以设置「至少包含一个文本证据和一个图谱关系证据」作为某些问题的终止条件喵。
例如比较两个方法时，系统可以要求同时拿到两种方法的性能指标证据和图谱关系证据喵。
面试官可能会问：ReAct 和普通多步 RAG 的区别是什么喵。
你可以回答：普通多步 RAG 的步骤通常由开发者预设，而 ReAct 是由 LLM 根据 Observation 动态决定下一步工具调用，并通过终止条件控制风险喵。
面试官可能会问：你怎么发现 Agent 在乱调工具喵。
你可以回答：我会记录每一步 Thought、Action、Observation、工具耗时、返回证据数量和最终答案引用，并用无效调用率、重复检索率和答案支持率评估喵。
第三个概念是长文本问答策略喵。
科研问答经常面对论文全文、综述、多篇论文比较和复杂背景问题喵。
长文本问答不能简单依赖一次向量检索喵。
更稳妥的策略是分层检索、逐段摘要和证据合并喵。
分层检索可以先在论文标题、摘要、关键词层面粗召回喵。
然后再进入章节、段落、句子或 chunk 层面做细召回喵。
对于你的项目，可以先用 RoBERTa 或 Transformers embedding 在 FAISS 中召回相关论文片段喵。
随后使用 Neo4j 扩展相关实体，例如方法、任务、数据集、指标和引用关系喵。
接着用 rerank 对文本证据和图谱扩展证据进行统一排序喵。
最后用 summarize 将多个证据片段压缩成可进入 LLM 上下文的证据包喵。
逐段摘要的关键不是「越短越好」，而是保留与问题相关的结论、条件、指标、比较对象和来源喵。
证据合并的关键是处理重复、互补和冲突喵。
重复证据可以合并成一个高置信结论喵。
互补证据可以组成更完整的解释链喵。
冲突证据需要在答案中显式标注不确定性喵。
不同问题类型应该走不同回答流程喵。
事实型问题通常要求短路径、强引用和低发散喵。
例如「论文 X 在数据集 Y 上的 F1 是多少」应该优先精确检索论文片段，并用图谱验证论文、数据集和指标实体关系喵。
事实型回答应该直接给结论、引用来源，并避免扩展太多背景喵。
综述型问题通常要求宽召回、聚类归纳和结构化小节喵。
例如「知识图谱增强 RAG 的主要方法有哪些」可以先召回多篇相关论文，再按方法类别、图谱构建、检索增强、推理融合和评估指标组织答案喵。
综述型回答应该展示分类框架、代表工作、共性趋势和局限喵。
比较型问题通常要求对齐维度喵。
例如「GraphRAG 和普通 RAG 在科研问答中的差异是什么」应该按检索范围、结构化推理、证据可追溯性、成本和适用场景比较喵。
比较型回答不能只列两个方法各自特点，而要在同一维度上逐项对齐喵。
答案结构控制也很重要喵。
事实型答案可以用「结论、证据、备注」结构喵。
综述型答案可以用「背景、方法分类、代表证据、局限、总结」结构喵。
比较型答案可以用表格或并列小节喵。
引用来源应该尽量绑定到具体论文、段落、图谱路径或证据 ID 喵。
不确定性表达不是软弱，而是科研问答系统可信度的一部分喵。
当证据不足时，答案应该说「当前检索证据未覆盖该结论」喵。
当证据冲突时，答案应该说「不同来源给出不一致结果，可能与实验设置或指标口径有关」喵。
当图谱只提供关联而非因果时，答案应该避免把关系路径说成因果证明喵。
面试官可能会问：长文本为什么不能直接塞给大模型喵。
你可以回答：直接塞全文会带来上下文成本、注意力稀释、来源不可控和答案不可复现问题，所以我用分层检索、摘要压缩和证据合并来控制输入质量喵。
面试官可能会问：怎么保证摘要没有丢关键信息喵。
你可以回答：摘要不是最终证据替代品，而是带来源 ID 的中间压缩结果，最终答案仍需保留可回溯引用，并对关键数值、实体关系和比较结论做校验喵。
## 代码示例
下面是一个最小 Agent 骨架，展示工具 schema、简易 ReAct 循环、工具分发和结果回填喵。
```python
from dataclasses import dataclass
from typing import Any, Dict, List, Literal, Optional


ToolName = Literal["search_papers", "query_graph", "final_answer"]


TOOL_SCHEMAS = [
    {
        "name": "search_papers",
        "description": "Search paper chunks from a FAISS index.",
        "parameters": {
            "type": "object",
            "properties": {
                "query": {"type": "string"},
                "top_k": {"type": "integer", "minimum": 1, "maximum": 10},
            },
            "required": ["query", "top_k"],
        },
    },
    {
        "name": "query_graph",
        "description": "Query research entities and relations from Neo4j.",
        "parameters": {
            "type": "object",
            "properties": {
                "entity": {"type": "string"},
                "relation": {
                    "type": "string",
                    "enum": ["method_of", "evaluated_on", "improves_metric", "cites"],
                },
                "max_hops": {"type": "integer", "minimum": 1, "maximum": 2},
            },
            "required": ["entity", "relation", "max_hops"],
        },
    },
]


@dataclass
class ToolCall:
    name: ToolName
    arguments: Dict[str, Any]


def validate_tool_call(call: ToolCall) -> Optional[str]:
    if call.name == "search_papers":
        if not isinstance(call.arguments.get("query"), str):
            return "search_papers.query must be a string"
        top_k = call.arguments.get("top_k")
        if not isinstance(top_k, int) or not 1 <= top_k <= 10:
            return "search_papers.top_k must be an integer between 1 and 10"
        return None

    if call.name == "query_graph":
        if not isinstance(call.arguments.get("entity"), str):
            return "query_graph.entity must be a string"
        if call.arguments.get("relation") not in {
            "method_of",
            "evaluated_on",
            "improves_metric",
            "cites",
        }:
            return "query_graph.relation is not allowed"
        max_hops = call.arguments.get("max_hops")
        if not isinstance(max_hops, int) or not 1 <= max_hops <= 2:
            return "query_graph.max_hops must be 1 or 2"
        return None

    if call.name == "final_answer":
        return None

    return f"unknown tool: {call.name}"


def search_papers(query: str, top_k: int) -> Dict[str, Any]:
    # Pseudo FAISS retrieval.
    return {
        "ok": True,
        "results": [
            {
                "id": "paper_17_chunk_03",
                "title": "Graph-enhanced Retrieval for Scientific QA",
                "score": 0.86,
                "text": "Graph expansion improves evidence coverage for multi-hop scientific questions.",
            }
        ][:top_k],
    }


def query_graph(entity: str, relation: str, max_hops: int) -> Dict[str, Any]:
    # Pseudo Neo4j query.
    return {
        "ok": True,
        "paths": [
            {
                "source": entity,
                "relation": relation,
                "target": "scientific question answering",
                "evidence": "The method is evaluated on scientific QA benchmarks.",
                "hops": 1,
            }
        ],
    }


def dispatch_tool(call: ToolCall) -> Dict[str, Any]:
    error = validate_tool_call(call)
    if error:
        return {"ok": False, "error_type": "validation_error", "message": error, "retryable": True}

    if call.name == "search_papers":
        return search_papers(**call.arguments)

    if call.name == "query_graph":
        return query_graph(**call.arguments)

    return {"ok": False, "error_type": "unknown_tool", "message": call.name, "retryable": False}


def fake_llm_plan(question: str, observations: List[Dict[str, Any]]) -> ToolCall:
    # Replace this function with a real LLM function-calling response.
    if not observations:
        return ToolCall(
            name="search_papers",
            arguments={"query": question, "top_k": 5},
        )

    last = observations[-1]
    if last.get("ok") and "results" in last:
        return ToolCall(
            name="query_graph",
            arguments={"entity": "GraphRAG", "relation": "method_of", "max_hops": 1},
        )

    return ToolCall(
        name="final_answer",
        arguments={
            "answer": "GraphRAG can combine semantic retrieval with graph expansion to improve evidence coverage.",
            "citations": ["paper_17_chunk_03"],
        },
    )


def run_agent(question: str, max_steps: int = 4) -> Dict[str, Any]:
    observations: List[Dict[str, Any]] = []

    for step in range(max_steps):
        call = fake_llm_plan(question, observations)

        if call.name == "final_answer":
            return {
                "answer": call.arguments["answer"],
                "citations": call.arguments.get("citations", []),
                "steps": step,
                "observations": observations,
            }

        observation = dispatch_tool(call)
        observations.append(
            {
                "tool": call.name,
                "arguments": call.arguments,
                "observation": observation,
            }
        )

    return {
        "answer": "I do not have enough verified evidence to answer confidently.",
        "citations": [],
        "steps": max_steps,
        "observations": observations,
    }


if __name__ == "__main__":
    result = run_agent("How does GraphRAG improve scientific question answering?")
    print(result["answer"])
```
这个示例里，`TOOL_SCHEMAS` 表示暴露给 LLM 的工具契约喵。
真实系统中，这些 schema 会交给支持 Function Calling 的模型，让模型输出结构化工具调用喵。
`validate_tool_call` 表示工具调用前的参数校验层喵。
它保证 `top_k`、`relation`、`max_hops` 这类字段不会越界或乱填喵。
`search_papers` 是 FAISS 检索的伪实现喵。
真实项目中，它会把问题编码成向量，再到 FAISS index 中召回相似论文 chunk 喵。
`query_graph` 是 Neo4j 图查询的伪实现喵。
真实项目中，它会把实体和关系映射到 Cypher 查询，再返回路径、节点、关系和证据元数据喵。
`dispatch_tool` 是工具分发器喵。
它把模型提出的工具调用变成真实函数执行，并把异常统一包装成结构化 observation 喵。
`fake_llm_plan` 只是模拟 LLM 的决策喵。
第一步没有观察结果时，它选择先做论文检索喵。
看到检索结果后，它继续查询图谱关系喵。
真实系统中，这个函数会由 LLM 根据系统提示、工具 schema 和历史 observation 生成喵。
`run_agent` 展示了 ReAct 循环的最小形态喵。
它每一步让模型规划工具调用，执行工具，把 Observation 回填，再进入下一轮喵。
`max_steps` 是防止循环失控的硬限制喵。
当模型给出 `final_answer` 时，循环提前终止喵。
如果达到最大步数仍没有答案，系统返回保守失败信息，而不是硬编一个结论喵。
这个骨架还缺少真实 LLM 调用、重排、摘要、证据合并和引用格式化喵。
但它已经包含面试中最关键的工程思想：schema 约束、参数校验、工具分发、结果回填、循环终止和保守失败喵。
## 常见误区
第一个误区是以为 Function Calling 是模型真的执行了函数喵。
Function Calling 只是让模型生成结构化调用请求，真实执行发生在你的程序、服务或框架里喵。
第二个误区是 ReAct 循环不设上限喵。
没有最大步数、超时、重复检测和终止条件的 Agent 很容易陷入反复检索或无效查询喵。
第三个误区是把所有工具结果全量塞回上下文喵。
这会造成上下文膨胀、注意力稀释、成本上升和答案引用混乱喵。
第四个误区是以为 Agent 一定比固定 RAG 流水线好喵。
对于稳定、简单、流程明确的问题，固定 RAG 可能更快、更便宜、更容易评估喵。
第五个误区是忽略工具结果的可信度和来源喵。
科研问答不能只看模型说得是否流畅，还要看结论是否被具体证据、论文片段或图谱路径支持喵。
## 苏格拉底式检查
如果用户问「GraphRAG 为什么能提升科研问答准确率」，你的 Agent 应该先调用 FAISS、先查 Neo4j，还是先判断问题类型，为什么喵。
如果 Neo4j 图谱显示方法 A 关联了数据集 B，但 FAISS 检索到的论文片段没有明确支持这个实验结果，你的系统应该如何表达答案中的不确定性喵。

## 快速总结
- Agent 与工具调用是科研问答系统的编排层，负责把 LLM、FAISS、Neo4j、重排、摘要和 RAG 串成动态决策流程喵。
- Function Calling 通过 schema、参数校验、异常返回和结果回填，把工具调用变成可控、可追踪、可评估的工程接口喵。
- ReAct 通过 Thought、Action、Observation 循环让系统根据中间证据调整检索路线，但必须设置终止条件和失控防护喵。
- 长文本问答要按事实型、综述型和比较型问题分流，并结合分层检索、逐段摘要、证据合并和引用控制喵。
- 面试表达重点是说明你如何用 Agent 编排混合检索链路，并如何控制工具误用、上下文膨胀和证据可信度喵。
